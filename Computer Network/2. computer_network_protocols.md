## 1. HTTP/1.1, HTTP/2, HTTP/3

### What is HTTP?

**HTTP (HyperText Transfer Protocol)** is the foundation of data communication on the web. It's a **request-response protocol** — a client (browser, mobile app, your backend service) sends a request, and the server sends back a response.

**Analogy:**
> HTTP is like ordering food at a restaurant. You (client) give your order (request) to the waiter (HTTP), the kitchen (server) prepares it, and the waiter brings it back (response). Every order is a separate interaction.

---

### The Request-Response Cycle

```
CLIENT                                          SERVER
  |                                               |
  |------- HTTP Request ------------------------->|
  |        GET /users/1 HTTP/1.1                  |
  |        Host: api.example.com                  |
  |        Accept: application/json               |
  |        Authorization: Bearer abc123           |
  |                                               |
  |<------ HTTP Response --------------------------|
  |        HTTP/1.1 200 OK                        |
  |        Content-Type: application/json         |
  |        Content-Length: 87                     |
  |                                               |
  |        { "id": 1, "name": "Alice" }           |
  |                                               |
```

---

### Anatomy of an HTTP Request

```
GET /api/users?page=1 HTTP/1.1          ← Request Line (Method + Path + Version)
Host: api.example.com                   ←┐
Accept: application/json                  │ Headers (key: value)
Authorization: Bearer eyJhbGc...          │
Content-Type: application/json          ←┘
                                        ← Blank line (separates headers from body)
{ "filter": "active" }                  ← Body (optional, used with POST/PUT/PATCH)
```

### Anatomy of an HTTP Response

```
HTTP/1.1 200 OK                         ← Status Line (Version + Status Code + Reason)
Content-Type: application/json          ←┐
Content-Length: 87                        │ Response Headers
Cache-Control: max-age=3600             ←┘
                                        ← Blank line
{ "id": 1, "name": "Alice" }           ← Response Body
```

---

### HTTP Methods

Each method has a clear semantic meaning. As a backend dev, you MUST know these cold.

| Method | Purpose | Has Body? | Idempotent? | Safe? |
|--------|---------|-----------|-------------|-------|
| GET | Read/fetch a resource | No | Yes | Yes |
| POST | Create a new resource | Yes | No | No |
| PUT | Replace a resource entirely | Yes | Yes | No |
| PATCH | Partially update a resource | Yes | No | No |
| DELETE | Remove a resource | No | Yes | No |
| HEAD | Like GET but returns only headers | No | Yes | Yes |
| OPTIONS | Ask server what methods it supports | No | Yes | Yes |

**Idempotent** = calling it multiple times has the same effect as calling it once.
**Safe** = it doesn't change server state (read-only).

```
GET    /users         → List all users
GET    /users/1       → Get user with id 1
POST   /users         → Create a new user
PUT    /users/1       → Replace user 1 entirely
PATCH  /users/1       → Update only specific fields of user 1
DELETE /users/1       → Delete user 1
```

---

### HTTP Status Codes

Status codes tell you what happened with your request. They're grouped into 5 classes:

```
1xx → Informational  (request received, continuing)
2xx → Success        (request was successful)
3xx → Redirection    (client must take further action)
4xx → Client Error   (you sent something wrong)
5xx → Server Error   (server messed up)
```

**Most important ones for backend devs:**

```
200 OK              → Standard success
201 Created         → Resource created successfully (after POST)
204 No Content      → Success but no body (after DELETE)

301 Moved Permanently → Resource moved; update your bookmarks
302 Found (Redirect)  → Temporary redirect
304 Not Modified      → Cached version is still valid (use ETag)

400 Bad Request     → Malformed request (invalid JSON, missing fields)
401 Unauthorized    → Not authenticated (no/invalid token)
403 Forbidden       → Authenticated but not authorized (no permission)
404 Not Found       → Resource doesn't exist
405 Method Not Allowed → You used GET on a POST-only endpoint
409 Conflict        → Resource conflict (duplicate email)
422 Unprocessable   → Validation failed (correct format, wrong data)
429 Too Many Requests → Rate limited

500 Internal Server Error → Unhandled exception on server
502 Bad Gateway     → Upstream server (proxy/load balancer) got bad response
503 Service Unavailable → Server is overloaded or down
504 Gateway Timeout → Upstream server didn't respond in time
```

---

### Important HTTP Headers

**Request Headers:**
```
Host: api.example.com               → Which server (required in HTTP/1.1)
Authorization: Bearer <token>        → Auth credentials
Content-Type: application/json       → Format of the request body
Accept: application/json             → What format you want in response
User-Agent: Mozilla/5.0 ...          → Info about the client
Cookie: session=abc123               → Client's stored cookies
If-None-Match: "etag-value"          → Conditional request (caching)
```

**Response Headers:**
```
Content-Type: application/json       → Format of the response body
Content-Length: 256                  → Size of body in bytes
Cache-Control: max-age=3600          → How long to cache the response
Set-Cookie: session=abc; HttpOnly    → Sets a cookie on the client
ETag: "33a64df5"                     → Version identifier for caching
Location: /users/42                  → Used with 201/301/302 redirects
X-RateLimit-Remaining: 99           → Custom header (rate limit info)
```

---

### HTTP/1.1 — The Workhorse

Released in 1997, HTTP/1.1 is the version most developers learned first.

**Key features:**
- **Persistent connections** — keep TCP connection alive across multiple requests (`Connection: keep-alive`). Before this, every request opened and closed a new TCP connection.
- **Chunked transfer encoding** — stream response in chunks without knowing Content-Length upfront.
- **Host header required** — enables multiple websites on the same IP (virtual hosting).

**Big problem — Head-of-Line Blocking:**
```
Request 1: GET /large-image.jpg   ← Takes 2 seconds
Request 2: GET /tiny-script.js    ← Must WAIT for Request 1 to finish
Request 3: GET /styles.css        ← Must WAIT for Request 2

Even though scripts.js is tiny, it waits behind the large image.
This is called HEAD-OF-LINE BLOCKING.
```

Browsers worked around this by opening **6 parallel TCP connections per domain** — ugly hack.

---

### HTTP/2 — The Performance Upgrade

Released in 2015. Same semantics (methods, headers, status codes) — but completely redesigned transport.

**Key improvements:**

**1. Multiplexing**
```
HTTP/1.1: One request at a time per connection
┌─────────────────────────────────────┐
│ Req1 ──────► | Req2 ──► | Req3 ──► │  (sequential)
└─────────────────────────────────────┘

HTTP/2: Multiple requests simultaneously on ONE connection
┌─────────────────────────────────────┐
│ Req1 ──────────────────────────────►│
│ Req2 ──────────────────────────────►│  (parallel streams)
│ Req3 ──────────────────────────────►│
└─────────────────────────────────────┘
```

**2. Header Compression (HPACK)**
- HTTP/1.1: Headers are plain text and repeated on every request.
- HTTP/2: Headers are compressed. Repeated headers (like `Authorization`) are sent only once and referenced by index.

**3. Server Push**
- Server can proactively send resources the client will need, before it asks.
```
Client: GET /index.html
Server: Here's index.html... AND I'm also pushing styles.css and app.js
        (because I know you'll need them next)
```

**4. Binary Protocol**
- HTTP/1.1 is text-based (human readable).
- HTTP/2 is binary (faster to parse, more compact).

---

### HTTP/3 — The Modern Web

Released in 2022. Built on **QUIC** instead of TCP.

**Why replace TCP?**

Even with HTTP/2 multiplexing, there's still a TCP-level problem:
```
HTTP/2 over TCP:
Stream 1: ████████████
Stream 2: ████████████
Stream 3: ██░░░░░████  ← Packet lost here

ALL streams must WAIT for the lost packet to be retransmitted.
TCP doesn't know these are independent streams — it sees one byte stream.
```

**QUIC (Quick UDP Internet Connections) solves this:**
- Built on **UDP** (not TCP).
- Implements its own **reliability per stream**.
- A lost packet in Stream 3 doesn't block Stream 1 or Stream 2.

**HTTP/3 advantages:**
```
1. No head-of-line blocking at the transport layer
2. Faster connection setup (0-RTT or 1-RTT vs TCP's 3-way handshake + TLS)
3. Connection migration — your connection survives switching from WiFi to 4G
   (QUIC uses Connection IDs, not IP+port tuples)
4. Built-in TLS 1.3 — encryption is mandatory
```

---

### HTTP Evolution Summary

| Feature | HTTP/1.1 | HTTP/2 | HTTP/3 |
|---------|----------|--------|--------|
| Year | 1997 | 2015 | 2022 |
| Transport | TCP | TCP | QUIC (UDP) |
| Format | Text | Binary | Binary |
| Multiplexing | ❌ | ✅ | ✅ |
| Header Compression | ❌ | ✅ HPACK | ✅ QPACK |
| Server Push | ❌ | ✅ | ✅ |
| HOL Blocking | TCP + App | App only | ❌ None |
| TLS Required | ❌ | ❌ (but common) | ✅ Always |

---

### Interview-Ready Summary

> "HTTP is the request-response protocol of the web. HTTP/1.1 introduced persistent connections but suffers from head-of-line blocking — one slow request blocks others. HTTP/2 solved this with multiplexing (multiple streams on one TCP connection), binary framing, and header compression. But TCP itself caused blocking when a packet was lost. HTTP/3 replaced TCP with QUIC (built on UDP), giving true independent streams, faster handshakes, and connection migration. As a backend dev, I work with HTTP methods (GET, POST, PUT, PATCH, DELETE), status codes (200, 201, 400, 401, 403, 404, 500), and headers (Authorization, Content-Type, Cache-Control) every day."

## 2. HTTPS & TLS/SSL

### What is HTTPS?

**HTTPS = HTTP + TLS**

It's plain HTTP but with an encrypted tunnel (TLS) around it. Without HTTPS:
- Anyone on the same network can read your requests and responses (passwords, tokens, data).
- Anyone can impersonate your server (man-in-the-middle attack).

**Analogy:**
> HTTP is like sending a postcard — anyone who handles it can read the message. HTTPS is like putting the postcard in a sealed, tamper-proof envelope that only the recipient can open.

---

### SSL vs TLS — What's the Difference?

- **SSL** (Secure Sockets Layer) — the original protocol, now **deprecated** (SSL 2.0, 3.0 are insecure).
- **TLS** (Transport Layer Security) — the modern, secure replacement.
- TLS 1.2 and **TLS 1.3** are what's used today.
- People still say "SSL" casually, but they almost always mean TLS.

**Current standard:** TLS 1.3 (2018) — faster and more secure than TLS 1.2.

---

### The TLS Handshake (TLS 1.2)

Before any encrypted data flows, the client and server must:
1. Agree on which encryption algorithms to use.
2. Verify the server's identity.
3. Exchange keys to encrypt the session.

```
CLIENT                                          SERVER
  |                                               |
  |--- ClientHello -------------------------------->
  |    "I support TLS 1.2/1.3"                   |
  |    "Here are the cipher suites I support"     |
  |    "Here's a random number (client_random)"   |
  |                                               |
  |<-- ServerHello --------------------------------|
  |    "Let's use TLS 1.2"                        |
  |    "Let's use AES-256-GCM cipher"             |
  |    "Here's my random number (server_random)"  |
  |                                               |
  |<-- Certificate --------------------------------|
  |    "Here's my SSL certificate"                |
  |    (contains server's public key)             |
  |                                               |
  |    [Client verifies certificate with CA]      |
  |                                               |
  |--- ClientKeyExchange ------------------------->|
  |    "Here's the pre-master secret"             |
  |    (encrypted with server's public key)       |
  |                                               |
  |    [Both sides derive session keys from       |
  |     client_random + server_random +           |
  |     pre-master secret]                        |
  |                                               |
  |--- ChangeCipherSpec + Finished --------------->|
  |    "I'm switching to encrypted mode"          |
  |                                               |
  |<-- ChangeCipherSpec + Finished ----------------|
  |    "Me too! Handshake complete."              |
  |                                               |
  |====== Encrypted HTTP Communication ========== |
```

This takes **2 round trips (2-RTT)** before data flows.

---

### TLS 1.3 Handshake — Faster

TLS 1.3 reduced the handshake to **1 round trip (1-RTT)**:

```
CLIENT                                          SERVER
  |                                               |
  |--- ClientHello + Key Share ------------------->|
  |    (includes key exchange data upfront)       |
  |                                               |
  |<-- ServerHello + Key Share + Certificate ------|
  |    + Finished (all in one response!)          |
  |                                               |
  |--- Finished + HTTP Request ------------------->|
  |    (can send data immediately!)               |
  |                                               |
  |<-- HTTP Response ------------------------------|
```

**0-RTT (Session Resumption):**
If you've connected to this server before, TLS 1.3 allows sending data on the **very first packet** — zero round trips. Used for resuming connections quickly.

---

### SSL Certificates

A certificate is a **digital identity document** for your server. It contains:

```
+------------------------------------------------------+
| SSL Certificate                                      |
|------------------------------------------------------|
| Subject:      api.example.com                        |
| Issued by:    Let's Encrypt Authority X3 (CA)        |
| Valid from:   2024-01-01                             |
| Valid until:  2024-04-01                             |
| Public Key:   RSA 2048-bit (or EC P-256)             |
| Fingerprint:  SHA-256: 4a:bc:de:...                  |
| SANs:         api.example.com, www.example.com       |
+------------------------------------------------------+
```

**Certificate Chain (Chain of Trust):**
```
Your Certificate (api.example.com)
      ↑ signed by
Intermediate CA (Let's Encrypt Authority X3)
      ↑ signed by
Root CA (ISRG Root X1)
      ↑ trusted by
Your Browser/OS (pre-installed trust store)
```

Your browser trusts Root CAs that come pre-installed with the OS. Because Root CA signed the Intermediate CA, and Intermediate CA signed your cert, your browser trusts your cert.

**Types of certificates:**
```
DV (Domain Validated)    → Verifies you own the domain. Cheapest, fastest.
                           "https" padlock. Used for most websites.

OV (Organization Validated) → Also verifies your organization is real.
                              More trust, used for business sites.

EV (Extended Validated)  → Strictest verification. Banks, financial sites.

Wildcard Certificate     → *.example.com — covers all subdomains.

SAN Certificate          → Multiple domains in one cert
                           (api.example.com + www.example.com)
```

**Getting a free certificate:**
```bash
# Let's Encrypt with Certbot (most popular)
certbot --nginx -d example.com -d www.example.com

# Certificate is auto-renewed every 90 days
```

---

### Cipher Suites

A **cipher suite** is a combination of algorithms used for the TLS connection. It specifies:

```
TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
 │     │     │        │    │     │
 │     │     │        │    │     └── Hash for integrity (SHA-384)
 │     │     │        │    └──────── Mode (GCM = Galois Counter Mode)
 │     │     │        └───────────── Key length (256 bits)
 │     │     └────────────────────── Encryption algorithm (AES)
 │     └──────────────────────────── Auth algorithm (RSA)
 └────────────────────────────────── Key exchange (ECDHE)
```

**Key components:**
- **Key Exchange (ECDHE)** — How client and server securely agree on a shared secret. ECDHE provides **perfect forward secrecy** (PFS) — even if the private key is stolen later, past sessions can't be decrypted.
- **Authentication (RSA/ECDSA)** — How the server proves its identity using the certificate.
- **Encryption (AES-256-GCM)** — The symmetric cipher used after the handshake.
- **Hash (SHA-384)** — Integrity check to detect tampering.

**TLS 1.3 approved cipher suites (simplified, only the best):**
```
TLS_AES_256_GCM_SHA384       ← Preferred
TLS_CHACHA20_POLY1305_SHA256 ← Good for mobile (faster on CPUs without AES hardware)
TLS_AES_128_GCM_SHA256       ← Acceptable
```

---

### How Your Backend Uses TLS

```
Without TLS (direct server):
  Client → [plain HTTP] → Your Node/Python/Go server on :80

With TLS terminated at load balancer (most common in production):
  Client → [HTTPS] → Load Balancer/Nginx (terminates TLS) → [HTTP] → Your server on :8080

With TLS terminated at your server:
  Client → [HTTPS] → Your server on :443 (handles TLS itself)
```

**Nginx TLS config example:**
```nginx
server {
    listen 443 ssl;
    server_name api.example.com;

    ssl_certificate     /etc/letsencrypt/live/api.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/api.example.com/privkey.pem;

    ssl_protocols TLSv1.2 TLSv1.3;          # Only allow modern TLS
    ssl_ciphers HIGH:!aNULL:!MD5;            # Disable weak ciphers
    ssl_prefer_server_ciphers on;
}
```

---

### Interview-Ready Summary

> "HTTPS is HTTP wrapped in TLS encryption. TLS ensures confidentiality (data is encrypted), integrity (data isn't tampered), and authentication (server identity is verified via certificates). The TLS handshake in 1.2 takes 2 round trips — the client and server negotiate cipher suites, the server presents its certificate, they exchange keys, and derive session keys. TLS 1.3 improved this to 1-RTT with 0-RTT session resumption. Certificates are signed by trusted Certificate Authorities (like Let's Encrypt) forming a chain of trust. Cipher suites define the algorithms used — I always prefer ECDHE for key exchange because it provides perfect forward secrecy."

## 3. TCP vs UDP

### Overview

Both TCP and UDP are **Layer 4 (Transport Layer)** protocols that sit on top of IP. They define how data is packaged and delivered between two endpoints.

**Simple distinction:**
- **TCP** — like a phone call. You establish a connection, talk, and hang up. Every word is heard in order.
- **UDP** — like shouting across a room. You just shout. Maybe they hear you, maybe not. No guarantee of order.

---

### TCP (Transmission Control Protocol)

TCP provides a **reliable, ordered, error-checked** stream of bytes between two endpoints.

#### The 3-Way Handshake (Connection Establishment)

Before any data flows, TCP must establish a connection:

```
CLIENT                          SERVER
  |                               |
  |------- SYN ------------------->|   "Hey, I want to connect"
  |        Seq=100                 |   (SYN = Synchronize sequence numbers)
  |                               |
  |<------ SYN-ACK ---------------|   "OK, I'm ready. Here's my seq number"
  |        Seq=200, Ack=101        |
  |                               |
  |------- ACK ------------------->|   "Got it. Let's go!"
  |        Ack=201                 |
  |                               |
  |====== Data Exchange ========= |
```

This adds **1.5 round trips** of latency before any data is sent.

#### TCP Connection Teardown (4-Way)

```
CLIENT                          SERVER
  |------- FIN ------------------->|   "I'm done sending"
  |<------ ACK -------------------|   "Got it"
  |<------ FIN -------------------|   "I'm done too"
  |------- ACK ------------------->|   "Acknowledged. Connection closed."
```

#### TCP Key Features

**1. Reliability (Acknowledgment + Retransmission)**
```
Sender sends: Packet 1, Packet 2, Packet 3
Receiver ACKs: "Got 1", "Got 2", (Packet 3 lost)
Sender waits for ACK of 3 → timeout → retransmits Packet 3
```

**2. Ordering (Sequence Numbers)**
```
Packets might arrive: 3, 1, 2 (out of order)
TCP reorders them:    1, 2, 3 before delivering to app
```

**3. Flow Control (Sliding Window)**
- Prevents the sender from overwhelming the receiver.
- Receiver tells sender: "I can only handle 64KB right now."
- Sender limits how much it sends.
```
Receiver: "My buffer has space for 65535 bytes" → Window Size
Sender: "I won't send more than 65535 bytes before getting an ACK"
```

**4. Congestion Control**
- Prevents sender from overwhelming the **network** (not just receiver).

```
Slow Start:
  - Start with a small window, double it each RTT
  - Window: 1 → 2 → 4 → 8 → 16 segments...
  - Until threshold reached, then grows linearly

Congestion detected (packet loss / timeout):
  - Cut window in half (Congestion Avoidance)
  - Start growing again linearly
```

This is called **AIMD** (Additive Increase, Multiplicative Decrease).

---

### UDP (User Datagram Protocol)

UDP is **connectionless** and **unreliable** — it just sends packets (datagrams) without any setup or guarantees.

```
CLIENT                          SERVER
  |                               |
  |--- Datagram 1 ---------------->|   Fire and forget
  |--- Datagram 2 ---------------->|   No handshake
  |--- Datagram 3 ---------------->|   No ACK
  |                               |   May arrive out of order
  |                               |   May not arrive at all
```

**UDP Header is tiny (8 bytes):**
```
Source Port (2 bytes)
Destination Port (2 bytes)
Length (2 bytes)
Checksum (2 bytes)
Data...
```

Compare to TCP header which is 20+ bytes. UDP has much lower overhead.

---

### TCP vs UDP Comparison

| Feature | TCP | UDP |
|---------|-----|-----|
| Connection | Connection-oriented (3-way handshake) | Connectionless |
| Reliability | Guaranteed delivery | No guarantee |
| Ordering | In-order delivery | No ordering |
| Speed | Slower (overhead) | Faster |
| Header Size | 20–60 bytes | 8 bytes |
| Error Checking | Yes (checksums + retransmit) | Checksum only (no retransmit) |
| Flow Control | Yes (sliding window) | No |
| Congestion Control | Yes (slow start, AIMD) | No |
| Use Cases | HTTP, HTTPS, SSH, databases | DNS, video streaming, gaming, VoIP |

---

### When to Use Which?

**Use TCP when:**
- Data **must** arrive and be **correct** (financial transactions, file transfers, API calls, databases, emails).
- Order matters (you're streaming sequential log data).
- You can tolerate a small latency increase for reliability.

```
✅ REST APIs         → TCP (HTTP)
✅ Database queries  → TCP (MySQL, PostgreSQL, Redis)
✅ File downloads    → TCP (FTP, SFTP)
✅ SSH connections   → TCP
✅ Email (SMTP)      → TCP
```

**Use UDP when:**
- Speed and low latency matter more than perfect delivery.
- Losing a packet is acceptable (old video frame vs new frame).
- You implement your own reliability logic (QUIC does this).

```
✅ DNS lookups       → UDP (fast, tiny request/response)
✅ Video streaming   → UDP (a dropped frame is OK, just show the next one)
✅ Online gaming     → UDP (old position data is useless anyway)
✅ VoIP / Video call → UDP (a gap in audio/video is better than delay)
✅ IoT sensors       → UDP (fire-and-forget sensor readings)
✅ HTTP/3 (QUIC)     → UDP (implements its own reliability)
```

---

### TCP in Backend: What You Actually Deal With

**Connection pooling:**
```
❌ Bad: Open new TCP connection for every DB query
        (3-way handshake adds latency each time)

✅ Good: Use a connection pool — reuse existing TCP connections
         pg Pool in Node.js, HikariCP in Java, SQLAlchemy pool in Python
```

**Timeout settings (critical for production):**
```
connect_timeout  → How long to wait for TCP handshake (e.g., 5s)
read_timeout     → How long to wait for data after connected (e.g., 30s)
write_timeout    → How long to wait to send data (e.g., 10s)
idle_timeout     → Close idle connections after (e.g., 60s)
```

**TCP Keep-Alive:**
```
Problem: A connection sits idle for 10 minutes. Firewall silently drops it.
         But client and server don't know!

Solution: TCP Keep-Alive — send a small probe packet every N seconds
          to keep the connection alive and detect dead peers.
```

---

### Interview-Ready Summary

> "TCP is a reliable, ordered, connection-oriented protocol. It uses a 3-way handshake to establish connections, acknowledgments and retransmission for reliability, sequence numbers for ordering, a sliding window for flow control, and slow start / AIMD for congestion control. UDP is connectionless and unreliable — just sends datagrams with zero setup overhead, making it much faster. I use TCP for anything where correctness matters — HTTP, databases, SSH. I use UDP where speed matters and some loss is acceptable — DNS, video streaming, gaming, and VoIP. HTTP/3 (QUIC) is interesting because it implements reliability on top of UDP, giving us the best of both worlds."

## 4. WebSockets

### What is a WebSocket?

**WebSocket** is a protocol that provides a **persistent, full-duplex** (two-way) communication channel over a single TCP connection.

**The problem HTTP has:**
```
HTTP: Client must always initiate. Server can't push data to client.
      Client polls every N seconds: "Any new messages?" — wasteful!

WebSocket: Once connected, BOTH sides can send data at any time.
           Server can push: "Hey, new message for you!" — efficient!
```

**Analogy:**
> HTTP is like sending letters — you write, mail, wait, get a reply. Each letter is independent. WebSocket is like a phone call — once connected, both sides can speak and listen simultaneously, anytime.

---

### HTTP vs WebSocket

```
HTTP (Request-Response):
Client: "Any new messages?"  → Server: "No"  (wasted trip)
Client: "Any new messages?"  → Server: "No"  (wasted trip)
Client: "Any new messages?"  → Server: "Yes! Here's the message"

WebSocket (Persistent + Push):
[Connection established once]
Server: "New message: Hello!"   (instant, no polling)
Server: "New message: World!"   (instant)
Client: "Got it, here's a reply" (instant)
```

---

### WebSocket Handshake (How it Starts)

WebSocket starts as an **HTTP request** and then **upgrades**:

```
CLIENT                                          SERVER
  |                                               |
  |--- HTTP Upgrade Request ---------------------->|
  |    GET /chat HTTP/1.1                         |
  |    Host: example.com                          |
  |    Upgrade: websocket                         |
  |    Connection: Upgrade                        |
  |    Sec-WebSocket-Key: dGhlIHNhbXBsZQ==        |
  |    Sec-WebSocket-Version: 13                  |
  |                                               |
  |<-- HTTP 101 Switching Protocols ---------------|
  |    HTTP/1.1 101 Switching Protocols           |
  |    Upgrade: websocket                         |
  |    Connection: Upgrade                        |
  |    Sec-WebSocket-Accept: s3pPLMBiT...         |
  |                                               |
  |====== WebSocket Frames (Full-Duplex) ========|
  |<-- Server Push: {"event": "message", ...} ----|
  |--- Client Send: {"type": "ping"} ------------>|
  |<-- Server Push: {"event": "typing"} ----------|
```

After the 101 response, the HTTP connection is **hijacked** and becomes a WebSocket connection. No more HTTP — raw WebSocket frames from now on.

---

### WebSocket Frame Structure

WebSocket sends data as **frames** (not HTTP requests):

```
Frame:
┌─────────────────────────────────────────────────┐
│ FIN | RSV | Opcode | Mask | Payload Length      │
│ Payload Data...                                  │
└─────────────────────────────────────────────────┘

Opcode types:
  0x0 → Continuation frame
  0x1 → Text frame (UTF-8 string)
  0x2 → Binary frame
  0x8 → Close connection
  0x9 → Ping
  0xA → Pong
```

- Client frames must be **masked** (XOR with a random key) — prevents cache poisoning attacks.
- Server frames are **unmasked**.

---

### WebSocket in Practice (Node.js Example)

**Server side:**
```javascript
const WebSocket = require('ws');
const wss = new WebSocket.Server({ port: 8080 });

wss.on('connection', (ws) => {
  console.log('Client connected');

  // Receive message from client
  ws.on('message', (message) => {
    console.log('Received:', message);

    // Broadcast to all connected clients
    wss.clients.forEach((client) => {
      if (client.readyState === WebSocket.OPEN) {
        client.send(JSON.stringify({ event: 'message', data: message }));
      }
    });
  });

  ws.on('close', () => console.log('Client disconnected'));

  // Push data to client anytime (server-initiated!)
  ws.send(JSON.stringify({ event: 'welcome', data: 'Connected!' }));
});
```

**Client side:**
```javascript
const ws = new WebSocket('wss://example.com/chat');

ws.onopen = () => ws.send(JSON.stringify({ type: 'join', room: 'general' }));
ws.onmessage = (event) => console.log('Got:', JSON.parse(event.data));
ws.onclose = () => console.log('Disconnected');
ws.onerror = (err) => console.error('Error:', err);
```

### WebSocket vs HTTP Polling vs SSE

| Feature | Short Polling | Long Polling | SSE | WebSocket |
|---------|---------------|--------------|-----|-----------|
| Protocol | HTTP | HTTP | HTTP | WS |
| Direction | Client → Server | Client → Server | Server → Client | Both ways |
| Connection | New each time | Held open | Persistent | Persistent |
| Latency | High | Medium | Low | Very Low |
| Overhead | High | Medium | Low | Low |
| Use case | Simple updates | Chat (old way) | Live feeds | Real-time bidirectional |

**SSE (Server-Sent Events):** Server pushes events to client over HTTP. One-way only. Great for live dashboards, notifications. WebSocket is for bidirectional (chat, gaming, collaboration).

✅ Chat applications (WhatsApp Web, Slack)
✅ Live sports scores / stock price tickers
✅ Collaborative tools (Google Docs, Figma)
✅ Multiplayer games
✅ Live notifications

### WebSocket Gotchas for Backend Devs

**Scaling WebSockets is hard:**
```
Problem: WebSocket connections are STATEFUL.
         User A is connected to Server 1.
         User B is connected to Server 2.
         User A sends message to User B.
         Server 1 doesn't know where User B is!

Solution: Use a message broker (Redis Pub/Sub, Kafka) to relay messages between servers.

Server 1 (User A) ──publishes──► Redis ──delivers──► Server 2 (User B)
```

**Reconnection logic (client must handle):**
```javascript
function connectWebSocket() {
  const ws = new WebSocket('wss://example.com/ws');

  ws.onclose = () => {
    console.log('Disconnected, reconnecting in 3s...');
    setTimeout(connectWebSocket, 3000);  // Auto-reconnect
  };
}
```

**Heartbeat / Ping-Pong:**
```
Problem: Idle connections may be dropped by load balancers or proxies.
Solution: Send a ping frame every 30 seconds.
          If no pong received, close and reconnect.
```

---

### Interview-Ready Summary

> "WebSocket provides persistent, full-duplex communication over a single TCP connection. It starts as an HTTP request with an `Upgrade: websocket` header — once the server responds with 101 Switching Protocols, the connection is kept open and both sides can send messages anytime without the overhead of HTTP headers. This is perfect for real-time use cases like chat, live notifications, collaborative editing, and gaming. The key challenge in production is scaling — since WebSocket connections are stateful, you need Redis Pub/Sub or a message broker to relay messages across multiple server instances."

## 5. gRPC & Protobuf

### What is gRPC?

**gRPC** (Google Remote Procedure Call) is a high-performance, open-source framework for **service-to-service communication**. Instead of HTTP + JSON (REST), it uses:
- **HTTP/2** as the transport layer.
- **Protocol Buffers (Protobuf)** as the serialization format (binary, not JSON).

**Analogy:**
> REST is like communicating by writing English letters (human-readable, flexible, but verbose). gRPC is like speaking a strict, compressed code language — both sides must know the codebook upfront, but communication is much faster and more efficient.

---

### Why gRPC Over REST?

| Feature | REST (HTTP/JSON) | gRPC (HTTP/2 + Protobuf) |
|---------|-----------------|--------------------------|
| Protocol | HTTP/1.1 or 2 | HTTP/2 always |
| Data Format | JSON (text) | Protobuf (binary) |
| Performance | Slower (verbose) | 5–10x faster |
| Schema | Optional (OpenAPI) | Required (.proto file) |
| Code Generation | Manual or tools | Auto-generated |
| Streaming | Limited (SSE/WS) | Built-in (4 types) |
| Browser Support | ✅ Native | ❌ Needs grpc-web |
| Error Handling | HTTP status codes | gRPC status codes |

**When to use gRPC:**
```
✅ Internal microservice-to-service communication
✅ High-throughput, low-latency scenarios
✅ Polyglot environments (Go service talking to Python service)
✅ Streaming data (logs, sensor data, chat)
✅ Mobile clients (bandwidth efficient)

❌ Public APIs (browser support limited)
❌ Simple CRUD with no performance needs
❌ Teams unfamiliar with Protobuf toolchain
```

---

### Protocol Buffers (Protobuf)

**Protobuf** is a language-neutral, platform-neutral way to **serialize structured data**. You define a schema in a `.proto` file, and Protobuf generates code for any language.

**Step 1: Define your schema (.proto file)**

```protobuf
// user.proto
syntax = "proto3";

package user;

// Define the service (what methods it exposes)
service UserService {
  rpc GetUser (GetUserRequest) returns (User);
  rpc CreateUser (CreateUserRequest) returns (User);
  rpc ListUsers (ListUsersRequest) returns (stream User);  // server streaming
}

// Define message types (like JSON schema)
message User {
  int32 id = 1;           // field number (NOT value)
  string name = 2;
  string email = 3;
  bool is_active = 4;
  repeated string roles = 5;  // array of strings
}

message GetUserRequest {
  int32 id = 1;
}

message CreateUserRequest {
  string name = 1;
  string email = 2;
}

message ListUsersRequest {
  int32 page = 1;
  int32 page_size = 2;
}
```

**Step 2: Generate code**

```bash
# Generate Go code from .proto
protoc --go_out=. --go-grpc_out=. user.proto

# Generate Python code
python -m grpc_tools.protoc -I. --python_out=. --grpc_python_out=. user.proto

# Generate Node.js code
protoc --js_out=. --grpc-web_out=. user.proto
```

This generates server and client code automatically!

---

### Why Binary is Faster Than JSON

```
JSON (text):
{ "id": 1, "name": "Alice", "email": "alice@example.com", "is_active": true }
→ 71 bytes, needs parsing (string to object)

Protobuf (binary):
08 01 12 05 41 6C 69 63 65 1A 11 61 6C 69 63 65 40 65 78 61 6D 70 6C 65 2E 63 6F 6D 20 01
→ ~30 bytes, much faster to parse (direct memory mapping)
```

**Protobuf field encoding:**
```
Field 1 (id=1):    08 01       → tag=1, type=varint, value=1
Field 2 (name):    12 05 "Alice" → tag=2, type=string, length=5, bytes
Field 4 (active):  20 01       → tag=4, type=varint, value=true(1)
```

Notice: **field names are NOT in the binary output** — just numbers. This is why Protobuf is compact. The `.proto` schema maps numbers back to field names.

---

### gRPC Communication Patterns

gRPC supports **4 types of RPC** (HTTP/2 streams make this possible):

**1. Unary RPC (like a REST call)**
```
Client: GetUser(id=1) ──────────────────────────────────────► Server
Client:                ◄──────────────────── User{id:1,...} ── Server
```

**2. Server Streaming RPC**
```
Client: ListUsers(page=1) ──────────────────────────────────► Server
Client: ◄── User{1} ── User{2} ── User{3} ── ... ── Server
(server streams many responses to one request)
```

**3. Client Streaming RPC**
```
Client: Upload(chunk1) ──────────────────────────────────────► Server
Client: Upload(chunk2) ──────────────────────────────────────► Server
Client: Upload(chunk3) ──────────────────────────────────────► Server
Client:                ◄──────────────── UploadResult{...} ── Server
(client streams many requests, server sends one response)
```

**4. Bidirectional Streaming RPC**
```
Client: ◄──────────────────────────────────────────────────── Server
Client: ──── msg1 ───────────────────────────────────────────► Server
Client: ◄──────────────────── response1 ──────────────────── Server
Client: ──── msg2 ───────────────────────────────────────────► Server
(both sides stream simultaneously — like WebSocket but typed)
```

---

### gRPC Server Example (Node.js)

```javascript
const grpc = require('@grpc/grpc-js');
const protoLoader = require('@grpc/proto-loader');

const packageDef = protoLoader.loadSync('user.proto');
const proto = grpc.loadPackageDefinition(packageDef).user;

// Implement the service methods
const server = new grpc.Server();
server.addService(proto.UserService.service, {
  getUser: (call, callback) => {
    const { id } = call.request;
    const user = db.findUser(id);
    callback(null, user);                      // return response
  },

  listUsers: (call) => {
    const users = db.getUsers();
    users.forEach(user => call.write(user));   // stream responses
    call.end();
  }
});

server.bindAsync('0.0.0.0:50051', grpc.ServerCredentials.createInsecure(), () => {
  server.start();
  console.log('gRPC server running on :50051');
});
```

---

### gRPC Status Codes

gRPC has its own status codes (different from HTTP):

```
OK (0)                → Success
CANCELLED (1)         → Client cancelled the request
UNKNOWN (2)           → Unknown error
INVALID_ARGUMENT (3)  → Bad input (like 400)
NOT_FOUND (5)         → Resource not found (like 404)
ALREADY_EXISTS (6)    → Duplicate (like 409)
PERMISSION_DENIED (7) → No permission (like 403)
UNAUTHENTICATED (16)  → Not authenticated (like 401)
UNAVAILABLE (14)      → Service down (like 503)
DEADLINE_EXCEEDED (4) → Timeout (like 504)
```

---

### Interview-Ready Summary

> "gRPC is a high-performance RPC framework that uses HTTP/2 for transport and Protocol Buffers (Protobuf) for serialization. It's significantly faster than REST+JSON because Protobuf is a compact binary format (no field names in the wire format, just field numbers), HTTP/2 provides multiplexing, and code is auto-generated from `.proto` schema files. gRPC supports 4 RPC types: unary (like REST), server streaming, client streaming, and bidirectional streaming. It's ideal for internal microservice communication, polyglot environments, and high-throughput scenarios. The tradeoff is that it's not browser-friendly without a proxy like grpc-web, and teams need to learn the Protobuf toolchain."

## 6. SMTP / IMAP

### Overview

Email protocols define how emails are **sent**, **received**, and **stored**. As a backend developer building notification systems, password resets, or transactional emails, you need to understand these.

```
SMTP  → Sending email (Simple Mail Transfer Protocol)
IMAP  → Reading email from server (Internet Message Access Protocol)
POP3  → Downloading email from server (Post Office Protocol 3) — older
```

---

### SMTP — Sending Email

**SMTP** is used to **send and relay** email messages. Your app uses SMTP to send emails via a mail server (Gmail, SendGrid, AWS SES, Mailgun, etc.).

**How email travels from sender to recipient:**

```
Your App (sender)
    │
    │ SMTP (port 587, STARTTLS)
    ▼
Your SMTP Server (e.g., AWS SES, SendGrid)
    │
    │ SMTP (port 25, server-to-server)
    ▼
Recipient's Mail Server (e.g., Gmail)
    │
    │ IMAP/POP3 (port 993/995)
    ▼
Recipient's Email Client (Gmail app, Outlook)
```

**SMTP Ports:**
```
Port 25   → Server-to-server (MTA-to-MTA) email relay. Often blocked by ISPs.
Port 465  → SMTP over SSL (SMTPS). Legacy but still used.
Port 587  → SMTP with STARTTLS. Modern standard for client submission.
Port 2525 → Alternative to 587 (some cloud providers use this).
```

**SMTP Conversation (raw protocol):**
```
CLIENT (your app)                    SERVER (mail server)
  |                                      |
  |                                      220 smtp.gmail.com ESMTP
  |--- EHLO myapp.example.com ---------->|   (hello, identify yourself)
  |<-- 250-smtp.gmail.com ---------------|   (here are my capabilities)
  |<-- 250-STARTTLS                      |
  |<-- 250-AUTH LOGIN PLAIN              |
  |                                      |
  |--- STARTTLS ------------------------->|   (upgrade to TLS)
  |<-- 220 Ready to start TLS ------------|
  |    [TLS Handshake]                   |
  |                                      |
  |--- AUTH LOGIN ----------------------->|   (authenticate)
  |--- username (base64 encoded) -------->|
  |--- password (base64 encoded) -------->|
  |<-- 235 Authentication successful -----|
  |                                      |
  |--- MAIL FROM:<app@example.com> ------->|   (who's sending)
  |<-- 250 OK                             |
  |--- RCPT TO:<user@gmail.com> ---------->|   (who's receiving)
  |<-- 250 OK                             |
  |--- DATA -------------------------------->|   (start the email)
  |<-- 354 Start input                    |
  |--- From: app@example.com              |   (email headers + body)
  |--- To: user@gmail.com                 |
  |--- Subject: Your OTP is 123456        |
  |---                                    |   (blank line separates headers/body)
  |--- Your OTP is 123456. Valid 10 mins. |
  |--- . --------------------------------->|   (single dot = end of message)
  |<-- 250 OK: Message queued             |
  |--- QUIT -------------------------------->|
  |<-- 221 Bye                            |
```

---

### Sending Email from Backend (Node.js with Nodemailer)

```javascript
const nodemailer = require('nodemailer');

// Create transporter (SMTP connection)
const transporter = nodemailer.createTransport({
  host: 'smtp.gmail.com',
  port: 587,
  secure: false,     // Use STARTTLS (true = SSL on port 465)
  auth: {
    user: 'your@gmail.com',
    pass: 'your-app-password'   // Not your Gmail password! Use App Password.
  }
});

// Send email
async function sendOTP(userEmail, otp) {
  const info = await transporter.sendMail({
    from: '"MyApp" <no-reply@myapp.com>',
    to: userEmail,
    subject: 'Your OTP Code',
    text: `Your OTP is: ${otp}. Valid for 10 minutes.`,
    html: `<b>Your OTP is: ${otp}</b><p>Valid for 10 minutes.</p>`
  });

  console.log('Email sent:', info.messageId);
}
```

**In production, use a transactional email provider:**
```
SendGrid   → sendgrid.com  (popular, generous free tier)
AWS SES    → Very cheap (cost-effective at scale)
Mailgun    → Developer-friendly
Postmark   → Best deliverability for transactional email
Resend     → Modern, developer-first
```

---

### Email Authentication (SPF, DKIM, DMARC)

Without these, your emails land in spam. These are critical for any production notification system.

**SPF (Sender Policy Framework):**
> "Only these servers are allowed to send email for my domain."

```
DNS TXT Record:
example.com. IN TXT "v=spf1 include:sendgrid.net include:amazonses.com ~all"
                                  ↑                     ↑               ↑
                            SendGrid allowed      SES allowed    soft fail others
```

**DKIM (DomainKeys Identified Mail):**
> "I sign every email with my private key. Verify with my public key in DNS."

```
Email header added:
DKIM-Signature: v=1; a=rsa-sha256; d=example.com; s=selector1;
                h=from:to:subject; bh=hash_of_body; b=signature...

DNS TXT Record (public key):
selector1._domainkey.example.com. IN TXT "v=DKIM1; k=rsa; p=MIGfMA0G..."
```

**DMARC (Domain-based Message Authentication, Reporting & Conformance):**
> "What should receivers do with emails that fail SPF or DKIM?"

```
DNS TXT Record:
_dmarc.example.com. IN TXT "v=DMARC1; p=quarantine; rua=mailto:dmarc@example.com"
                                          ↑                     ↑
                                    quarantine/reject/none    report failures here
```

---

### IMAP — Reading Email

**IMAP (Internet Message Access Protocol)** lets an email client **read and manage emails** stored on a mail server. It's for your users reading email — your backend usually doesn't implement an IMAP server (unless you're building an email client).

**IMAP vs POP3:**

| Feature | IMAP | POP3 |
|---------|------|------|
| Email stored | On server | Downloaded locally |
| Multi-device | ✅ (read on phone + laptop) | ❌ (downloaded to one device) |
| Port | 993 (SSL), 143 (STARTTLS) | 995 (SSL), 110 (plain) |
| Use today | Standard | Legacy |

**IMAP features:**
- Emails remain on server — read from multiple devices.
- Supports **folders/labels** (INBOX, Sent, Drafts, Spam).
- **Partial fetch** — download just the email header without body.
- **IDLE command** — server pushes notifications of new email (no polling).

**When a backend dev uses IMAP:**
```
Reading inbound email:
  - Support ticket system (read customer emails)
  - Email-to-action pipeline (parse emails to create tasks)
  - Email forwarding/routing systems
```

**Node.js IMAP example:**
```javascript
const Imap = require('imap');

const imap = new Imap({
  user: 'support@example.com',
  password: 'password',
  host: 'imap.gmail.com',
  port: 993,
  tls: true
});

imap.once('ready', () => {
  imap.openBox('INBOX', false, (err, box) => {
    // Search for unseen emails
    imap.search(['UNSEEN'], (err, results) => {
      const fetch = imap.fetch(results, { bodies: '' });
      fetch.on('message', (msg) => {
        msg.on('body', (stream) => {
          // Process email stream
          stream.pipe(process.stdout);
        });
      });
    });
  });
});

imap.connect();
```

### Email in Production: What Backend Devs Really Do

```
Scenario 1: Transactional emails (OTP, password reset, invoices)
  → Use SendGrid/SES SDK → they handle SMTP for you
  → Set up SPF, DKIM, DMARC in DNS
  → Track opens/clicks via webhooks

Scenario 2: Notification system
  → User triggers action → queue message → worker sends email via SMTP
  → Store email status in DB (sent/failed/bounced)
  → Handle bounces and unsubscribes (CAN-SPAM compliance)

Scenario 3: Inbound email processing
  → User replies to notification email → parse it → create action
  → Use IMAP to poll inbox, or use inbound email webhook (SendGrid/Mailgun)
```

**Handling bounces and complaints (critical!):**
```
Hard bounce → Invalid email address → Remove from DB immediately
Soft bounce → Temporary issue (mailbox full) → Retry 3x then remove
Complaint   → User marked as spam → Remove and never email again
```

---

### Interview-Ready Summary

> "SMTP is the protocol for sending and relaying email — my app connects to an SMTP server (like SendGrid or AWS SES) on port 587 with STARTTLS, authenticates, and sends the message. For any production email system, I configure SPF (which servers can send for my domain), DKIM (cryptographic signature on each email), and DMARC (policy for failed auth) as DNS records — without these, emails land in spam. IMAP is for reading email stored on a server and supports multi-device access, unlike the older POP3 which downloads and removes emails. In backend systems, I use transactional email providers like SendGrid or AWS SES for reliability, track delivery status via webhooks, and properly handle bounces and complaints to protect sender reputation."
