## 1. Service Discovery

### The Problem Service Discovery Solves

In a monolith, everything is in one process — no need to find anything. In microservices, services need to talk to each other, and finding WHERE a service lives is non-trivial.

```
Monolith (no discovery needed):
  UserModule.getOrders(userId)  ← direct function call, always works

Microservices (discovery needed):
  Order Service needs to call Payment Service
  But WHERE is Payment Service?
    → IP changes when container restarts
    → Multiple instances run simultaneously (which one?)
    → Instances scale up and down dynamically
    → New version deploys replace old IPs

  You can't hardcode IPs in microservices. They change constantly.
```

**Analogy:**
> Service discovery is like a company's internal phone directory. Instead of memorizing everyone's extension (IP:port), you look up "Alice in Payments" and get her current extension. If Alice moves desks (new IP), the directory is updated — everyone else automatically finds her at the new location.

---

### Two Models of Service Discovery

**Client-Side Discovery:**
```
Service A (client)
  1. Queries service registry: "Where is Payment Service?"
  2. Registry returns: [10.0.1.5:8080, 10.0.1.6:8080, 10.0.1.7:8080]
  3. Client picks one (round-robin, random, etc.)
  4. Calls directly: 10.0.1.5:8080/charge

  Client KNOWS about all instances and does its own load balancing.
  Examples: Netflix Ribbon (with Eureka), Consul + client libraries

  ✅ No extra hop (client calls server directly)
  ✅ Client can implement smart routing (latency-based, zone-aware)
  ❌ Every language/service needs discovery logic built in
  ❌ More complex client code
```

**Server-Side Discovery:**
```
Service A (client)
  1. Calls: payment-service/charge (logical name, not IP)
  2. DNS or Load Balancer resolves the name → picks an instance
  3. Routes to: 10.0.1.5:8080

  Client doesn't know about registry — just calls a name.
  Examples: Kubernetes Services, AWS ALB, Nginx + Consul Template

  ✅ Client is simple (just use a hostname)
  ✅ Discovery logic is in the infrastructure layer
  ❌ Extra network hop through load balancer
  ❌ Less control for the client
```

### Consul

**Consul** (by HashiCorp) is a full-featured service discovery and configuration tool. Works across any infrastructure — bare metal, VMs, containers, cloud.

#### How Consul Works

```
┌─────────────────────────────────────────────────────────────────┐
│                        Consul Cluster                            │
│                                                                  │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐                    │
│  │ Server 1 │   │ Server 2 │   │ Server 3 │  ← Raft consensus  │
│  │ (leader) │   │          │   │          │    (odd number!)    │
│  └──────────┘   └──────────┘   └──────────┘                    │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │                    Service Registry                       │   │
│  │  payment-service: [10.0.1.5:8080, 10.0.1.6:8080]       │   │
│  │  user-service:    [10.0.2.1:3000]                        │   │
│  │  order-service:   [10.0.3.1:4000, 10.0.3.2:4000]       │   │
│  └──────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
          ↑ register/deregister          ↑ query
   Consul Agents (on every node)    Services/Clients

Each service node runs a Consul Agent:
  → Registers the service with the cluster on startup
  → Runs health checks continuously
  → Deregisters on shutdown or health check failure
```

**Service Registration (via HTTP API):**
```json
// POST http://consul:8500/v1/agent/service/register
{
  "ID":      "payment-service-1",
  "Name":    "payment-service",
  "Address": "10.0.1.5",
  "Port":    8080,
  "Tags":    ["v2", "production"],
  "Check": {
    "HTTP":     "http://10.0.1.5:8080/health",
    "Interval": "10s",
    "Timeout":  "3s",
    "DeregisterCriticalServiceAfter": "30s"
  }
}
```

**Service Discovery Query:**
```bash
# DNS query (simplest — any language can use DNS)
dig @127.0.0.1 -p 8600 payment-service.service.consul
# Returns: A records for all healthy payment-service instances

# With SRV record (includes port)
dig @127.0.0.1 -p 8600 payment-service.service.consul SRV

# HTTP API query
curl http://consul:8500/v1/health/service/payment-service?passing=true
# Returns: list of healthy instances with their IPs and ports

# Filter by tag
curl http://consul:8500/v1/health/service/payment-service?tag=v2&passing=true
```

**Node.js example — querying Consul:**
```javascript
const Consul = require('consul');
const consul = new Consul({ host: 'consul.internal', port: 8500 });

async function getPaymentServiceUrl() {
  const services = await consul.health.service({
    service: 'payment-service',
    passing: true   // Only healthy instances
  });

  if (services.length === 0) {
    throw new Error('No healthy payment-service instances found');
  }

  // Simple round-robin (in real life: use client-side LB library)
  const service = services[Math.floor(Math.random() * services.length)];
  const { Address, Port } = service.Service;
  return `http://${Address}:${Port}`;
}

// Usage
const baseUrl = await getPaymentServiceUrl();
const result = await axios.post(`${baseUrl}/charge`, { amount: 99 });
```

**Consul Key-Value Store (bonus feature):**
```javascript
// Store config in Consul K/V
await consul.kv.set('config/payment-service/fee-percentage', '2.5');

// Read config (with watch — auto-updates when value changes!)
const watch = consul.watch({
  method: consul.kv.get,
  options: { key: 'config/payment-service/fee-percentage' }
});

watch.on('change', (data) => {
  feePercentage = parseFloat(data.Value);
  console.log(`Fee updated to: ${feePercentage}%`);
});
```

### Kubernetes DNS

Kubernetes has **built-in service discovery** via DNS — no external tool needed. Every `Service` object gets a DNS name automatically.

#### Kubernetes Service Types

```
┌──────────────────────────────────────────────────────────────────┐
│  Kubernetes Cluster                                               │
│                                                                   │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │  Namespace: production                                       │ │
│  │                                                              │ │
│  │  Service: payment-service  ←── DNS: payment-service.production.svc.cluster.local
│  │  ClusterIP: 10.96.0.100                                      │ │
│  │       ↓ load balances to                                     │ │
│  │  Pod: payment-pod-1  (10.244.1.5:8080)                      │ │
│  │  Pod: payment-pod-2  (10.244.1.6:8080)                      │ │
│  │  Pod: payment-pod-3  (10.244.2.1:8080)                      │ │
│  └─────────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────────┘
```

**DNS names in Kubernetes:**
```
Full name:  <service>.<namespace>.svc.cluster.local
Short name: <service>                    (within same namespace)
            <service>.<namespace>         (across namespaces)

Examples:
  payment-service.production.svc.cluster.local   ← full
  payment-service                                 ← within same namespace
  payment-service.production                      ← from other namespace

When order-service (in production namespace) calls payment-service:
  axios.get('http://payment-service/health')      ← just works!
  DNS resolves to ClusterIP → kube-proxy routes to a healthy Pod
```

**Kubernetes Service YAML:**
```yaml
# Service definition (the discovery endpoint)
apiVersion: v1
kind: Service
metadata:
  name: payment-service
  namespace: production
spec:
  selector:
    app: payment          # Routes to pods with label app=payment
  ports:
    - protocol: TCP
      port: 80            # Service port (what callers use)
      targetPort: 8080    # Pod port (what the container listens on)
  type: ClusterIP         # Internal only (default)
```

```yaml
# Pod/Deployment that the Service routes to
apiVersion: apps/v1
kind: Deployment
metadata:
  name: payment-deployment
spec:
  replicas: 3
  selector:
    matchLabels:
      app: payment        # Must match Service selector
  template:
    metadata:
      labels:
        app: payment
    spec:
      containers:
      - name: payment
        image: myrepo/payment-service:v2
        ports:
        - containerPort: 8080
```

**How Kubernetes DNS works under the hood:**
```
CoreDNS runs inside the cluster as a DNS server.

When a pod does: curl http://payment-service/charge
  1. Pod's DNS resolver queries CoreDNS: "payment-service.production.svc.cluster.local?"
  2. CoreDNS returns ClusterIP: 10.96.0.100
  3. Pod sends request to 10.96.0.100:80
  4. kube-proxy (iptables/IPVS rules) on the node intercepts it
  5. kube-proxy picks a healthy Pod IP (round-robin)
  6. Request delivered to Pod: 10.244.1.5:8080

The Service ClusterIP is VIRTUAL — no process actually listens on it.
kube-proxy intercepts traffic at the network level and redirects it.
```

**Service types:**
```
ClusterIP (default):
  Virtual IP only reachable within the cluster.
  Used for: internal service-to-service communication.

NodePort:
  Opens a port (30000-32767) on EVERY node.
  External: <node-ip>:<nodeport> → Service → Pod
  Used for: simple external access in dev/test.

LoadBalancer:
  Provisions a cloud load balancer (AWS ALB, GCP LB).
  Gets a public IP.
  Used for: production external access.

ExternalName:
  Maps to an external DNS name.
  mysql.production.svc.cluster.local → db.external-company.com
  Used for: migrating external dependencies to internal gradually.

Headless Service (ClusterIP: None):
  DNS returns Pod IPs directly (no virtual IP, no load balancing).
  Used for: StatefulSets (databases), when client needs direct pod access.
```

### Eureka (Netflix)

**Eureka** is Netflix's service registry, part of the Spring Cloud ecosystem. Used widely in Java/Spring Boot microservices.

```
Architecture:
  Eureka Server (registry)
    ↑ register (heartbeat every 30s)
  Eureka Clients (your services)
    ↑ fetch registry (every 30s, cached locally)
  Client-side load balancing (Ribbon/Spring Cloud LoadBalancer)
    picks instance from local registry cache

Self-preservation mode:
  If Eureka stops receiving heartbeats from many services simultaneously,
  it assumes its own network is failing (not the services).
  It stops evicting instances to prevent false removals.
  Good for: network partitions
  Bad for:  makes it hard to quickly remove dead instances
```

**Spring Boot registration (auto-configured):**
```yaml
# application.yml
spring:
  application:
    name: payment-service   # Service name in registry

eureka:
  client:
    serviceUrl:
      defaultZone: http://eureka-server:8761/eureka/
  instance:
    preferIpAddress: true   # Register with IP, not hostname
    lease-renewal-interval-in-seconds: 10   # Heartbeat every 10s (default 30)
    lease-expiration-duration-in-seconds: 30 # Remove if no heartbeat for 30s
```

```java
// Calling another service (Spring Boot)
@Service
public class OrderService {

    @Autowired
    private RestTemplate restTemplate;  // or WebClient

    public Payment chargeUser(String userId, BigDecimal amount) {
        // Just use service name — Ribbon resolves to an instance
        return restTemplate.postForObject(
            "http://payment-service/charge",   // "payment-service" → discovered!
            new ChargeRequest(userId, amount),
            Payment.class
        );
    }
}
```

### Interview-Ready Summary

> "Service discovery solves the problem of finding service instances in dynamic environments where IPs change constantly. There are two models: client-side (client queries the registry and picks an instance directly) and server-side (client uses a logical name, infrastructure resolves it). Consul is a language-agnostic service registry with DNS and HTTP interfaces, health check integration, and a K/V store for config — works on any infrastructure. Kubernetes has built-in service discovery through DNS: every Service object gets a DNS name like `payment-service.production.svc.cluster.local`, CoreDNS resolves it to a virtual ClusterIP, and kube-proxy routes traffic to healthy pods. Service types include ClusterIP (internal), NodePort (external dev), LoadBalancer (external prod), and Headless (direct pod access for stateful workloads). Eureka is Netflix's Java-centric registry with client-side load balancing via Ribbon — still common in Spring Boot ecosystems."

## 2. API Gateways

### What is an API Gateway?

An **API Gateway** is the single entry point for all client requests to your microservices. Instead of clients calling individual services directly, they call the gateway, which routes, transforms, and enforces policies.

**Without API Gateway:**
```
Mobile App → payment-service:8001
           → user-service:8002
           → order-service:8003
           → inventory-service:8004

Problems:
  ❌ Client needs to know every service's URL
  ❌ Auth logic duplicated in every service
  ❌ Rate limiting duplicated in every service
  ❌ CORS handling in every service
  ❌ Service changes break client directly
  ❌ N different SSL certs
```

**With API Gateway:**
```
Mobile App → API Gateway (api.myapp.com)
               ├── /users/*      → user-service
               ├── /orders/*     → order-service
               ├── /payments/*   → payment-service
               └── /inventory/*  → inventory-service

The gateway handles:
  ✅ Single HTTPS endpoint for all clients
  ✅ Authentication (verify JWT once, forward identity)
  ✅ Rate limiting (per client, per endpoint)
  ✅ Request routing (URL-based, header-based)
  ✅ SSL termination (one cert, one place)
  ✅ CORS (configure once)
  ✅ Request/response transformation
  ✅ Logging and tracing (every request logged centrally)
  ✅ Load balancing (across service instances)
```

**Analogy:**
> An API Gateway is like an airport's main terminal. All passengers (clients) arrive at one terminal (gateway). The terminal handles security checks (auth), routes passengers to the correct gate (routing), enforces baggage limits (rate limiting), and logs all arrivals (centralized logging). Airlines (microservices) don't need their own check-in counters for security.

---

### Core API Gateway Functions

#### 1. Routing

Route requests to the correct backend service based on URL path, host, headers, or method.

```
Path-based routing:
  GET  /api/users/42         → user-service:3000
  GET  /api/orders           → order-service:4000
  POST /api/payments/charge  → payment-service:8080
  GET  /api/products         → product-service:5000

Host-based routing:
  api.myapp.com/users        → user-service
  admin.myapp.com/users      → admin-user-service  (different service!)

Header-based routing:
  X-API-Version: v2          → new service version
  X-Beta-User: true          → beta feature service

Method-based routing:
  GET  /api/products → read-replica-service  (scale reads separately)
  POST /api/products → write-service
```

#### 2. Authentication & Authorization at the Edge

Verify identity ONCE at the gateway. Never let unauthenticated requests reach services.

```
Client → Gateway
  1. Extract JWT from Authorization header
  2. Verify JWT signature and expiry
  3. If invalid → return 401 immediately (never hits backend)
  4. If valid → extract claims (userId, roles)
  5. Add claims to forwarded request headers:
     X-User-ID: 42
     X-User-Role: admin
     X-User-Email: alice@example.com

Backend service receives:
  GET /orders
  X-User-ID: 42
  X-User-Role: admin

  Backend trusts these headers (internal network — no JWT verification needed!)
  Backend applies authorization: "User 42 with role admin can see all orders"
```

**Kong Gateway auth plugin example:**
```yaml
# Kong declarative config
plugins:
  - name: jwt
    config:
      secret_is_base64: false
      key_claim_name: iss
      claims_to_verify:
        - exp
        - nbf
```

#### 3. Rate Limiting at the Edge

Apply rate limits before requests hit your services. Protects entire system.

```
Rate limit strategies at gateway:
  Global:        100 req/min total (prevents DDoS)
  Per IP:        10 req/min per IP (anonymous users)
  Per API key:   1000 req/min per API key
  Per user:      200 req/min per userId (from JWT)
  Per endpoint:  /auth/login → 5 req/min (brute force protection)

Headers returned to client:
  X-RateLimit-Limit:     100
  X-RateLimit-Remaining: 37
  X-RateLimit-Reset:     1700000060
  Retry-After:           60  (on 429)
```

#### 4. Request/Response Transformation

Modify requests before forwarding to backend, and responses before returning to client.

```
Request transformations:
  → Strip the /api prefix:    /api/users → /users
  → Add headers:              X-Request-ID: uuid, X-Gateway: true
  → Strip sensitive headers:  Remove Cookie header before forwarding
  → Rewrite URLs:             /v1/users → /v2/users (API versioning)

Response transformations:
  → Add CORS headers
  → Strip internal headers (X-Internal-Debug, Server: nginx/1.24)
  → Compress response (gzip)
  → Filter fields (hide sensitive DB fields)
  → Aggregate responses from multiple services (BFF pattern)
```

### API Gateway Products

#### AWS API Gateway

```
Two types:
  REST API Gateway:  Traditional, feature-rich, more expensive
  HTTP API Gateway:  Newer, faster, cheaper, fewer features (use this by default)

```


**Kong config example (declarative):**
```yaml
_format_version: "3.0"

services:
  - name: user-service
    url: http://user-service.production.svc.cluster.local:3000
    routes:
      - name: users-route
        paths: ["/api/users"]
        methods: ["GET", "POST", "PUT", "DELETE"]
    plugins:
      - name: jwt
      - name: rate-limiting
        config:
          minute: 200
          policy: local
      - name: request-transformer
        config:
          add:
            headers: ["X-Gateway: kong", "X-Request-ID: $(uuid)"]
          remove:
            headers: ["Cookie"]

  - name: payment-service
    url: http://payment-service.production.svc.cluster.local:8080
    routes:
      - name: payments-route
        paths: ["/api/payments"]
    plugins:
      - name: jwt
      - name: rate-limiting
        config:
          minute: 50           # Stricter limit for payments
          policy: local
      - name: ip-restriction
        config:
          allow: ["10.0.0.0/8"]  # Only internal IPs + gateway
```

#### Nginx as an API Gateway

```nginx
# Nginx minimal API Gateway config

upstream user_service {
    server user-service:3000;
    keepalive 20;
}

upstream order_service {
    server order-service:4000;
    keepalive 20;
}

server {
    listen 443 ssl http2;
    server_name api.myapp.com;

    # Auth subrequest — validate JWT before routing
    location = /_auth {
        internal;
        proxy_pass http://auth-service:8080/validate;
        proxy_pass_request_body off;
        proxy_set_header Content-Length "";
        proxy_set_header X-Original-URI $request_uri;
    }

    location /api/users {
        auth_request /_auth;                              # Verify JWT
        auth_request_set $user_id $upstream_http_x_user_id;
        proxy_set_header X-User-ID $user_id;             # Forward identity

        proxy_pass http://user_service;
        proxy_set_header X-Request-ID $request_id;       # Correlation ID
    }

    location /api/orders {
        auth_request /_auth;
        auth_request_set $user_id $upstream_http_x_user_id;
        proxy_set_header X-User-ID $user_id;

        proxy_pass http://order_service;
    }

    # Rate limiting
    limit_req_zone $binary_remote_addr zone=api:10m rate=100r/m;
    limit_req zone=api burst=20 nodelay;
}
```

### BFF — Backend for Frontend Pattern

A variation of API Gateway — a **dedicated gateway per client type**.

```
Problem: Mobile needs different data shape than Web,
         and IoT devices need minimal payloads.

One-size-fits-all gateway returns a super-set of data.
Mobile downloads 50 fields, only uses 10. Wasted bandwidth.

Solution: BFF
  Web BFF    → Aggregates and formats for web browser
  Mobile BFF → Aggregates and formats for mobile app
  IoT BFF    → Minimal payloads for constrained devices

                    ┌──► Web BFF    → Aggregated response for browser
Client types        │
  Web Browser ──────┤
  Mobile App  ──────┼──► Mobile BFF → Minimal response for mobile
  IoT Device  ──────┤
                    └──► IoT BFF    → Tiny payload for IoT

  All BFFs → user-service, order-service, product-service...
```

---

### Interview-Ready Summary

> "An API Gateway is the single entry point for all clients — it handles routing, auth, rate limiting, SSL termination, CORS, and logging in one place, so individual microservices don't duplicate this logic. At the routing layer, it dispatches requests by URL path, host, or headers to the right backend service. For auth, it verifies JWT once and forwards user identity in headers (X-User-ID, X-User-Role) — backends trust these without re-verifying the token. Rate limiting at the gateway protects the entire system before requests reach services. AWS API Gateway integrates natively with Lambda/ECS with a hard 29-second timeout. Kong is the most popular self-hosted option — it runs on top of Nginx with a rich plugin ecosystem for auth, rate limiting, transformations, and tracing. For Kubernetes, Kong Ingress Controller or AWS ALB Ingress Controller handle this. The BFF pattern takes it further — a separate gateway per client type (web, mobile, IoT) that aggregates and shapes data differently for each."

## 3. Service Meshes

### What is a Service Mesh?

A **service mesh** is an infrastructure layer that handles **all service-to-service communication** in a microservices architecture — without changing application code.

**The problem it solves:**
```
In a large microservices system (50+ services), every service needs:
  ✅ mTLS encryption (service-to-service)
  ✅ Retry logic
  ✅ Circuit breaking
  ✅ Load balancing
  ✅ Distributed tracing
  ✅ Health checking
  ✅ Traffic splitting (canary deployments)
  ✅ Rate limiting

Without service mesh:
  Every team implements this in their service (in Go, Java, Python, Node...)
  Inconsistent implementations, duplicated code, drift over time.

With service mesh:
  Infrastructure handles ALL of this transparently.
  Application code just makes plain HTTP/gRPC calls.
  The mesh intercepts and handles everything else.
```

---

### Sidecar Proxy Pattern

The core mechanism of service meshes — a **sidecar proxy** is injected alongside every service pod.

```
WITHOUT sidecar:
  Pod: [Your App]  ← app handles all networking concerns

WITH sidecar (Istio/Linkerd):
  Pod: [Your App] + [Sidecar Proxy]
                        ↑
                   Envoy/Linkerd2-proxy
                   intercepts ALL traffic
                   in AND out of the pod

Traffic flow:
  Service A's sidecar → (mTLS) → Service B's sidecar → Service B's app
  Service A's app never talks directly to Service B!
  The sidecar handles: mTLS, retry, circuit break, tracing, metrics.
```

**How sidecar injection works (Kubernetes):**
```yaml
# Label your namespace:
kubectl label namespace production istio-injection=enabled

# Istio automatically injects a sidecar into every new pod in this namespace.
# No changes to your Deployment YAML needed!

# Result: your pod goes from this:
Containers: [payment-app]

# To this automatically:
Containers: [payment-app, istio-proxy (Envoy)]
```

---

### Istio

**Istio** is the most popular and feature-rich service mesh. Uses **Envoy** as the sidecar proxy.

#### Istio Architecture

```
┌────────────────────────────────────────────────────────────────┐
│                    Istio Control Plane                          │
│                                                                 │
│  istiod (single binary):                                        │
│    • Pilot   → Pushes routing config to Envoy sidecars         │
│    • Citadel → Issues and rotates mTLS certificates (SPIFFE)   │
│    • Galley  → Validates config, watches Kubernetes API        │
└────────────────────────────────────────────────────────────────┘
             ↓ xDS protocol (Envoy API — push config)
┌──────────────────────────────────────────────────────────────────┐
│                      Data Plane                                   │
│                                                                   │
│  Pod A                              Pod B                         │

│  │ App  │ Envoy Proxy │────mTLS───►│ Envoy Proxy │ App  │        │
│  └────────────────────┘            └────────────────────┘        │
│                                                                   │
│  Envoy collects metrics → Prometheus                              │
│  Envoy sends traces → Jaeger/Zipkin                              │
└──────────────────────────────────────────────────────────────────┘
```

#### Istio Traffic Management

**VirtualService — routing rules:**
```yaml
# Route 90% to v1, 10% to v2 (canary deployment)
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: payment-service
spec:
  hosts:
    - payment-service
  http:
    - match:
        - headers:
            x-beta-user:
              exact: "true"     # Beta users always get v2
      route:
        - destination:
            host: payment-service
            subset: v2
    - route:                    # Everyone else: 90/10 split
        - destination:
            host: payment-service
            subset: v1
          weight: 90
        - destination:
            host: payment-service
            subset: v2
          weight: 10
```

**DestinationRule — load balancing + circuit breaking:**
```yaml
apiVersion: networking.istio.io/v1alpha3
kind: DestinationRule
metadata:
  name: payment-service
spec:
  host: payment-service
  trafficPolicy:
    loadBalancer:
      simple: LEAST_CONN          # Load balancing algorithm
    connectionPool:
      tcp:
        maxConnections: 100       # Max connections to this service
      http:
        h2UpgradePolicy: UPGRADE  # Use HTTP/2 if possible
    outlierDetection:             # Circuit breaking!
      consecutive5xxErrors: 5     # Open circuit after 5 consecutive errors
      interval: 10s               # Check every 10 seconds
      baseEjectionTime: 30s       # Eject unhealthy instance for 30s
      maxEjectionPercent: 50      # Eject at most 50% of instances
  subsets:
    - name: v1
      labels:
        version: v1
    - name: v2
      labels:
        version: v2
```

**Retry policy:**
```yaml
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: payment-service
spec:
  hosts:
    - payment-service
  http:
    - retries:
        attempts: 3                      # Retry 3 times
        perTryTimeout: 2s                # Each attempt times out after 2s
        retryOn: "5xx,gateway-error,connect-failure,retriable-4xx"
      timeout: 10s                       # Total timeout across all retries
      route:
        - destination:
            host: payment-service
```

#### Istio Security — mTLS

```yaml
# Enable strict mTLS across the entire mesh
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: production
spec:
  mtls:
    mode: STRICT    # All traffic MUST use mTLS (no plain HTTP allowed)

# Authorization policy — who can talk to whom
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: payment-service-policy
  namespace: production
spec:
  selector:
    matchLabels:
      app: payment
  rules:
    - from:
        - source:
            principals: ["cluster.local/ns/production/sa/order-service"]
      to:
        - operation:
            methods: ["POST"]
            paths: ["/charge"]
    # Only order-service can POST to /charge on payment-service
    # All other callers are rejected — zero trust!
```

### Linkerd

**Linkerd** is the lightweight, simpler alternative to Istio. Uses its own **linkerd2-proxy** (written in Rust — very fast, low overhead).

```
│ Traffic management│ ✅ Advanced     │ Basic                      │
```

### Service Mesh Observability (The Golden Signals)

Service meshes give you the **golden signals** without any code changes:

```
1. Latency (how long requests take)
   → Istio: exported to Prometheus, view in Grafana
   → Linkerd: linkerd viz stat deploy

2. Traffic (request rate)
   → Requests per second per service
   → Automatically tracked by sidecar

3. Errors (error rate)
   → 5xx rate per service, per route
   → Circuit breaker triggers visible

4. Saturation (how full is the service)
   → Connection pool usage, queue depth

linkerd viz top deploy/payment-service      # Live request stream
linkerd viz routes deploy/payment-service   # Per-route stats

Distributed tracing (Istio):
  Every request gets a trace ID propagated through headers.
  Envoy reports spans to Jaeger/Zipkin.
  You see the full call graph: Gateway → Order → Payment → DB
  With exact latency for each hop.
```

---

### Interview-Ready Summary

> "A service mesh is an infrastructure layer that handles all service-to-service communication concerns — mTLS encryption, retries, circuit breaking, load balancing, and distributed tracing — without changing application code. It works through the sidecar pattern: a proxy (Envoy for Istio, linkerd2-proxy for Linkerd) is injected alongside every pod and intercepts all inbound and outbound traffic. Istio is the most feature-rich option — its control plane (istiod) pushes routing config and certificates to Envoy sidecars. VirtualService defines routing rules (canary splits, header-based routing, retries), DestinationRule defines load balancing and circuit breaking. AuthorizationPolicy enforces zero-trust: only order-service can call payment-service's /charge endpoint, everything else is denied. Linkerd is the lightweight alternative with lower memory overhead (~10MB vs ~50MB per pod). The biggest win of a service mesh is observability — you get latency, traffic, error rate, and distributed traces across all services without writing a single line of instrumentation code."

## 4. Container Networking

### Docker Networking

Docker creates virtual networks to allow containers to communicate. Understanding this is fundamental for debugging container issues.

#### Docker Network Types

**1. Bridge Network (default)**
```
Host Machine
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  docker0 bridge (172.17.0.1/16) ← virtual switch           │
│       │                                                      │
│       ├── Container 1 (172.17.0.2) nginx                    │
│       ├── Container 2 (172.17.0.3) node-app                 │
│       └── Container 3 (172.17.0.4) redis                    │
│                                                              │
│  Containers on the SAME bridge can talk to each other.      │
│  External access requires port mapping (-p 8080:80)         │
└──────────────────────────────────────────────────────────────┘
          │
    Host network interface (e.g., eth0: 192.168.1.5)
          │
       Internet

Container ↔ Container (same bridge): direct, no NAT
Container → Internet: NAT via host
Internet → Container: port forwarding (-p host_port:container_port)
```

**Default bridge vs custom bridge:**
```bash
# Default bridge: containers communicate by IP only
docker run --name app1 myapp1
docker run --name app2 myapp2
# app2 cannot reach app1 by name — only by 172.17.0.2

# Custom bridge: containers communicate by NAME (DNS resolution!)
docker network create my-network
docker run --name app1 --network my-network myapp1
docker run --name app2 --network my-network myapp2
# app2 can reach app1 at http://app1:8080  ← name resolution works!

# Why? Custom bridges have an embedded DNS server.
# Default bridge is legacy — always use custom bridges.
```

**2. Host Network**
```
Container shares the host's network stack directly.
No isolation, no NAT.

docker run --network host nginx
# nginx listens on port 80 → host port 80 (no -p mapping needed)
# Container sees host's IP, host's network interfaces

Use case: Maximum performance (no NAT overhead), specific low-level networking.
Warning: No isolation — if container opens port 80, it's on the host too.
```

**3. None Network**
```
Container has no network access at all.
docker run --network none myapp

Use case: Completely isolated batch jobs, security sandboxing.
```

**4. Overlay Network (Docker Swarm / multi-host)**
```
Spans multiple Docker hosts — containers on different machines can communicate.
Uses VXLAN encapsulation to create a virtual L2 network over L3.

Host 1 (192.168.1.5):
  Container A (10.0.0.2) ─────────────┐
                                        │ VXLAN tunnel
Host 2 (192.168.1.6):                  │
  Container B (10.0.0.3) ─────────────┘

Container A → Container B: 10.0.0.3 (direct, overlay handles routing)
```

---

#### Docker Compose Networking

```yaml
# docker-compose.yml
version: '3.8'
services:
  nginx:
    image: nginx
    ports:
      - "443:443"       # Host:Container port mapping
    networks:
      - frontend

  app:
    build: .
    networks:
      - frontend        # Can receive from nginx
      - backend         # Can reach database
    environment:
      - DB_HOST=postgres  # Use service name for DNS resolution!

  postgres:
    image: postgres
    networks:
      - backend         # Only app can reach it (not nginx)
    volumes:
      - pgdata:/var/lib/postgresql/data

  redis:
    image: redis
    networks:
      - backend

networks:
  frontend:             # nginx ↔ app
  backend:              # app ↔ postgres ↔ redis (isolated!)

volumes:
  pgdata:

# Result:
# nginx can talk to app (both on frontend network)
# app can talk to postgres and redis (all on backend network)
# nginx CANNOT reach postgres directly (different networks)
# This is network segmentation with Docker!
```

### Kubernetes Networking Model

Kubernetes networking has strict rules that make it predictable:

```
The 4 Kubernetes Networking Requirements:
  1. Every Pod gets its own unique IP address
  2. Pods on same node communicate without NAT
  3. Pods on different nodes communicate without NAT
  4. The IP a pod sees for itself = the IP other pods use to reach it
     (no NAT distortion — makes debugging simple)
```

#### Pod Networking

```
Node 1 (192.168.1.5)                    Node 2 (192.168.1.6)
┌──────────────────────────────────┐    ┌──────────────────────────────────┐
│ Pod A (10.244.1.2)               │    │ Pod C (10.244.2.2)               │
│ Pod B (10.244.1.3)               │    │ Pod D (10.244.2.3)               │
│                                  │    │                                  │
│ veth pair: pod ↔ node bridge     │    │ veth pair: pod ↔ node bridge     │
│ cbr0 bridge: 10.244.1.0/24      │    │ cbr0 bridge: 10.244.2.0/24      │
└──────────────────┬───────────────┘    └──────────────────┬───────────────┘
                   │                                        │
            eth0: 192.168.1.5  ──── CNI Overlay ────  eth0: 192.168.1.6
                   │                (Flannel/Calico)         │
                   └──────────────── Router ─────────────────┘

Pod A (10.244.1.2) → Pod C (10.244.2.2):
  Packet: src=10.244.1.2, dst=10.244.2.2
  Goes to cbr0 → eth0 → CNI encapsulates → Node 2's eth0 → Pod C's veth
  No NAT! Pod sees each other's real IPs.
```

#### CNI (Container Network Interface)

**CNI** plugins implement the Kubernetes networking model. Popular options:

```
Flannel:
  → Simplest CNI, easy to set up
  → Uses VXLAN overlay (encapsulates packets in UDP)
  → No network policy support (use with Calico for policies)
  → Good for: simple clusters, development

Calico:
  → Uses BGP routing (no overlay, packets travel natively)
  → Supports NetworkPolicy (firewall between pods)
  → High performance (no encapsulation overhead)
  → Good for: production, security-sensitive, high-performance

Cilium:
  → Uses eBPF (kernel-level networking — no iptables!)
  → Extremely high performance
  → Rich network policy (L7 aware — can filter by HTTP method/path)
  → Good for: large-scale production, service mesh integration

Weave Net:
  → Simple overlay, supports encryption
  → Good for: small-medium clusters

AWS VPC CNI (on EKS):
  → Pods get real VPC IPs (from ENI — Elastic Network Interface)
  → No overlay — packets route natively in AWS VPC
  → Pods directly accessible with VPC security groups
  → Limitation: limited IPs per node (based on instance type)
```

#### Kubernetes NetworkPolicy

Control traffic between pods (acts like a firewall):

```yaml
# Deny all ingress to the payment namespace by default
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
  namespace: payment
spec:
  podSelector: {}       # Applies to ALL pods in this namespace
  policyTypes:
    - Ingress           # Control incoming traffic

---
# Allow ONLY order-service to reach payment-service on port 8080
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-order-to-payment
  namespace: payment
spec:
  podSelector:
    matchLabels:
      app: payment-service    # Applies to payment pods
  policyTypes:
    - Ingress
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              name: order         # From the order namespace
          podSelector:
            matchLabels:
              app: order-service  # Specifically from order-service pods
      ports:
        - protocol: TCP
          port: 8080
```

#### kube-proxy and Service Networking

When a pod calls a **Service** (ClusterIP), kube-proxy intercepts it:

```
Pod A calls: http://payment-service (resolves to ClusterIP: 10.96.0.100)

kube-proxy (on every node) maintains iptables/IPVS rules:
  Destination: 10.96.0.100:80
  → DNAT to: 10.244.1.5:8080  (Pod 1) — 33% chance
          or 10.244.1.6:8080  (Pod 2) — 33% chance
          or 10.244.2.1:8080  (Pod 3) — 33% chance

Two modes:
  iptables mode (default):
    → Probabilistic DNAT rules
    → No real load balancing (just probability)
    → Hard to debug (thousands of iptables rules at scale)

  IPVS mode (recommended at scale):
    → Dedicated load balancing kernel module
    → Faster, more algorithms (rr, lc, sh, etc.)
    → Scales better (hash table vs linear scan)

Enable IPVS: kube-proxy --proxy-mode=ipvs
```

---

### Interview-Ready Summary

> "Docker networking defaults to a bridge network — containers on the same bridge communicate directly, external access requires port mapping. Custom bridge networks add DNS — containers resolve each other by name. Docker Compose creates isolated networks per project and lets you segment: nginx and app share a frontend network, app and database share a backend network, so nginx can't directly reach the database. Kubernetes has four networking requirements: every pod gets a unique IP, pods communicate without NAT (even across nodes). CNI plugins implement this — Flannel uses VXLAN overlay (simple), Calico uses BGP routing (high-performance, supports NetworkPolicy), Cilium uses eBPF for kernel-level networking. Services have a virtual ClusterIP that kube-proxy intercepts and DNAT's to real pod IPs using iptables or IPVS. NetworkPolicy is the Kubernetes firewall — I use default-deny-ingress in sensitive namespaces and explicitly allow only needed connections."

## 5. VPCs & Private Subnets

### What is a VPC?

A **VPC** (Virtual Private Cloud) is your own isolated, private network within a cloud provider. It's a logically isolated section of the cloud where you launch resources.

**Analogy:**
> The cloud provider's data center is a huge apartment building. A VPC is your private apartment — completely isolated from other tenants even though you share the building. You decide the floor plan (CIDR block), which rooms are accessible from outside (public subnets), which are internal only (private subnets), and who has keys (security groups).

---

### AWS VPC Architecture

```
AWS Region: us-east-1
```

### VPC Core Components

**1. CIDR Block**
```
Your VPC gets a CIDR block — defines the IP range for ALL resources.

Common choices:
  10.0.0.0/16   → 65,534 usable IPs (recommended for production)
  10.0.0.0/24   → 254 usable IPs (too small — don't use for VPC)
  172.16.0.0/16 → 65,534 IPs (alternative private range)

Rules:
  → Must be from private ranges (10.x, 172.16-31.x, 192.168.x)
  → Can't be changed after creation (choose wisely!)
  → Subnets divide the VPC CIDR into smaller chunks
  → Reserve space for future subnets when planning
```

**2. Subnets**
```
Subnets are subdivisions of your VPC CIDR within a single AZ.

Planning example for 10.0.0.0/16:
  Public Subnets (internet-facing resources):
    us-east-1a:  10.0.1.0/24   (254 IPs)
    us-east-1b:  10.0.2.0/24   (254 IPs)
    us-east-1c:  10.0.3.0/24   (254 IPs)

  Private App Subnets (backend servers):
    us-east-1a:  10.0.10.0/24  (254 IPs)
    us-east-1b:  10.0.11.0/24  (254 IPs)
    us-east-1c:  10.0.12.0/24  (254 IPs)

  Private DB Subnets (databases):
    us-east-1a:  10.0.20.0/24  (254 IPs)
    us-east-1b:  10.0.21.0/24  (254 IPs)
    us-east-1c:  10.0.22.0/24  (254 IPs)

  Reserved for future use:
    10.0.100.0/22 → 1,022 IPs for future microservices

Note: AWS reserves 5 IPs per subnet (first 4 + last):
  10.0.1.0  → Network address
  10.0.1.1  → VPC router
  10.0.1.2  → DNS resolver
  10.0.1.3  → Reserved for future use
  10.0.1.255 → Broadcast
  Usable: 10.0.1.4 – 10.0.1.254 = 251 IPs
```

**3. Route Tables**
```
Route tables control WHERE traffic goes from a subnet.

Public Subnet Route Table:
┌─────────────────┬──────────────────────────┐
│ Destination     │ Target                   │
├─────────────────┼──────────────────────────┤
│ 10.0.0.0/16     │ local  ← VPC traffic    │
│ 0.0.0.0/0       │ igw-xxx ← Internet GW  │
└─────────────────┴──────────────────────────┘
(anything not in VPC → goes to internet)

Private App Subnet Route Table:
┌─────────────────┬──────────────────────────┐
│ Destination     │ Target                   │
├─────────────────┼──────────────────────────┤
│ 10.0.0.0/16     │ local  ← VPC traffic    │
│ 0.0.0.0/0       │ nat-xxx ← NAT Gateway  │
└─────────────────┴──────────────────────────┘
(internet traffic goes THROUGH NAT, not directly)

DB Subnet Route Table:
┌─────────────────┬──────────────────────────┐
│ Destination     │ Target                   │
├─────────────────┼──────────────────────────┤
│ 10.0.0.0/16     │ local  ← VPC traffic only│
└─────────────────┴──────────────────────────┘
(NO internet access at all — completely isolated)
```

**4. Internet Gateway (IGW)**
```
Attached to the VPC (one per VPC).
Enables internet access for public subnets.
Performs NAT for instances with public IPs.

For a resource to be "public" it needs:
  ✅ Be in a subnet with route to IGW (0.0.0.0/0 → igw-xxx)
  ✅ Have a public IP or Elastic IP assigned
  ✅ Security group allows inbound traffic
```

**5. NAT Gateway**
```
Allows private subnet resources to reach the internet (outbound only).
Lives in a PUBLIC subnet with an Elastic IP.
Private subnets route internet traffic to NAT GW.

Per AZ: Deploy one NAT Gateway per AZ for high availability.
  NAT GW in AZ-a → private subnets in AZ-a use it
  NAT GW in AZ-b → private subnets in AZ-b use it
  (If AZ-a goes down, AZ-b's resources still have internet access)

Cost: ~$0.045/hour + $0.045/GB data processed
      ≈ $32/month just to exist + data transfer costs
      Use VPC Endpoints for AWS services to avoid NAT costs!
```

**6. VPC Endpoints**
```
Connect to AWS services PRIVATELY (without going through internet/NAT).

Without endpoint:
  Private EC2 → NAT GW → Internet → S3 (costly, slower, less secure)

With endpoint:
  Private EC2 → VPC Endpoint → S3 (free within region, private, fast)

Types:
  Gateway Endpoint (free):  S3, DynamoDB
  Interface Endpoint (paid): Most other AWS services (SQS, SNS, Secrets Manager, etc.)
                             Creates an ENI with private IP in your subnet

Cost savings: If you have high S3 traffic, a Gateway endpoint eliminates NAT costs.
```

### VPC Peering & Transit Gateway

**VPC Peering:**
```
Connect two VPCs privately (same or different accounts/regions).

VPC A (10.0.0.0/16) ◄──── Peering ────► VPC B (172.16.0.0/16)

Each VPC adds routes:
  VPC A route table: 172.16.0.0/16 → pcx-xxx (peering connection)
  VPC B route table: 10.0.0.0/16  → pcx-xxx

Limitations:
  ❌ Non-transitive: A↔B and B↔C does NOT mean A can reach C
  ❌ CIDR ranges cannot overlap
  ❌ Gets complex at scale (N² connections for N VPCs)

Use when: 2-3 VPCs, simple connectivity
```

**Transit Gateway:**
```
Hub-and-spoke model. All VPCs connect to one Transit Gateway.
VPCs don't connect to each other — they route through TGW.

    VPC A ─────┐
    VPC B ─────┼──► Transit Gateway ◄── On-premises (VPN/Direct Connect)
    VPC C ─────┤
    VPC D ─────┘

✅ Transitive: A → TGW → C works!
✅ Scale: thousands of VPCs
✅ Cross-account, cross-region
✅ Centralized routing management
❌ Cost: ~$0.05/hour + $0.02/GB

Use when: Multiple VPCs, multi-account, complex connectivity
```

### GCP & Azure VPC Equivalents

**GCP (Google Cloud Platform):**
```
VPC is GLOBAL (spans all regions — unique to GCP!)
  One VPC → subnets in us-central1, europe-west1, asia-east1 all connected.
  No peering needed for multi-region within same VPC.

Key concepts:
  VPC Network → global virtual network
  Subnetwork  → regional subnet (auto-mode or custom)
  Cloud Router → BGP routing for VPN/Interconnect
  Cloud NAT    → Managed NAT for private instances
  Firewall Rules → applied at network level (not subnet) by tags
                   e.g., tag "web-server" → allow 80, 443

Firewall rules by tags (different from AWS security groups):
  gcloud compute firewall-rules create allow-http \
    --network=my-vpc \
    --allow=tcp:80,tcp:443 \
    --target-tags=web-server
```

**Azure:**
```
VNet (Virtual Network) = AWS VPC

Key concepts:
  VNet         → Virtual Network (regional, not global)
  Subnet       → subdivision of VNet (no AZ concept — regions have zones)
  NSG          → Network Security Group (like AWS Security Group + NACL combined)
  Azure LB     → Load Balancer
  Application GW → L7 Load Balancer (like AWS ALB)
  Private Link  → Private endpoint to Azure services (like AWS Interface Endpoint)
  VNet Peering → Connect VNets (same or different regions)

NSG rules (bidirectional, stateful):
  Priority  | Name          | Port  | Source         | Action
  100       | AllowHTTPS    | 443   | Internet       | Allow
  200       | AllowHTTP     | 80    | Internet       | Allow
  300       | AllowSSH      | 22    | 10.0.0.0/8     | Allow
  65000     | DenyAll       | *     | *              | Deny
```

### AWS VPC Terraform (Infrastructure as Code)

```hcl
# vpc.tf — Complete 3-tier VPC

provider "aws" {
  region = "us-east-1"
}

# VPC
resource "aws_vpc" "main" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_hostnames = true   # Gives EC2 instances DNS names
  enable_dns_support   = true

  tags = { Name = "production-vpc" }
}

# Internet Gateway
resource "aws_internet_gateway" "main" {
  vpc_id = aws_vpc.main.id
  tags   = { Name = "main-igw" }
}

# Public Subnets (2 AZs for HA)
resource "aws_subnet" "public" {
  count                   = 2
  vpc_id                  = aws_vpc.main.id
  cidr_block              = "10.0.${count.index + 1}.0/24"
  availability_zone       = data.aws_availability_zones.available.names[count.index]
  map_public_ip_on_launch = true    # Auto-assign public IP
  tags = { Name = "public-${count.index + 1}" }
}

# Private App Subnets
resource "aws_subnet" "private_app" {
  count             = 2
  vpc_id            = aws_vpc.main.id
  cidr_block        = "10.0.${count.index + 10}.0/24"
  availability_zone = data.aws_availability_zones.available.names[count.index]
  tags = { Name = "private-app-${count.index + 1}" }
}

# NAT Gateway (one per AZ for HA)
resource "aws_eip" "nat" {
  count  = 2
  domain = "vpc"
}

resource "aws_nat_gateway" "main" {
  count         = 2
  allocation_id = aws_eip.nat[count.index].id
  subnet_id     = aws_subnet.public[count.index].id
  tags = { Name = "nat-gw-${count.index + 1}" }
}

# Route Tables
resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id
  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.main.id  # Public → IGW
  }
}

resource "aws_route_table" "private" {
  count  = 2
  vpc_id = aws_vpc.main.id
  route {
    cidr_block     = "0.0.0.0/0"
    nat_gateway_id = aws_nat_gateway.main[count.index].id  # Private → NAT
  }
}

# Route Table Associations
resource "aws_route_table_association" "public" {
  count          = 2
  subnet_id      = aws_subnet.public[count.index].id
  route_table_id = aws_route_table.public.id
}

resource "aws_route_table_association" "private" {
  count          = 2
  subnet_id      = aws_subnet.private_app[count.index].id
  route_table_id = aws_route_table.private[count.index].id
}
```

### Interview-Ready Summary

> "A VPC is an isolated virtual network in the cloud where you control IP ranges, subnets, routing, and access control. The CIDR block (e.g., 10.0.0.0/16) defines all available IPs; subnets subdivide it within each AZ. Public subnets have a route to the Internet Gateway and resources with public IPs (load balancers, bastion hosts, NAT Gateways). Private subnets have no direct internet access — app servers live here, reaching the internet only through a NAT Gateway in the public subnet for outbound calls. DB subnets have no route to the internet at all. Route tables control traffic flow: public subnet routes 0.0.0.0/0 to IGW, private routes it to NAT GW, DB subnet has only local VPC routes. For multi-VPC architectures, Transit Gateway is the modern approach — it replaces VPC Peering's N² connections with a hub-and-spoke model. VPC Endpoints let private resources access S3/DynamoDB without going through NAT, saving cost and improving security. In GCP, VPCs are global (subnets span regions in one network), while Azure uses VNets with NSGs for traffic control."
