## 1. Caching Layers

### What is Caching?

**Caching** is storing the result of an expensive operation so future requests can be served faster without repeating the work. It's the single most effective technique for improving performance.

**Analogy:**
> Imagine you're a chef. Every customer orders "today's special." Without caching, you cook it fresh every single time. With caching, you cook a big batch in the morning and serve from it all day. When the batch runs out (cache miss), you cook more.

**The caching hierarchy (fastest → slowest, smallest → largest):**

```
L1 CPU Cache      → ~1 ns      (nanoseconds, kilobytes)
L2 CPU Cache      → ~4 ns
L3 CPU Cache      → ~10 ns
RAM               → ~100 ns    (in-process cache)
Redis / Memcached → ~1 ms      (network, shared across servers)
SSD (local disk)  → ~100 µs
CDN Edge Cache    → ~5 ms      (geographically close)
Origin Database   → ~10-100 ms (the expensive source of truth)
```

**Every layer you avoid going to the right saves latency.**

---

### Layer 1 — In-Process / Application Cache

The fastest cache — lives in your application's memory. No network hop.

```javascript
// Simple in-memory cache (single server only!)
const cache = new Map();

async function getUser(id) {
  const cacheKey = `user:${id}`;

  // Check cache first
  if (cache.has(cacheKey)) {
    return cache.get(cacheKey);   // ← served in microseconds
  }

  // Cache miss — fetch from DB
  const user = await db.users.findById(id);

  // Store in cache with expiry
  cache.set(cacheKey, user);
  setTimeout(() => cache.delete(cacheKey), 60_000); // expire after 60s

  return user;
}
```

**Problems with in-process cache:**
```
❌ Not shared across servers
   Server 1 caches user:42
   Next request goes to Server 2 → cache miss → hits DB again

❌ Lost on restart
   Every deploy clears the cache → thundering herd on startup

❌ No memory limits (can grow unbounded)
   Solution: use LRU (Least Recently Used) eviction

✅ Good for:
   Read-heavy, small datasets that rarely change
   Config values, feature flags, lookup tables
   Single-server setups or when cache coherence isn't critical
```

### Layer 2 — Redis (Shared Distributed Cache)

**Redis** (Remote Dictionary Server) is an in-memory data structure store used as a distributed cache. All your servers share the same Redis instance — one cache miss hits Redis, not the DB.

```
Server 1 ─────┐
Server 2 ─────┼──► Redis ──► (miss) ──► Database
Server 3 ─────┘

Server 1 caches user:42 in Redis.
Server 2 requests user:42 → Redis HIT → no DB query ✅
```

#### Redis Caching Patterns

**Pattern 1: Cache-Aside (Lazy Loading) — Most Common**

Application checks cache first. On miss, fetches from DB and populates cache.

```javascript
const redis = require('redis');
const client = redis.createClient();

async function getUser(id) {
  const cacheKey = `user:${id}`;

  // 1. Check Redis
  const cached = await client.get(cacheKey);
  if (cached) {
    return JSON.parse(cached);       // ← Cache HIT
  }

  // 2. Cache MISS — fetch from DB
  const user = await db.users.findById(id);
  if (!user) return null;

  // 3. Store in Redis with TTL (Time To Live)
  await client.setEx(
    cacheKey,
    3600,                            // Expire in 1 hour (seconds)
    JSON.stringify(user)
  );

  return user;                       // ← Cache populated for next time
}

// Invalidate cache when user is updated
async function updateUser(id, data) {
  const user = await db.users.update(id, data);
  await client.del(`user:${id}`);    // ← Remove stale cache entry
  return user;
}
```

**Pattern 2: Write-Through**

Write to cache AND database simultaneously on every write. Cache always stays warm.

```javascript
async function updateUser(id, data) {
  // Write to DB
  const user = await db.users.update(id, data);

  // Write to cache immediately (cache stays fresh)
  await client.setEx(`user:${id}`, 3600, JSON.stringify(user));

  return user;
}
```

```
✅ Cache always has fresh data
✅ No cache misses after first write
❌ Every write hits both DB AND Redis (slightly slower writes)
❌ Cache fills up with data that might never be read
```

**Pattern 3: Write-Behind (Write-Back)**

Write to cache immediately, asynchronously persist to DB in the background.

```
Write → Redis (fast, instant ACK to client)
           ↓ async (queue / background worker)
       Database (writes batched, delayed)

✅ Very fast writes (return before DB write completes)
❌ Risk of data loss if Redis crashes before DB write
❌ Complex implementation
Use for: analytics counters, view counts, like counts (loss of a few OK)
```

**Pattern 4: Read-Through**

Cache sits in front of the DB. Application only talks to cache. Cache handles DB reads.

```
App → Cache → (on miss) → DB → Cache stores → returns to App

Unlike Cache-Aside where APP manages the cache population,
in Read-Through the CACHE itself fetches from DB on miss.

Used by: Redis as a database proxy, some ORM cache plugins
```

#### Redis Data Structures for Caching

Redis isn't just a key-value store — rich data structures enable powerful caching patterns.

```javascript
// String — simple value cache
await client.setEx('user:42', 3600, JSON.stringify(user));
const user = JSON.parse(await client.get('user:42'));

// Hash — cache an object's fields individually (update one field without re-serializing whole object)
await client.hSet('user:42', {
  name: 'Alice',
  email: 'alice@example.com',
  role: 'admin'
});
await client.expire('user:42', 3600);
const name = await client.hGet('user:42', 'name');   // Get just one field
const user = await client.hGetAll('user:42');          // Get all fields

// List — cache recent items (e.g., recent activity feed)
await client.lPush('user:42:recent', JSON.stringify(activity));
await client.lTrim('user:42:recent', 0, 99);           // Keep only last 100
const recent = await client.lRange('user:42:recent', 0, 9); // Get top 10

// Sorted Set — leaderboard, rate limiting
await client.zAdd('leaderboard', { score: 9500, value: 'alice' });
await client.zAdd('leaderboard', { score: 8200, value: 'bob' });
const top10 = await client.zRange('leaderboard', 0, 9, { REV: true });

// Set — cache unique values (e.g., user's permissions)
await client.sAdd('user:42:permissions', 'read:users', 'write:posts', 'delete:own');
const canWrite = await client.sIsMember('user:42:permissions', 'write:posts');

// Expiry on any structure
await client.expire('user:42', 3600);       // Expires in 1 hour
await client.expireAt('user:42', 1700000000); // Expires at Unix timestamp
```

#### Cache Eviction Policies (What Redis Does When Full)

When Redis runs out of memory, it evicts keys based on a configured policy:

```
noeviction         → Return error when memory full (default — bad for caches!)
allkeys-lru        → Evict least recently used keys from ALL keys ← most common for cache
volatile-lru       → Evict LRU keys only from keys WITH expiry set
allkeys-lfu        → Evict least frequently used keys (better than LRU for some patterns)
volatile-lfu       → LFU but only keys with expiry
allkeys-random     → Evict random keys
volatile-random    → Evict random keys with expiry
volatile-ttl       → Evict keys with shortest TTL first
```

**For a cache:** Use `allkeys-lru` or `allkeys-lfu`
**For a session store:** Use `volatile-lru` (only evict sessions, not app data)

```bash
# Redis config
maxmemory 2gb
maxmemory-policy allkeys-lru
```

### Layer 3 — HTTP Cache Headers

HTTP has a built-in caching system. Browsers, CDNs, and proxies all understand these headers. Free caching without touching your application code.

#### Cache-Control Header

```
Cache-Control: public, max-age=3600, s-maxage=86400, stale-while-revalidate=60

public            → Any cache (browser, CDN, proxy) can store this
private           → Only browser cache (not CDN) — for user-specific data
no-cache          → Must revalidate with server before serving (can still store)
no-store          → Never cache anywhere (sensitive data)
max-age=3600      → Browser: cache for 3600 seconds (1 hour)
s-maxage=86400    → CDN/shared caches: cache for 86400 seconds (overrides max-age)
immutable         → Content will never change (use with content-hashed URLs)
stale-while-revalidate=60 → Serve stale while fetching fresh in background (60s grace)
stale-if-error=3600       → Serve stale if origin returns error (resilience)
must-revalidate   → Do NOT serve stale under any circumstances
```

**Practical examples:**

```
# Hashed JS/CSS bundles (filename changes when content changes)
Cache-Control: public, max-age=31536000, immutable
→ Browser + CDN cache forever. Safe because URL changes on deploy.

# HTML pages (must always be fresh)
Cache-Control: public, no-cache
→ Can store, but must check with server first (uses ETag/Last-Modified)

# Public API data (product catalog)
Cache-Control: public, s-maxage=300, stale-while-revalidate=60
→ CDN caches 5 min. If stale, serves old while fetching new (no latency spike).

# User-specific API responses (my profile, my cart)
Cache-Control: private, no-cache
→ Browser can cache, but CDN cannot. Must revalidate.

# Auth endpoints, payments
Cache-Control: no-store
→ Never cached anywhere.
```

#### ETag — Entity Tag (Conditional Requests)

An **ETag** is a version identifier for a resource. It enables **conditional requests** — the client asks "has this changed since I last fetched it?"

```
First request:
  Client:  GET /api/users/42
  Server:  200 OK
           ETag: "abc123def456"
           Cache-Control: no-cache
           Body: { "name": "Alice", ... }

  Browser stores: URL + ETag + response body

Second request (browser re-checks):
  Client:  GET /api/users/42
           If-None-Match: "abc123def456"   ← "Only send if different from this"

  Server checks: Has user 42 changed since ETag "abc123def456"?

  → NOT changed:
    Server: 304 Not Modified               ← No body! Just a status.
    Browser serves from its own cache.
    Saves bandwidth (no body transmitted).

  → HAS changed:
    Server: 200 OK
            ETag: "xyz789new000"           ← New ETag
            Body: { "name": "Alice Smith", ... }  ← New data
```

**Generating ETags:**
```javascript
const crypto = require('crypto');

app.get('/api/users/:id', async (req, res) => {
  const user = await db.users.findById(req.params.id);

  // Generate ETag from content hash
  const etag = crypto
    .createHash('md5')
    .update(JSON.stringify(user))
    .digest('hex');

  // Check If-None-Match header
  if (req.headers['if-none-match'] === `"${etag}"`) {
    return res.status(304).end();   // Not Modified — no body sent
  }

  res.set('ETag', `"${etag}"`);
  res.set('Cache-Control', 'private, no-cache');
  res.json(user);
});
```

---

#### Last-Modified (Alternative to ETag)

```
First request:
  Server: 200 OK
          Last-Modified: Wed, 20 Jun 2025 12:00:00 GMT

Subsequent request:
  Client: GET /resource
          If-Modified-Since: Wed, 20 Jun 2025 12:00:00 GMT

  Server: 304 Not Modified  (if unchanged)
       or 200 OK + new content (if changed)
```

**ETag vs Last-Modified:**
```
ETag:           Exact content comparison (more precise)
                Works even if file content changes to same content (ETag stays same)
                Handles sub-second changes
Last-Modified:  Timestamp comparison (less precise, 1-second resolution)
                Simpler to implement
                Some servers only support one or the other

Best practice: Implement BOTH. ETag takes precedence if both are present.
```

### Cache Stampede / Thundering Herd Problem

When a popular cache key expires, many requests simultaneously hit the database.

```
Problem:
  10,000 users/second hit /api/trending-products
  Cache TTL expires at 12:00:00 exactly
  All 10,000 requests see cache miss simultaneously
  All 10,000 requests hit DB at the same time → DB overwhelmed 💥

Solutions:

1. Probabilistic Early Expiration (PER):
   As TTL approaches, randomly start refreshing early
   Not all processes hit expiry at the same moment

2. Cache Locking (Mutex):
   First miss acquires a lock, fetches from DB, populates cache
   Other misses wait for lock, then read from (now populated) cache

3. Stale-while-revalidate:
   Return stale data immediately while ONE process refreshes in background
   No stampede because stale data is served to everyone else
```

```javascript
// Cache Locking Pattern
const locks = new Map();

async function getWithLock(cacheKey, fetchFn, ttl) {
  // 1. Check cache
  const cached = await client.get(cacheKey);
  if (cached) return JSON.parse(cached);

  // 2. Cache miss — check if someone else is already fetching
  if (locks.has(cacheKey)) {
    // Wait for the in-flight fetch to complete
    return locks.get(cacheKey);
  }

  // 3. Acquire lock — start fetching
  const fetchPromise = (async () => {
    const data = await fetchFn();
    await client.setEx(cacheKey, ttl, JSON.stringify(data));
    locks.delete(cacheKey);
    return data;
  })();

  locks.set(cacheKey, fetchPromise);
  return fetchPromise;   // All concurrent callers await the SAME promise
}

// Usage
const trending = await getWithLock(
  'trending:products',
  () => db.products.findTrending(),
  300
);
```

### Caching Strategy Summary

```
What to cache:
  ✅ Expensive DB queries with repeated results (user profiles, product catalogs)
  ✅ Computed/aggregated results (leaderboards, totals, reports)
  ✅ External API responses (rate-limited 3rd party calls)
  ✅ Session data (auth tokens, shopping carts)
  ✅ Static assets (JS, CSS, images via CDN)

What NOT to cache:
  ❌ Highly personalized data (real-time feed, one-time tokens)
  ❌ Frequently changing data where stale = wrong (stock prices, seat availability)
  ❌ Write-heavy data (causes constant invalidation)
  ❌ Cheap queries (no point caching SELECT id FROM users WHERE id=1)

Cache key design:
  user:{id}              → user:42
  user:{id}:orders       → user:42:orders
  product:{id}:v{version} → product:7:v3  (versioned cache)
  search:{hash_of_query} → search:abc123
```

---

### Interview-Ready Summary

> "Caching has multiple layers: in-process memory (microseconds, not shared), Redis (milliseconds, shared across servers), HTTP cache headers (browser and CDN), and CDN edge caches (geographically close). The main patterns are: Cache-Aside (app checks cache, on miss fetches DB and populates cache — most common), Write-Through (write to cache and DB simultaneously), and Write-Behind (write cache immediately, async DB sync). Redis supports rich data structures — strings, hashes, lists, sorted sets — each suited to different caching patterns. HTTP caching uses Cache-Control to set TTL and scope (public vs private), ETags for conditional requests (304 Not Modified saves bandwidth), and `stale-while-revalidate` to serve stale content while refreshing in the background. The cache stampede problem — many simultaneous misses hitting the DB — is solved with mutex locking, stale-while-revalidate, or probabilistic early expiration."

## 2. Connection Pooling

### The Problem — TCP Connections Are Expensive

Every time your app opens a new TCP connection to a database or service, there's overhead:

```
Opening a new DB connection costs:
  1. TCP 3-way handshake         (~1 RTT)
  2. TLS handshake               (~1-2 RTT, if TLS)
  3. Database authentication     (username/password check)
  4. Session setup               (set timezone, encoding, etc.)
  5. Allocating memory on DB     (per-connection buffer, ~5-10 MB each)

Total cost: ~50-100ms per new connection
PostgreSQL: supports ~100-200 concurrent connections before degrading
MySQL:      supports ~151 concurrent connections (default max_connections)

Without pooling:
  100 API requests/second
  × 1 new DB connection each
  = 100 connections opened/closed per second
  = 100 × 50ms overhead = 5 seconds of PURE connection overhead per second
  = DB exhausted, app slows to a crawl 😱
```

**Analogy:**
> Connection pooling is like a taxi company that keeps cars running and available, rather than manufacturing a new car for each customer and scrapping it after the trip. Reusing connections saves the enormous overhead of creation and teardown.

---

### What is Connection Pooling?

A **connection pool** maintains a set of open, ready-to-use connections. Instead of creating a new connection for each request, the app borrows one from the pool, uses it, and returns it.

```
WITHOUT pooling:
  Request 1 → [open connection] → query → [close connection]
  Request 2 → [open connection] → query → [close connection]
  Request 3 → [open connection] → query → [close connection]
  Each request pays the full 50ms connection overhead!

WITH pooling:
  App starts → Pool creates 10 connections (warm pool)

  Request 1 → [borrow conn #1] → query → [return conn #1]
  Request 2 → [borrow conn #2] → query → [return conn #2]
  Request 3 → [borrow conn #1 again] → query → [return]
              ↑ Already open! No overhead!

  Pool always has connections ready → queries start immediately ✅
```

---

### Pool Configuration — The Critical Settings

```
Pool settings (example: pg-pool for PostgreSQL):

const pool = new Pool({
  max:              10,    // Maximum connections in pool
  min:              2,     // Minimum connections to keep alive (warm pool)
  idleTimeoutMillis: 30000, // Close idle connections after 30s
  connectionTimeoutMillis: 2000, // Throw if can't get connection in 2s
  maxUses:          7500,  // Close connection after 7500 uses (prevent memory leaks)
});
```

**Sizing the pool correctly is critical:**

```
Too small (e.g., max: 2 for 100 req/s):
  → All requests queue waiting for a connection
  → Latency spikes, timeouts, poor throughput

Too large (e.g., max: 500 for a small DB):
  → Each PostgreSQL connection uses ~5-10 MB RAM
  → 500 connections = 2.5-5 GB RAM just for connections
  → DB overwhelmed managing too many connections
  → Performance degrades

Right size formula:
  Rule of thumb: pool_size = (num_cpu_cores × 2) + effective_spindle_count

For most apps:
  Start with 10-20 connections per server instance
  Monitor: if connections are always all in-use → increase
           if many connections are always idle → decrease

If running 5 server instances × 10 pool size = 50 total DB connections
Make sure DB's max_connections > 50 (plus headroom for monitoring tools)
```

### Database Connection Pooling in Practice

**PostgreSQL with pg (Node.js):**

```javascript
const { Pool } = require('pg');

// Create pool once at app startup (NOT per request!)
const pool = new Pool({
  host:     process.env.DB_HOST,
  port:     5432,
  database: process.env.DB_NAME,
  user:     process.env.DB_USER,
  password: process.env.DB_PASSWORD,
  max:      20,              // Max 20 connections
  min:      5,               // Keep 5 warm
  idleTimeoutMillis:   30000, // Close idle after 30s
  connectionTimeoutMillis: 3000, // Fail fast if pool exhausted
});

// Pool events for monitoring
pool.on('connect', () => console.log('New DB connection created'));
pool.on('acquire', () => console.log('Connection borrowed from pool'));
pool.on('remove', () => console.log('Connection removed from pool'));
pool.on('error', (err) => console.error('Idle connection error:', err));

// Usage in request handler
app.get('/users/:id', async (req, res) => {
  const client = await pool.connect();   // Borrow from pool (instant if available)
  try {
    const result = await client.query('SELECT * FROM users WHERE id = $1', [req.params.id]);
    res.json(result.rows[0]);
  } finally {
    client.release();                    // ALWAYS return to pool (even on error)
  }
});

// Or simpler — pool.query() handles borrow/return automatically
app.get('/users', async (req, res) => {
  const result = await pool.query('SELECT * FROM users LIMIT 20');
  res.json(result.rows);
});
```

**Prisma (Node.js) — connection pool built-in:**

### HTTP Connection Pooling (Keep-Alive)

Not just databases — HTTP connections between services are also expensive. HTTP connection pooling reuses TCP connections for multiple HTTP requests.

```
WITHOUT HTTP keep-alive (HTTP/1.0 default):
  [TCP connect] → GET /api/users → response → [TCP close]
  [TCP connect] → GET /api/orders → response → [TCP close]
  Each request: full TCP handshake overhead

WITH HTTP keep-alive (HTTP/1.1 default):
  [TCP connect] → GET /api/users → response
              → GET /api/orders → response
              → GET /api/products → response
              → [TCP close eventually]
  One connection handles multiple requests!
```

**Node.js HTTP agent (connection pooling for outbound requests):**

```javascript
const http  = require('http');
const https = require('https');
const axios = require('axios');

// Create an axios instance with connection pooling
const apiClient = axios.create({
  baseURL: 'https://api.external-service.com',
  timeout: 5000,
  // Custom HTTPS agent with pooling
  httpsAgent: new https.Agent({
    keepAlive:       true,   // Reuse TCP connections
    maxSockets:      50,     // Max concurrent connections to this host
    maxFreeSockets:  10,     // Keep 10 idle connections alive
    timeout:         60000,  // Idle connection timeout: 60s
    freeSocketTimeout: 30000 // Release idle sockets after 30s
  })
});

// All requests through apiClient reuse pooled connections
const users   = await apiClient.get('/users');
const orders  = await apiClient.get('/orders');
const products = await apiClient.get('/products');
// Much faster — no TCP handshake overhead for 2nd, 3rd requests
```

### PgBouncer — External Connection Pooler

For very high concurrency, even per-app pools hit limits. **PgBouncer** is a dedicated connection pooler that sits between your app and PostgreSQL.

```
100 App Servers                     PostgreSQL
  × 20 connections each    →  PgBouncer  →  100 connections total
= 2,000 connections claimed           ↑
                              Pools + multiplexes
                              2,000 app connections
                              → 100 real DB connections

3 pooling modes:
  Session pooling:     App gets connection for entire session.
                       Not much benefit over no pooler.

  Transaction pooling: App gets connection only during a transaction.
                       Released immediately after COMMIT/ROLLBACK.
                       Most efficient! (1 DB conn handles many app sessions)
                       ⚠️  Named prepared statements don't work in this mode.

  Statement pooling:   Connection returned after each statement.
                       Very aggressive. Few apps support this.
```

### Redis Connection Pooling

```javascript
// ioredis — built-in connection pooling via cluster or single client
const Redis = require('ioredis');

// Single Redis client (actually maintains one connection, auto-reconnects)
const redis = new Redis({

const { createPool } = require('generic-pool');

  create:  async () => new Redis({ host: 'redis.internal' }),
  destroy: async (client) => client.quit()

  max: 10,      // Max 10 Redis connections
  min: 2,       // Keep 2 idle
});

const client = await pool.acquire();
try {
  await client.set('key', 'value');
} finally {
  pool.release(client);
}
```

### Interview-Ready Summary

> "Opening a database connection costs 50-100ms — TCP handshake, authentication, session setup, and memory allocation on the DB server. Connection pooling maintains a set of pre-opened connections that requests borrow and return. Pool sizing is critical: too small causes queuing and timeouts, too large exhausts DB memory (PostgreSQL uses ~5-10MB per connection). I set min connections (warm pool) and max connections based on roughly 2× CPU cores, and always configure a connectionTimeout to fail fast when the pool is exhausted rather than waiting indefinitely. For extremely high concurrency, PgBouncer sits between the app and DB in transaction pooling mode — 2,000 app connections share 100 real DB connections. The same principle applies to outbound HTTP — I configure axios with a custom `https.Agent` with `keepAlive: true` so TCP connections to external services are reused across requests."

## 3. Keep-Alive & Timeouts

### What is Keep-Alive?

**Keep-Alive** is a mechanism that keeps TCP connections open after a request/response cycle, allowing them to be reused for subsequent requests.

**Without Keep-Alive:**
```
Request 1:
  [TCP handshake: SYN → SYN-ACK → ACK]  ~1 RTT
  [TLS handshake]                         ~2 RTT
  [HTTP Request] → [HTTP Response]
  [TCP teardown: FIN → FIN-ACK]
  Total: ~4 RTT + transfer time

Request 2:
  [TCP handshake again]
  [TLS handshake again]
  ... repeated overhead every time!
```

**With Keep-Alive:**
```
Request 1:
  [TCP handshake: SYN → SYN-ACK → ACK]  ~1 RTT
  [TLS handshake]                         ~2 RTT
  [HTTP Request 1] → [HTTP Response 1]
  ← Connection kept open →
  [HTTP Request 2] → [HTTP Response 2]   ← No handshake! Just the request.
  [HTTP Request 3] → [HTTP Response 3]   ← Same connection!
  ...
  [TCP teardown after idle timeout]
```

**HTTP Keep-Alive headers:**
```
Request:
  Connection: keep-alive

Response:
  Connection: keep-alive
  Keep-Alive: timeout=60, max=1000
  │                  │         │
  │                  │         └── Max requests on this connection
  │                  └── Close if idle for 60 seconds
  └── Header name
```

**Note: In HTTP/1.1, keep-alive is ON by default.** You opt out with `Connection: close`.
In HTTP/2, keep-alive is always on — connections are persistent with multiplexing.

---

### Nginx Keep-Alive Configuration

```nginx
http {
    # Keep-alive with CLIENTS (browsers, CDNs)
    keepalive_timeout     65;    # Keep connection open 65s after last request
    keepalive_requests  1000;    # Max requests per connection before closing

    # Keep-alive with BACKENDS (upstream servers)
    upstream backend {
        server 10.0.1.1:8080;
        server 10.0.1.2:8080;
        keepalive 32;            # Keep 32 idle connections per worker to backend
    }

    server {
        location / {
            proxy_pass http://backend;
            proxy_http_version 1.1;           # HTTP/1.1 required for keep-alive
            proxy_set_header Connection "";   # Clear "close" header from client
        }
    }
}
```

**Without `proxy_http_version 1.1` + `proxy_set_header Connection ""`:**
Nginx defaults to HTTP/1.0 to backends → no keep-alive → new TCP connection per request → all the overhead you were trying to avoid.

### Timeouts — The Complete Picture

Timeouts prevent your service from hanging forever waiting for something that will never come. Every external call needs a timeout. Every server needs to protect itself.

**The full timeout chain:**

```
Client ──[connect_timeout]──► Load Balancer ──[proxy_connect_timeout]──► App Server
                                                                              │
                                                                         [db connect_timeout]
                                                                              │
                                                                         Database

Each arrow needs its own timeout. If any one is missing,
you can have indefinitely hanging requests.
```

---

### Types of Timeouts (Know All of These)

**1. Connection Timeout**
How long to wait to **establish** a connection. If the server doesn't respond to the TCP SYN within this time, fail.

```
Typical values: 2-5 seconds
Too short: false failures on slow networks
Too long:  users wait a long time to see "server down" error

// Node.js
const client = await pool.connect();
// pool.connectionTimeoutMillis = 3000 → fail after 3s if no connection available
```

**2. Read Timeout (Receive Timeout)**
How long to wait for **data to arrive** after a connection is established.

```
Typical values: 5-60 seconds (depends on operation)

Examples:
  Simple DB query:    5-10s read timeout
  Large report query: 30-120s read timeout
  File upload:        300s read timeout

// Axios
axios.get('/api', { timeout: 5000 })  // 5s total (connect + read)

// Separate connect + read:
const http = require('http');
req.setTimeout(5000);  // 5s for response to start arriving
```

**3. Write Timeout (Send Timeout)**
How long to wait when **sending data** (e.g., request body).

```
Nginx:
  proxy_send_timeout 10s;  // 10s between successive sends to backend

Node.js HTTP server:
  server.timeout = 30000;  // 30s for entire request including upload
```

**4. Idle Timeout (Keep-Alive Timeout)**
How long an idle connection can stay open before being closed.

```
Server side:
  Nginx: keepalive_timeout 65s
  Node.js: server.keepAliveTimeout = 65000ms

Client/Pool side:
  pg pool: idleTimeoutMillis = 30000 (close idle DB connection after 30s)
  axios agent: freeSocketTimeout = 30000 (close idle HTTP socket after 30s)

AWS Load Balancer: idle_timeout = 60s (default)
  → If your backend processes a request for > 60s, LB closes connection
  → Set longer than your longest request processing time!
```

**5. Socket Timeout**
Total time from when the socket is opened to when the last byte is received. Umbrella timeout.

**6. Request Timeout / Deadline**
Total end-to-end time allowed for a complete request, including retries.

```javascript
// Context deadline / AbortController (Node.js)
const controller = new AbortController();
const timeoutId = setTimeout(() => controller.abort(), 5000);  // 5s deadline

try {
  const response = await fetch('https://api.example.com/data', {
    signal: controller.signal
  });
  const data = await response.json();
} catch (err) {
  if (err.name === 'AbortError') {
    console.error('Request timed out after 5s');
  }
} finally {
  clearTimeout(timeoutId);
}
```

### Timeout Configuration by Layer

### The Keep-Alive Timeout Trap

A very common production bug: **Load balancer closes connections before the server expects**.

```
Problem:
  AWS ALB default idle timeout: 60 seconds
  Node.js server default keepAliveTimeout: 5 seconds (very short!)

What happens:
  1. ALB and Node.js establish a keep-alive connection
  2. Request finishes, connection goes idle
  3. After 5 seconds, Node.js closes the connection
  4. ALB still thinks connection is alive (its 60s timer hasn't expired)
  5. ALB tries to send next request on closed connection
  6. Client sees: 502 Bad Gateway ❌

Fix: Set server.keepAliveTimeout > load balancer's idle timeout

server.keepAliveTimeout = 65000;  // 65s > ALB's 60s default
server.headersTimeout = 66000;    // Must be > keepAliveTimeout
```

---

### Interview-Ready Summary

> "Keep-alive reuses TCP connections across multiple HTTP requests, eliminating the overhead of repeated handshakes. HTTP/1.1 enables it by default; for Nginx proxying to backends, I explicitly set `proxy_http_version 1.1` and `proxy_set_header Connection ''` to enable upstream keep-alive. Timeouts are critical for reliability — every external call needs them. The key types are: connection timeout (how long to wait to establish connection, typically 2-5s), read timeout (how long to wait for data after connected), write timeout (how long for sending), and idle timeout (how long idle keep-alive connections live). A critical production gotcha: Node.js's default `keepAliveTimeout` is 5 seconds, much shorter than AWS ALB's 60-second idle timeout — this causes 502 errors because the LB sends requests on connections the server already closed. Always set `keepAliveTimeout` to slightly higher than your load balancer's idle timeout."

## 4. Latency vs Bandwidth

### The Difference — Crystal Clear

These are two completely different dimensions of network performance that people confuse constantly.

```
LATENCY  = How long does it take for one bit to travel from A to B?
           Measured in: milliseconds (ms)
           Analogy: the time it takes to drive from your house to a store

BANDWIDTH = How much data can travel per second?
            Measured in: Mbps, Gbps
            Analogy: the number of lanes on the highway

They are INDEPENDENT of each other:
  High bandwidth + High latency: Fiber ocean cable (fast but far)
  Low bandwidth  + Low latency:  Bluetooth (close but slow)
  High bandwidth + Low latency:  Local network (fast + close) ← ideal
  Low bandwidth  + High latency: 2G mobile (slow + far) ← worst
```

**The pipe analogy:**
```
Latency  = Length of the pipe (how long water takes to arrive)
Bandwidth = Diameter of the pipe (how much water flows per second)

A longer pipe doesn't change how wide it is.
A wider pipe doesn't make water travel faster.
```

---

### Components of Latency

**Latency is the SUM of multiple delays:**

```
Total Latency = Propagation + Transmission + Processing + Queuing

1. Propagation Delay
   Time for signal to travel through the medium (speed of light in cable ≈ 2/3 speed of light in vacuum)

   New York → London:   ~70ms  (5,500 km ÷ 200,000 km/s)
   New York → LA:       ~20ms  (4,500 km)
   New York → Singapore: ~170ms (15,000 km)

   You CANNOT reduce propagation delay with software.
   The only fix: get closer (CDN, edge computing, choose nearby AWS region).

2. Transmission Delay
   Time to push all bits of a packet onto the wire.
   = Packet size ÷ Bandwidth

   10 KB packet over 1 Gbps link: 10,000 bytes × 8 ÷ 1,000,000,000 = 0.08ms
   10 KB packet over 1 Mbps link: 10,000 × 8 ÷ 1,000,000 = 80ms ← significant!

   Fix: smaller packets, higher bandwidth link.

3. Processing Delay
   Time for routers, servers to process the packet.
   CPU parsing, routing table lookup, application logic.
   Fix: faster hardware, optimize code, reduce computation.

4. Queuing Delay
   Time spent waiting in buffers when the network is congested.
   Most variable — can spike from 0ms to seconds under load.
   Fix: rate limiting, QoS, faster link (reduce queuing), TCP BBR.
```

---

### RTT — Round Trip Time

In practice, you care about **RTT** — the time for a request to go out AND a response to come back. A web request includes multiple RTTs:

```
DNS Resolution:     1 RTT (to DNS server and back)
TCP Handshake:      1 RTT (SYN + SYN-ACK)
TLS Handshake:      1-2 RTT (certificate exchange)
HTTP Request:       1 RTT (request + response)

Total for first HTTPS request: 4-5 RTTs

At 100ms RTT (US to Europe):
  4-5 RTTs = 400-500ms of pure round-trip overhead
  BEFORE any application data is processed!

This is why CDNs, HTTP/2, and TLS 1.3 matter so much:
  CDN:   Reduces RTT from 100ms to 5ms → 4 RTTs = 20ms instead of 400ms
  HTTP/2: Multiplexes — multiple requests in 1 RTT
  TLS 1.3: 1-RTT handshake vs TLS 1.2's 2-RTT
```

---

### Latency vs Bandwidth: When Does Each Matter?

```
LATENCY-BOUND scenarios (increasing bandwidth won't help):
  Small requests: API calls, DB queries, DNS lookups
  Interactive: real-time chat, gaming, live video
  Many sequential requests: waterfall of dependencies
  Slow start: TCP slow start limits initial throughput

  Example: Your API returns 1 KB of JSON.
  On 1 Gbps link: transmission = 0.008ms (negligible)
  Latency = 50ms RTT
  Making link 10 Gbps: now transmission = 0.0008ms — no improvement!
  The bottleneck is latency, not bandwidth.

BANDWIDTH-BOUND scenarios (reducing latency won't help):
  Large transfers: video uploads/downloads, backups, bulk exports
  Parallel downloads: many independent large files
  High-throughput streaming: 4K video, data pipelines

  Example: Downloading a 1 GB file over 100ms RTT link.
  On 10 Mbps: takes 800 seconds (bandwidth bound)
  On 1 Gbps: takes 8 seconds
  Reducing RTT to 10ms: almost no change — it's bandwidth that matters!
```

---

### How to Measure Latency

```bash

ping -c 10 api.example.com
# Look at avg: 45.2ms, and variability (min/max/stddev)

# Traceroute — see each hop and its latency
traceroute api.example.com
# Shows: your router → ISP → backbone → destination

# Output:

```

### Latency Optimization Strategies

**1. Reduce Network Hops (CDN / Edge)**
```
Origin in US-East → Users in India: ~200ms
CDN PoP in Mumbai → Users in India:  ~5ms

Deploy to CDN, use edge computing.
Choose the right AWS region for your users.
```

**2. Connection Reuse (Keep-Alive, HTTP/2)**
```
HTTP/1.1 + Keep-Alive:   saves 1 TCP handshake RTT per connection (after first)
HTTP/2 multiplexing:      ALL resources in 1 RTT (vs N sequential RTTs)
TLS 1.3 session resumption: 0-RTT for returning clients
```

**3. Reduce Payload Size**
```
Transmission delay = payload / bandwidth
Smaller payload → less transmission time

Techniques:
  gzip/brotli compression (text: 60-80% reduction)
  Return only requested fields (GraphQL, sparse fieldsets)
  Pagination (don't return 10,000 records)
  Binary protocols (Protobuf vs JSON: 5-10x smaller)
```

**4. Parallelize (Eliminate Sequential Chains)**
```
Sequential (slow):
  await getUserProfile(id);           // 50ms
  await getUserOrders(id);            // 50ms
  await getUserRecommendations(id);   // 50ms
  Total: 150ms

Parallel (fast):
  const [profile, orders, recommendations] = await Promise.all([
    getUserProfile(id),               // ↓ all start simultaneously
    getUserOrders(id),                // ↓
    getUserRecommendations(id),       // ↓
  ]);
  Total: 50ms (max of concurrent, not sum)
```

**5. Reduce TTFB (Time To First Byte) — Application Optimizations**
```
TTFB = DNS + TCP + TLS + Server Processing Time

Optimize server processing:
  → Optimize slow DB queries (add indexes, avoid N+1)
  → Cache expensive computations (Redis)
  → Use streaming responses (send headers early, stream body)
  → Move work to background jobs (don't block the request)

Streaming response (Node.js):
  res.writeHead(200, { 'Content-Type': 'application/json' });
  res.write('{ "users": [');
  for await (const user of db.streamUsers()) {
    res.write(JSON.stringify(user) + ',');  // Stream each row
  }
  res.end(']}');
  // Client starts receiving data immediately, without waiting for full result
```

**6. Prefetch / Preconnect**
```html
<!-- Tell browser to resolve DNS early -->
<link rel="dns-prefetch" href="//api.myapp.com">

<!-- Tell browser to do full TCP+TLS handshake early -->
<link rel="preconnect" href="https://api.myapp.com">

<!-- Prefetch resource you know will be needed -->
<link rel="prefetch" href="/next-page.js">
```

### P99 Latency — Why Averages Lie

```
100 requests, latencies in ms:
  95 requests: 10ms
  4 requests:  50ms
  1 request:   2000ms (2 seconds)

Average: (95×10 + 4×50 + 1×2000) / 100 = 31.5ms
         "Average is 31ms — looks fine!"

But 1 in 100 users waits 2 seconds. That's your P99.
At 1,000 requests/second: 10 users/second have terrible experience!

Percentiles that matter:
  P50  (median):  50% of requests are faster than this
  P95:            95% faster → 1 in 20 is slower
  P99:            99% faster → 1 in 100 is slower
  P99.9:          1 in 1000 is slower (important at high scale)
  P99.99:         1 in 10,000 is slower (financial systems)

Always look at P99, not just average!
Monitor with: Prometheus + Grafana, Datadog APM, New Relic
```

---

### Interview-Ready Summary

> "Latency is the time for data to travel from A to B — it has four components: propagation (speed of light, irreducible), transmission (packet size ÷ bandwidth), processing (CPU/routing), and queuing (congestion). Bandwidth is throughput capacity. They're independent — a high-bandwidth link can still have high latency. Small API responses (1-2 KB) are almost always latency-bound, not bandwidth-bound. To optimize latency: use CDNs to reduce propagation delay, HTTP/2 multiplexing to avoid sequential RTTs, connection keep-alive to reuse TCP connections, payload compression, and parallelizing independent calls with `Promise.all`. For TTFB: index slow queries, cache with Redis, stream responses. Always monitor P99 latency, not just averages — a 2-second outlier at P99 means 1 in 100 users has a terrible experience, which at 1,000 req/s is 10 bad experiences per second."

## 5. Retry Strategies

### Why Retries?

Distributed systems fail transiently — network blips, brief overloads, restarting servers. A request that fails once often succeeds on the second attempt.

```
Without retries:
  Your service → External API → temporary network blip
  Result: 503 error returned to user ❌

With retries:
  Your service → External API → blip → RETRY
              → External API → SUCCESS ✅
  User never sees the error
```

**BUT retries can make things worse if done naively.**

---

### The Problem With Naive Retries

```
Scenario: Your API calls a payment service.
Payment service is overloaded (100% CPU, response time: 5s)

Without retries: 1,000 requests/second hitting payment service

With naive retries (retry immediately, 3 times):
  1,000 × 3 retries = 3,000 requests/second → payment service is now 3× more overloaded!
  This is called a RETRY STORM — retries make the outage worse!

Solution: Exponential backoff + jitter to spread out retries.
```

---

### Exponential Backoff

Wait **exponentially longer** between each retry attempt.

```
Attempt 1: Request fails
  Wait: 1 second
Attempt 2: Request fails
  Wait: 2 seconds  (1 × 2)
Attempt 3: Request fails
  Wait: 4 seconds  (2 × 2)
Attempt 4: Request fails
  Wait: 8 seconds  (4 × 2)
Attempt 5: Request fails
  Give up.

Formula:
  wait_time = base_delay × (2 ^ attempt_number)
  wait_time = 1s × 2^0 = 1s  (first retry)
  wait_time = 1s × 2^1 = 2s
  wait_time = 1s × 2^2 = 4s
  wait_time = 1s × 2^3 = 8s

Cap at max_delay (e.g., 30s) to prevent infinite waits:
  wait_time = min(base × 2^attempt, max_delay)
```

---

### Jitter — Preventing the Thundering Herd

Without jitter, all retrying clients wait the **same amount of time** and then all hit the server simultaneously.

```
WITHOUT jitter (synchronized retries):
  Time 0: 1,000 clients fail
  Time 1s: ALL 1,000 clients retry at exactly the same moment → 1,000 req spike
  Retry storm 💥

WITH jitter (randomized retry times):
  Time 0: 1,000 clients fail
  Client 1:   retries at 0.7s
  Client 2:   retries at 1.3s
  Client 3:   retries at 0.2s
  ...
  Retries spread out over the full window → smooth load ✅
```

**Jitter strategies:**

```
Full Jitter (best for most cases):
  wait_time = random(0, base × 2^attempt)
  Completely random within the exponential window

Equal Jitter:
  half = base × 2^attempt / 2
  wait_time = half + random(0, half)
  Keeps a minimum wait while still spreading out

Decorrelated Jitter (AWS recommendation):
  wait_time = random(base, previous_wait × 3)
  Can grow independently per client (best spread)
```

---

### Implementing Retry with Backoff + Jitter

```javascript
async function retryWithBackoff(fn, options = {}) {
  const {
    maxAttempts   = 3,
    baseDelay     = 1000,   // 1 second base
    maxDelay      = 30000,  // 30 second cap
    jitter        = true,
    shouldRetry   = (err) => true   // Retry all errors by default
  } = options;

  let lastError;

  for (let attempt = 0; attempt < maxAttempts; attempt++) {
    try {
      return await fn();   // Try the operation
    } catch (err) {
      lastError = err;

      // Don't retry if it's a non-retryable error
      if (!shouldRetry(err)) throw err;

      // Don't wait after the last attempt
      if (attempt === maxAttempts - 1) break;

      // Calculate backoff
      const exponential = baseDelay * Math.pow(2, attempt);
      const capped      = Math.min(exponential, maxDelay);

      // Add full jitter
      const delay = jitter ? Math.random() * capped : capped;

      console.log(`Attempt ${attempt + 1} failed. Retrying in ${Math.round(delay)}ms...`);
      await new Promise(resolve => setTimeout(resolve, delay));
    }
  }

  throw lastError;   // All attempts exhausted
}

// Usage
const user = await retryWithBackoff(
  () => apiClient.get('/api/users/42'),
  {
    maxAttempts: 4,
    baseDelay: 500,
    shouldRetry: (err) => {
      // Only retry on transient errors
      if (err.response?.status === 400) return false;  // Bad request — don't retry
      if (err.response?.status === 401) return false;  // Auth — don't retry
      if (err.response?.status === 404) return false;  // Not found — don't retry
      if (err.response?.status === 429) return true;   // Rate limited — retry after delay
      if (err.response?.status >= 500)  return true;   // Server error — retry
      if (err.code === 'ECONNRESET')    return true;   // Network error — retry
      return true;
    }
  }
);
```

---

### What to Retry vs What Not to Retry

```
✅ SAFE to retry (idempotent):
  Network errors (ECONNRESET, ETIMEDOUT)
  5xx server errors (500, 502, 503, 504)
  429 Too Many Requests (after respecting Retry-After header)
  GET, HEAD, OPTIONS requests (inherently safe)
  Idempotent operations (with idempotency key)

❌ NEVER retry blindly:
  4xx client errors (400, 401, 403, 404) — retrying won't help!
  POST without idempotency key — could create duplicate records!
  Non-idempotent PUT/PATCH without idempotency guarantee

Handling 429 (Rate Limited):
  If response includes Retry-After header:
    Retry-After: 30    ← wait exactly this many seconds
  Use that value instead of your own backoff calculation.

  const retryAfter = parseInt(err.response.headers['retry-after']) || backoffDelay;
  await new Promise(resolve => setTimeout(resolve, retryAfter * 1000));
```

---

### Circuit Breaker Pattern

**Circuit breaker** is the most important reliability pattern for microservices. It prevents cascading failures.

**The problem without circuit breaker:**
```
Payment service is DOWN (returns errors in 30 seconds)
Your API calls payment service → waits 30s → gets error → returns 503
User retries → same thing
100 users simultaneously → 100 connections waiting 30s each
Your server threads/connections exhausted → YOUR service goes down too!
The failure CASCADED. 😱
```

**Circuit Breaker States:**

```
                   failure_threshold exceeded
CLOSED ────────────────────────────────────────► OPEN
(normal, requests flow)                         (fails fast, no requests pass)
         ▲                                              │
         │                                              │ reset_timeout elapsed
         │       success                                ▼
         └────────────────────────── HALF-OPEN
                                     (probe: let one request through)
                                     → if success: CLOSED
                                     → if failure: OPEN again
```

```
CLOSED state (normal):
  → Requests pass through
  → Track failure rate (e.g., 50% of last 100 requests failed)
  → If failure_threshold exceeded → OPEN

OPEN state (tripped):
  → ALL requests fail immediately (don't call downstream at all)
  → Return cached fallback or error immediately
  → After reset_timeout (e.g., 30s): transition to HALF-OPEN
  → Service can recover without being bombarded

HALF-OPEN state (testing):
  → Allow ONE probe request through
  → If success → CLOSED (service recovered) ✅
  → If failure → OPEN again (service still down) ❌
```

**Circuit Breaker Implementation:**

```javascript
class CircuitBreaker {
  constructor(fn, options = {}) {
    this.fn = fn;
    this.state = 'CLOSED';

    this.failureThreshold  = options.failureThreshold  || 5;    // Open after 5 failures
    this.successThreshold  = options.successThreshold  || 2;    // Close after 2 successes
    this.timeout           = options.timeout           || 30000; // Stay open 30s
    this.requestTimeout    = options.requestTimeout    || 5000;  // 5s per request

    this.failureCount = 0;
    this.successCount = 0;
    this.lastFailureTime = null;
    this.nextAttemptTime = null;
  }

  async call(...args) {
    if (this.state === 'OPEN') {
      // Check if it's time to try again
      if (Date.now() < this.nextAttemptTime) {
        throw new Error(`Circuit OPEN — service unavailable. Retry after ${
          Math.ceil((this.nextAttemptTime - Date.now()) / 1000)
        }s`);
      }
      // Transition to HALF-OPEN
      this.state = 'HALF-OPEN';
      console.log('Circuit → HALF-OPEN: probing...');
    }

    try {
      // Execute with timeout
      const result = await Promise.race([
        this.fn(...args),
        new Promise((_, reject) =>
          setTimeout(() => reject(new Error('Request timeout')), this.requestTimeout)
        )
      ]);

      this.onSuccess();
      return result;

    } catch (err) {
      this.onFailure();
      throw err;
    }
  }

  onSuccess() {
    this.failureCount = 0;

    if (this.state === 'HALF-OPEN') {
      this.successCount++;
      if (this.successCount >= this.successThreshold) {
        this.state = 'CLOSED';
        this.successCount = 0;
        console.log('Circuit → CLOSED: service recovered ✅');
      }
    }
  }

  onFailure() {
    this.failureCount++;
    this.lastFailureTime = Date.now();
    this.successCount = 0;

    if (this.failureCount >= this.failureThreshold || this.state === 'HALF-OPEN') {
      this.state = 'OPEN';
      this.nextAttemptTime = Date.now() + this.timeout;
      console.log(`Circuit → OPEN: too many failures. Next attempt at ${
        new Date(this.nextAttemptTime).toISOString()
      }`);
    }
  }

  getState() {
    return {
      state:       this.state,
      failures:    this.failureCount,
      lastFailure: this.lastFailureTime
    };
  }
}

// Usage
const paymentBreaker = new CircuitBreaker(
  (orderId, amount) => paymentService.charge(orderId, amount),
  { failureThreshold: 5, timeout: 30000, requestTimeout: 5000 }
);

app.post('/checkout', async (req, res) => {
  try {
    const result = await paymentBreaker.call(req.body.orderId, req.body.amount);
    res.json({ success: true, result });

  } catch (err) {
    if (err.message.includes('Circuit OPEN')) {
      // Circuit is open — fail fast with informative message
      return res.status(503).json({
        error: 'Payment service temporarily unavailable',
        retryAfter: 30
      });
    }
    res.status(500).json({ error: err.message });
  }
});
```

### Retry + Circuit Breaker Together

The complete resilience pattern:

```javascript
async function resilientCall(circuitBreaker, fn, retryOptions) {
  return retryWithBackoff(
    () => circuitBreaker.call(fn),    // Circuit breaker wraps each attempt
    {
      ...retryOptions,
      shouldRetry: (err) => {
        // Don't retry if circuit is open (it will immediately fail anyway)
        if (err.message.includes('Circuit OPEN')) return false;
        // Retry on transient errors
        return isTransientError(err);
      }
    }
  );
}
```

**The combination:**
```
Request fails
  → Retry with backoff (transient? try again)
  → After failure_threshold failures: Circuit OPENS
  → All subsequent requests fail IMMEDIATELY (no waiting 30s per attempt!)
  → After 30s: Circuit → HALF-OPEN → one probe request
  → If service recovered: Circuit CLOSES → normal operation resumes
```

### Bulkhead Pattern (Bonus — Goes with Circuit Breaker)

Isolate failures so one slow service doesn't consume all resources.

```
Without bulkhead:
  100 threads total in app
  Payment service goes slow → 100 threads all stuck waiting
  No threads for user service, order service, etc.
  Entire app stalls 😱

With bulkhead (separate thread pools):
  Payment service pool:  20 threads max
  User service pool:     30 threads max
  Order service pool:    30 threads max
  Other:                 20 threads

Payment service slow → only 20 threads affected
User and order service still have 60 threads → app keeps running ✅

In Node.js: limit concurrent calls with a semaphore
const pLimit = require('p-limit');
const paymentLimit = pLimit(20);  // Max 20 concurrent payment calls

await paymentLimit(() => paymentService.charge(orderId, amount));
```

### Interview-Ready Summary

> "Retries make distributed systems resilient to transient failures, but naive immediate retries cause retry storms that worsen outages. Exponential backoff increases wait time geometrically between attempts (1s, 2s, 4s, 8s) so failing services have time to recover. Jitter randomizes the wait time to prevent all retrying clients from hitting the server simultaneously (thundering herd). Only retry idempotent or safe operations — never retry POST without an idempotency key, and never retry 4xx errors since they won't succeed. Circuit breaker is the critical pattern for cascading failure prevention — it has three states: Closed (normal), Open (fails fast after threshold failures), and Half-Open (probes with one request after timeout). When a service fails, the circuit opens and all subsequent calls fail immediately (rather than waiting for timeouts), protecting your server's resources. After the reset timeout, one probe request tests if the service recovered. I combine retries with circuit breakers: retry transient errors, but if the circuit opens, stop retrying immediately."
