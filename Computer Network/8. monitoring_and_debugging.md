## 1. curl & Postman

### What is curl?

**curl** (Client URL) is a command-line tool for making HTTP (and many other protocol) requests. It's available on every Unix/Linux/macOS system and is the most widely used tool for manual API testing and debugging. No GUI, no setup — just a command.

**Why every backend dev must know curl:**
```
✅ Available everywhere (servers, Docker containers, CI pipelines)
✅ Scriptable (automate API tests in bash)
✅ Shows raw HTTP headers (see exactly what's being sent/received)
✅ Test APIs directly on the server (no browser needed)
✅ Debug production issues without installing anything
✅ Copy-paste from browser DevTools → "Copy as cURL"
```

---

### curl Fundamentals

**Basic GET request:**
```bash
curl https://api.example.com/users

# With verbose output (-v) — shows request AND response headers
curl -v https://api.example.com/users

# Even more detail (--trace) — shows raw bytes
curl --trace - https://api.example.com/users
```

**Understanding curl -v output:**
```
* Trying 142.250.80.46:443...           ← DNS resolved, TCP connecting
* Connected to api.example.com          ← TCP connected
* TLS handshake...                      ← TLS negotiation
* SSL connection using TLSv1.3          ← TLS version used

> GET /users HTTP/1.1                   ← Request line (> = sent)
> Host: api.example.com                 ← Request headers
> User-Agent: curl/7.88.1
> Accept: */*
>                                       ← Blank line = end of headers

< HTTP/1.1 200 OK                       ← Response line (< = received)
< Content-Type: application/json        ← Response headers
< Content-Length: 256
< Cache-Control: no-cache
<                                       ← Blank line = end of headers

[{"id":1,"name":"Alice"}...]            ← Response body
```

### curl: HTTP Methods

```bash
# GET (default)
curl https://api.example.com/users
curl -X GET https://api.example.com/users   # explicit

# POST with JSON body
curl -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -d '{"name": "Alice", "email": "alice@example.com"}'

# POST with JSON from file
curl -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -d @user.json

# PUT — replace resource
curl -X PUT https://api.example.com/users/42 \
  -H "Content-Type: application/json" \
  -d '{"name": "Alice Smith", "email": "alice@example.com"}'

# PATCH — partial update
curl -X PATCH https://api.example.com/users/42 \
  -H "Content-Type: application/json" \
  -d '{"name": "Alice Smith"}'

# DELETE
curl -X DELETE https://api.example.com/users/42

# HEAD — headers only, no body (check if resource exists)
curl -I https://api.example.com/users/42

# OPTIONS — see allowed methods
curl -X OPTIONS -v https://api.example.com/users
```

---

### curl: Headers & Auth

```bash
# Add custom headers (-H)
curl https://api.example.com/users \
  -H "Authorization: Bearer eyJhbGc..." \
  -H "Accept: application/json" \
  -H "X-Request-ID: abc-123" \
  -H "X-Custom-Header: value"

# Basic auth (-u)
curl -u username:password https://api.example.com/protected

# Bearer token (JWT)
TOKEN="eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
curl https://api.example.com/profile \
  -H "Authorization: Bearer $TOKEN"

# API Key in header
curl https://api.example.com/data \
  -H "X-API-Key: your-api-key-here"

# API Key in query param
curl "https://api.example.com/data?api_key=your-api-key-here"

# Send cookies
curl https://api.example.com/dashboard \
  -H "Cookie: session=abc123; theme=dark"

# Save cookies to file and send on next request
curl -c cookies.txt https://api.example.com/login \
  -X POST -d '{"email":"a@b.com","password":"secret"}'

curl -b cookies.txt https://api.example.com/profile
```

---

### curl: Advanced & Production Debugging

```bash
# Follow redirects (-L) — e.g., HTTP → HTTPS
curl -L http://api.example.com/users

# Show only response headers (-I / --head)
curl -I https://api.example.com/users

# Show response headers + body (-i)
curl -i https://api.example.com/users

# Save response to file (-o)
curl -o response.json https://api.example.com/users

# Suppress progress, show only body (-s = silent)
curl -s https://api.example.com/users | jq .    # pipe to jq for pretty-print

# Fail with non-zero exit code on HTTP error (-f)
curl -f https://api.example.com/users || echo "Request failed!"

# Set timeout (--max-time = total, --connect-timeout = just TCP connect)
curl --max-time 10 --connect-timeout 5 https://api.example.com/users

# Test with specific IP (bypass DNS — test before DNS propagation)
curl --resolve api.example.com:443:54.123.45.67 https://api.example.com/users

# Use a specific network interface
curl --interface eth1 https://api.example.com/users

# Skip SSL certificate verification (⚠️ dev only, NEVER in production)
curl -k https://self-signed.example.com/api

# HTTP/2 explicitly
curl --http2 https://api.example.com/users

# Upload a file (multipart form)
curl -X POST https://api.example.com/upload \
  -F "file=@/path/to/file.pdf" \
  -F "description=My document"

# Multiple headers and measure timing (THE most useful curl command)
curl -s -o /dev/null -w "
  http_code:        %{http_code}
  time_namelookup:  %{time_namelookup}s
  time_connect:     %{time_connect}s
  time_appconnect:  %{time_appconnect}s
  time_ttfb:        %{time_starttransfer}s
  time_total:       %{time_total}s
  size_download:    %{size_download} bytes
" https://api.example.com/users

# Test with proxy (e.g., through Burp Suite for traffic inspection)
curl -x http://127.0.0.1:8080 https://api.example.com/users
```

**Decode JWT on the command line:**
```bash
# Decode JWT payload (middle section) — it's just Base64
echo "eyJzdWIiOiI0MiIsIm5hbWUiOiJBbGljZSJ9" | base64 -d
# → {"sub":"42","name":"Alice"}

# Pipe a full JWT (extract and decode the payload)
TOKEN="header.payload.signature"
echo $TOKEN | cut -d'.' -f2 | base64 -d 2>/dev/null | jq .
```

### curl Cheat Sheet for Common Scenarios

```bash
# Check if an API is up
curl -s -o /dev/null -w "%{http_code}" https://api.example.com/health
# → 200

# Test CORS preflight manually
curl -X OPTIONS https://api.example.com/users \
  -H "Origin: https://myapp.com" \
  -H "Access-Control-Request-Method: POST" \
  -H "Access-Control-Request-Headers: Authorization, Content-Type" \
  -v 2>&1 | grep -i "access-control"

# Send a webhook payload (simulate a webhook)
curl -X POST https://myapp.com/webhooks/stripe \
  -H "Content-Type: application/json" \
  -H "Stripe-Signature: t=123,v1=abc" \
  -d '{"type":"payment_intent.succeeded","data":{"object":{"amount":9900}}}'

# Test rate limiting (send 10 requests fast)
for i in {1..10}; do
  curl -s -o /dev/null -w "Request $i: %{http_code}\n" https://api.example.com/endpoint
done

# GraphQL query
curl -X POST https://api.example.com/graphql \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"query": "{ user(id: \"42\") { name email orders { id total } } }"}'
```

### Postman

**Postman** is a GUI tool for API testing. It's built for teams — saves requests, organizes them, runs automated tests, and generates documentation.

**Why Postman over curl:**
```
✅ Visual interface — easier for complex request bodies
✅ Collections — organize and share requests with your team
✅ Environment variables — switch between dev/staging/prod with one click
✅ Pre-request scripts — generate tokens, set timestamps automatically
✅ Tests — write assertions on responses (status code, body content)
✅ Mock servers — stub APIs before backend is built
✅ Newman — run Postman collections from CLI (for CI/CD)
✅ API documentation — auto-generate from your collection
```

---

### Postman: Key Concepts

**Environments & Variables:**
```javascript
// Environment: "Development"
base_url = http://localhost:3000
token    = (empty — set by login request)

// Environment: "Production"
base_url = https://api.myapp.com
token    = (empty — set by login request)

// Using variables in request URL:
{{base_url}}/users/42

// Using in headers:
Authorization: Bearer {{token}}
```

**Pre-request Script (auto-login before each request):**
```javascript
// Pre-request Script tab
const loginResponse = await pm.sendRequest({
    url: pm.environment.get('base_url') + '/auth/login',
    method: 'POST',
    header: { 'Content-Type': 'application/json' },
    body: {
        mode: 'raw',
        raw: JSON.stringify({
            email: pm.environment.get('email'),
            password: pm.environment.get('password')
        })
    }
});

const body = loginResponse.json();
pm.environment.set('token', body.accessToken);
// Now {{token}} is set for this request
```

**Tests (Postman assertions):**
```javascript
// Tests tab — runs after response is received
pm.test("Status is 200", () => {
    pm.response.to.have.status(200);
});

pm.test("Response has data array", () => {
    const body = pm.response.json();
    pm.expect(body).to.have.property('data');
    pm.expect(body.data).to.be.an('array');
    pm.expect(body.data.length).to.be.greaterThan(0);
});

pm.test("Response time under 500ms", () => {
    pm.expect(pm.response.responseTime).to.be.below(500);
});

pm.test("Content-Type is JSON", () => {
    pm.response.to.have.header('Content-Type', /json/);
});

// Set variable from response (for chained requests)
const body = pm.response.json();
pm.environment.set('userId', body.data.id);
// Now {{userId}} can be used in the next request: GET {{base_url}}/users/{{userId}}
```

**Newman — run Postman collections in CI/CD:**
```bash
# Install Newman
npm install -g newman

# Run a collection
newman run MyAPI.postman_collection.json \
  --environment Production.postman_environment.json \
  --reporters cli,html \
  --reporter-html-export report.html

# In GitHub Actions / GitLab CI:
newman run collection.json -e env.json --bail
# --bail: stop on first test failure (exit code 1 = CI fails)
```

---

### Interview-Ready Summary

> "curl is the go-to CLI tool for API testing — available everywhere with no setup. I use `-v` for verbose output (see all headers), `-i` to include response headers with the body, `-H` to add headers, `-d` for request bodies, and `-X` for methods. The timing format with `-w` is invaluable: it shows DNS lookup, TCP connect, TLS handshake, TTFB, and total time separately — pinpointing exactly where a request is slow. `--resolve` lets me test a server by IP before DNS propagates, and `-k` skips cert verification for self-signed certs in dev. Postman is the team-oriented GUI alternative — I use environments to switch between dev/staging/prod, pre-request scripts to auto-fetch tokens, and Tests to assert response codes and body structure. Newman runs Postman collections in CI/CD pipelines and fails the build if API tests don't pass."

## 2. Wireshark / tcpdump

### What is Packet Inspection?

Packet inspection tools capture **raw network packets** as they travel through a network interface. You see exactly what bytes are sent and received — including headers, payloads, TCP sequence numbers, everything.

**When you need packet inspection:**
```
✅ Debugging why an API call fails when you can't modify the client
✅ Verifying TLS cipher suites and handshake behavior
✅ Diagnosing TCP connection issues (RST, FIN, retransmissions)
✅ Analyzing performance (see exact timing of each packet)
✅ Debugging binary protocols (gRPC, WebSocket, custom protocols)
✅ Confirming what actually goes over the wire (vs what your code thinks)
✅ Debugging DNS issues at the packet level
✅ Security analysis (detect unexpected traffic)
```

**Analogy:**
> Application logs are like reading a diary about what happened. Packet inspection is like watching a security camera — you see every single thing that actually happened on the network, byte by byte.

---

### tcpdump — Command-Line Packet Capture

**tcpdump** is the standard CLI tool for capturing packets. Available on every Linux/macOS server. **This is what you use on production servers** (no GUI needed).

**Basic syntax:**
```bash
tcpdump [options] [filter expression]

# Must run as root or with sudo
# Ctrl+C to stop capture
```

**Capture basics:**
```bash
# Capture ALL traffic on all interfaces
sudo tcpdump

# Capture on specific interface
sudo tcpdump -i eth0       # specific interface
sudo tcpdump -i any        # all interfaces
sudo tcpdump -i lo         # loopback (localhost traffic)

# List available interfaces
sudo tcpdump -D

# Capture N packets then stop (-c)
sudo tcpdump -c 100 -i eth0

# Save capture to file (-w) — open later in Wireshark
sudo tcpdump -i eth0 -w capture.pcap

# Read from saved file (-r)
tcpdump -r capture.pcap
```

**Display options:**
```bash
# -n: Don't resolve IP to hostname (faster, clearer IPs)
# -nn: Don't resolve IPs OR port numbers (see actual port numbers, not "http")
# -v: Verbose (more packet details)
# -vv: Very verbose
# -X: Show packet contents in hex AND ASCII (see the payload!)
# -A: Show payload in ASCII only (great for HTTP)
# -S: Show absolute TCP sequence numbers
# -tttt: Human-readable timestamps

# Most useful combination for HTTP debugging:
sudo tcpdump -i eth0 -nnvvA -s 0 port 80
#                    ↑↑↑↑↑  ↑
#                    flags   -s 0 = capture full packet (no truncation)
```

---

### tcpdump Filter Expressions

This is where tcpdump becomes powerful. Filter exactly what you care about.

**By host:**
```bash
# Traffic to/from specific IP
sudo tcpdump host 192.168.1.5

# Traffic FROM a host
sudo tcpdump src host 192.168.1.5

# Traffic TO a host
sudo tcpdump dst host 192.168.1.5

# Traffic to/from domain
sudo tcpdump host api.example.com
```

**By port:**
```bash
# All traffic on port 80
sudo tcpdump port 80

# All HTTP + HTTPS
sudo tcpdump port 80 or port 443

# Traffic on port 5432 (PostgreSQL)
sudo tcpdump port 5432

# Port range
sudo tcpdump portrange 8080-8090
```

**By protocol:**
```bash
sudo tcpdump tcp       # TCP only
sudo tcpdump udp       # UDP only
sudo tcpdump icmp      # Ping/traceroute packets
sudo tcpdump arp       # ARP packets (MAC ↔ IP resolution)
```

**Combining filters:**
```bash
# AND: both conditions must match
sudo tcpdump host 192.168.1.5 and port 443

# OR: either condition
sudo tcpdump port 80 or port 8080

# NOT: exclude
sudo tcpdump not port 22    # Everything except SSH

# Complex: debug API server, ignore health checks
sudo tcpdump -i eth0 -nnvvA \
  "host api.example.com and port 443 and not (src host 10.0.1.1 and dst port 80)"
```

**Real-world debugging scenarios:**
```bash
# Watch all HTTP traffic on your app server
sudo tcpdump -i eth0 -nnA -s 0 'tcp port 8080'

# Debug database connections (see SQL queries over the wire)
sudo tcpdump -i lo -nnA -s 0 'tcp port 5432'
# Note: PostgreSQL wire protocol shows SQL in plaintext!

# Capture DNS queries (debug DNS issues)
sudo tcpdump -i eth0 -nnvv 'udp port 53'

# Watch for TCP resets (connection issues)
sudo tcpdump -i eth0 'tcp[tcpflags] & tcp-rst != 0'

# Watch for SYN packets (new connections being established)
sudo tcpdump -i eth0 'tcp[tcpflags] & tcp-syn != 0 and tcp[tcpflags] & tcp-ack = 0'

# Capture traffic to save for Wireshark analysis
sudo tcpdump -i eth0 -s 0 -w /tmp/debug.pcap port 443
# Transfer to your machine: scp server:/tmp/debug.pcap .
# Open in Wireshark for visual analysis
```

**Reading tcpdump output:**
```
14:23:45.123456 IP 192.168.1.5.54231 > 93.184.216.34.443: Flags [S] seq 0
│              │   │                   │                   │      │   │
│              │   │                   │                   │      │   └── TCP seq number
│              │   │                   │                   │      └── Flags: [S]=SYN, [SA]=SYN-ACK
│              │   │                   │                   │          [A]=ACK, [F]=FIN, [R]=RST, [P]=PUSH
│              │   │                   │                   └── Destination IP:port
│              │   └── Source IP:port (192.168.1.5, port 54231)
│              └── Protocol (IP)
└── Timestamp

Flags meaning:
  [S]    → SYN (connection request)
  [SA]   → SYN-ACK (connection accepted)
  [A]    → ACK (acknowledgment)
  [P.]   → PUSH + ACK (data with push flag)
  [F]    → FIN (graceful close)
  [R]    → RST (reset — abrupt close, connection refused, or error)
  [.]    → just ACK (the dot is ACK in tcpdump shorthand)
```

### Wireshark

**Wireshark** is the GUI version of tcpdump — much more powerful for deep analysis. Color-coded packets, protocol dissection, follow TCP streams, statistics.

**Wireshark display filter syntax (different from tcpdump BPF):**
```
# Filter by IP
ip.addr == 192.168.1.5
ip.src == 192.168.1.5
ip.dst == 192.168.1.5

# Filter by port
tcp.port == 443
tcp.port == 80 || tcp.port == 8080

# Filter by protocol
http
https
dns
tls
websocket
http2

# HTTP specific
http.request.method == "POST"
http.response.code == 200
http.response.code >= 400      # All error responses

# TLS
tls.handshake.type == 1        # Client Hello
tls.handshake.type == 2        # Server Hello

# TCP flags
tcp.flags.syn == 1             # All SYN packets
tcp.flags.reset == 1           # All RST packets (connection resets!)
tcp.analysis.retransmission    # All retransmissions (packet loss!)
tcp.analysis.duplicate_ack     # Duplicate ACKs (also indicates packet loss)
tcp.analysis.zero_window       # Receiver buffer full (flow control issue)

# Find HTTP errors
http.response.code >= 500

# Search packet contents (contains keyword)
tcp contains "password"
http contains "Bearer"
```

**Key Wireshark features:**
```
Follow Stream:
  Right-click a packet → Follow → TCP Stream
  → See the entire conversation as text
  → Perfect for HTTP debugging (see request + response together)

Statistics → I/O Graph:
  Visualize packet rate over time
  Spot traffic spikes, gaps

Statistics → TCP Stream Graph → Round Trip Time:
  Plot RTT over time
  Spot latency spikes, retransmissions

Analyze → Expert Information:
  Wireshark highlights: retransmissions, resets, out-of-order, duplicate ACKs
  Instant summary of network health issues

Decrypt TLS (if you have the private key):
  Edit → Preferences → Protocols → TLS
  Add your server's RSA private key
  Wireshark decrypts and shows plaintext HTTP!
```

---

### Interview-Ready Summary

> "tcpdump captures raw network packets from the CLI — essential for debugging on servers where you can't install GUI tools. Key flags: `-nn` (don't resolve names), `-A` (show ASCII payload — readable for HTTP), `-s 0` (capture full packet), `-w file.pcap` (save for Wireshark). Filters use BPF syntax: `host`, `port`, `tcp`, combined with `and`/`or`/`not`. I use it to debug TCP issues (look for RST flags — connection resets, SYN without ACK — refused connections), capture DNS queries, and see what actually goes over the wire. Wireshark is the GUI alternative — I capture on the server with tcpdump, save to .pcap, and open on my machine in Wireshark. Follow TCP Stream shows the full HTTP conversation. `tcp.flags.reset == 1` finds connection resets, `tcp.analysis.retransmission` finds packet loss. I can decrypt TLS traffic in Wireshark by providing the server's private key."

## 3. netstat / ss / lsof

### Why These Tools?

These three tools answer the most common network debugging questions:
- "What's listening on port X?"
- "Who's connected to my server right now?"
- "Which process is using port 8080?"
- "How many connections do I have to the database?"
- "Is my app leaking connections?"

---

### netstat — Network Statistics

**netstat** is the classic tool (available everywhere, but being replaced by `ss`).

```bash
# Show all listening ports
netstat -tlnp
# -t: TCP only
# -l: Listening sockets only
# -n: Don't resolve names (show IP:port numbers)
# -p: Show PID and program name (requires root)

# Output:
Proto  Local Address    Foreign Address  State   PID/Program
tcp    0.0.0.0:80      0.0.0.0:*        LISTEN  1234/nginx
tcp    0.0.0.0:443     0.0.0.0:*        LISTEN  1234/nginx
tcp    127.0.0.1:5432  0.0.0.0:*        LISTEN  5678/postgres
tcp    0.0.0.0:6379    0.0.0.0:*        LISTEN  9012/redis-server
tcp    0.0.0.0:8080    0.0.0.0:*        LISTEN  3456/node

# Show ALL connections (listening + established + time_wait etc.)
netstat -anp

# Show UDP ports too
netstat -tulnp

# Show established connections only
netstat -tnp | grep ESTABLISHED

# Count connections by state
netstat -ant | awk '{print $6}' | sort | uniq -c | sort -rn
# 1500 ESTABLISHED
#  234 TIME_WAIT
#   45 CLOSE_WAIT
#    3 FIN_WAIT2

# Watch connection count to a specific port (refresh every 2s)
watch -n 2 'netstat -ant | grep :5432 | wc -l'
```

**TCP connection states:**
```
LISTEN      → Server waiting for connections on this port
SYN_SENT    → Client sent SYN, waiting for SYN-ACK
SYN_RECEIVED → Server received SYN, sent SYN-ACK, waiting for ACK
ESTABLISHED → Connection active, data flowing
FIN_WAIT1   → We sent FIN (starting to close)
FIN_WAIT2   → We got ACK of our FIN, waiting for other side's FIN
CLOSE_WAIT  → Other side sent FIN, we need to close (app must call close())
TIME_WAIT   → We sent final ACK, waiting 2×MSL before fully closing
CLOSED      → No connection

Red flags:
  Many CLOSE_WAIT → Your application isn't closing connections properly!
                    Common connection leak bug.
  Many TIME_WAIT  → High connection turnover (normal, but can be tuned)
                    Each TIME_WAIT takes ~60s (2 minutes default) to clear
  ESTABLISHED growing → Connection pool growing, possible leak
```

---

### ss — Socket Statistics (Modern netstat)

**ss** is the modern replacement for netstat — faster, more features, better output.

```bash
# Show listening TCP sockets (equivalent to netstat -tlnp)
ss -tlnp

# Detailed output with process info
ss -tlnp
# State   Recv-Q  Send-Q  Local Address:Port  Peer Address:Port  Process
# LISTEN  0       128     0.0.0.0:80          0.0.0.0:*          users:(("nginx",pid=1234,fd=6))
# LISTEN  0       128     0.0.0.0:443         0.0.0.0:*          users:(("nginx",pid=1234,fd=7))
# LISTEN  0       511     127.0.0.1:5432      0.0.0.0:*          users:(("postgres",pid=5678,fd=5))

# All sockets (listening + established)
ss -antp

# Filter by state
ss -tnp state established           # All established
ss -tnp state time-wait             # All TIME_WAIT
ss -tnp state close-wait            # All CLOSE_WAIT (connection leak!)

# Filter by port
ss -tnp 'sport = :443'             # Local port 443
ss -tnp 'dport = :5432'            # Remote (destination) port 5432
ss -tnp '( dport = :5432 or sport = :5432 )'  # Either side

# Filter by remote address (all connections to a specific server)
ss -tnp dst 10.0.1.5

# Show send/receive queue sizes (Recv-Q/Send-Q)
ss -tn
# Recv-Q > 0 → Data in receive buffer not yet read by app (slow consumer!)
# Send-Q > 0 → Data in send buffer waiting to be sent (slow network!)

# Summary statistics
ss -s
# Total: 1523
# TCP:   1456 (estab 1234, closed 45, orphaned 2, timewait 178)
# Transport Total  IP   IPv6
# RAW       0      0    0
# UDP       8      4    4
# TCP       1456   892  564

# Watch established connections to DB pool
watch -n 1 'ss -tnp dst 10.0.2.100:5432 | wc -l'

# Show connection timer info (useful for TIME_WAIT duration)
ss -tnop state time-wait
# timer:(timewait,58sec,0)  ← this connection will expire in 58 seconds
```

### lsof — List Open Files

In Linux, **everything is a file** — including network sockets. `lsof` lists ALL open files, including sockets.

```bash
# All network connections
sudo lsof -i

# TCP only
sudo lsof -i TCP

# UDP only
sudo lsof -i UDP

# All connections on port 8080
sudo lsof -i :8080

# All connections on ports 80 and 443
sudo lsof -i :80 -i :443

# What process is using port 8080?
sudo lsof -i :8080
# COMMAND   PID     USER   FD   TYPE  DEVICE SIZE NODE NAME
# node     1234    ubuntu  22u  IPv4  12345  0t0  TCP  *:8080 (LISTEN)

# All connections from a specific process
sudo lsof -p 1234 -i

# All network connections for a specific process name
sudo lsof -c nginx -i

# All connections to a specific remote host
sudo lsof -i @10.0.1.5

# All connections to a specific remote host and port
sudo lsof -i @10.0.1.5:5432

# Count open connections for a process (detect connection leaks)
sudo lsof -p 1234 -i TCP | wc -l

# Watch for file descriptor leaks (connections not being closed)
watch -n 2 'sudo lsof -p $(pgrep node) -i TCP | wc -l'

# Find what's using a port when lsof returns nothing (kernel-level)
sudo fuser 8080/tcp          # Returns PID using port 8080
sudo fuser -k 8080/tcp       # Kill whatever is using it (⚠️ careful!)
```

**Diagnosing connection leaks with lsof:**
```bash
# Check how many connections your Node.js app has
sudo lsof -p $(pgrep -f "node server.js") -i TCP

# Output grows over time without connections closing? → CONNECTION LEAK!
# Compare:
sudo lsof -p PID -i TCP | grep ESTABLISHED | wc -l
# Run every minute — if number grows without bound → leak

# Check for CLOSE_WAIT specifically (app not calling socket.close())
sudo lsof -p PID -i TCP | grep CLOSE_WAIT
```

### Combining These Tools — Real Debugging Workflows

```bash
# SCENARIO 1: "Something is already using port 8080"
sudo lsof -i :8080
sudo ss -tlnp 'sport = :8080'
sudo fuser 8080/tcp     # Quick PID lookup

# SCENARIO 2: "Am I connected to the database?"
ss -tnp dst 10.0.2.100:5432
# Count active DB connections:
ss -tnp dst 10.0.2.100:5432 | grep ESTAB | wc -l

# SCENARIO 3: "How many clients are connected to my API?"
ss -tnp 'sport = :443' state established | wc -l

# SCENARIO 4: "I think my app is leaking connections"
# Step 1: Get your app's PID
pgrep -f "node app.js"    # → 1234

# Step 2: Watch connection count over time
watch -n 5 'sudo lsof -p 1234 -i TCP | grep -v LISTEN | wc -l'

# Step 3: If it grows → check connection states
sudo lsof -p 1234 -i TCP | awk '{print $NF}' | sort | uniq -c

# SCENARIO 5: "Is my server listening on the right interface?"
ss -tlnp
# 0.0.0.0:8080 → listening on ALL interfaces (correct for server)
# 127.0.0.1:8080 → listening on localhost ONLY (wrong if external access needed)
# :::8080 → IPv6 all interfaces

# SCENARIO 6: "Count connection states — is something wrong?"
ss -ant | awk 'NR>1 {print $1}' | sort | uniq -c | sort -rn
# Many CLOSE_WAIT → app not closing connections (code bug)
# Many TIME_WAIT → high connection turnover (may need TCP tuning)
# ESTABLISHED growing → connection pool growing (leak or load)
```

---

### Interview-Ready Summary

> "netstat and ss show network connections and listening ports. I use `ss -tlnp` to see what's listening on which port (the `-p` shows the process name and PID). For connection debugging, `ss -ant` shows all connections with states — CLOSE_WAIT means the application isn't closing connections (connection leak), TIME_WAIT is normal TCP cleanup but can pile up under high throughput. lsof is more powerful — it shows all open files including sockets, so `lsof -i :8080` tells me exactly which process owns a port, and `lsof -p PID -i TCP` shows all network connections for a specific process. I watch this with `watch -n 2` to detect connection leaks — if the count grows without bound, connections aren't being returned to the pool or closed. A common bug: CLOSE_WAIT connections in Node.js apps that don't call `socket.destroy()` in error handlers."

## 4. ping / traceroute / nslookup / dig

### ping — Is the Host Reachable?

**ping** sends ICMP Echo Request packets and measures how long responses take. It tests basic connectivity and measures latency.

```bash
# Basic ping
ping google.com
# PING google.com (142.250.80.46): 56 bytes
# 64 bytes from 142.250.80.46: icmp_seq=0 ttl=118 time=12.4 ms
# 64 bytes from 142.250.80.46: icmp_seq=1 ttl=118 time=11.8 ms
# --- google.com ping statistics ---
# 3 packets transmitted, 3 received, 0% packet loss
# round-trip min/avg/max/stddev = 11.8/12.1/12.4/0.3 ms

# Send N pings then stop (-c)
ping -c 5 google.com

# Set interval between pings (-i)
ping -i 0.2 google.com      # Ping 5 times per second (flood — use carefully)

# Set packet size (-s) — test with large packets (MTU issues)
ping -s 1472 google.com     # 1472 + 28 header = 1500 bytes (typical MTU)

# Ping with TTL limit (-t on macOS, --ttl on Linux)
ping --ttl 5 google.com     # Die after 5 router hops (like traceroute)

# Continuous ping but only show problems (useful for monitoring)
ping -i 1 api.example.com | grep -v "64 bytes"   # Only show failures

# Read ping output:
ttl=118     → Time To Live — decremented at each hop. Started at 128 or 255.
              128-118 = 10 hops away (approximately)
time=12.4ms → Round-trip time
0% packet loss → Connection is stable
```

**What ping tells you:**
```
ping succeeds → IP-level connectivity works (Layer 3 is fine)
ping fails     → Could be:
                 → Host is down
                 → Firewall blocks ICMP (⚠️ many firewalls block ping)
                 → DNS resolution failed (try ping by IP instead)
                 → Network route doesn't exist

⚠️ "Can't ping" does NOT mean "server is down"
   Modern firewalls often block ICMP deliberately.
   Use curl/telnet to test application-level connectivity instead.
```

---

### traceroute — Path Discovery

**traceroute** maps the network path from your machine to a destination, showing every router hop along the way and the latency at each hop.

```bash
# Basic traceroute
traceroute google.com

# On Linux, UDP-based (default):
traceroute google.com

# ICMP-based (sometimes more reliable through firewalls)
traceroute -I google.com    # Linux
traceroute google.com       # macOS uses ICMP by default

# TCP-based (best at bypassing firewalls — use your app's port)
traceroute -T -p 443 api.example.com   # TCP to port 443

# Don't resolve hostnames (-n) — faster
traceroute -n google.com

# Sample output:
traceroute to google.com (142.250.80.46), 30 hops max
 1  192.168.1.1       1.2 ms    1.0 ms    0.9 ms   ← Your router
 2  10.0.0.1          5.3 ms    5.1 ms    5.4 ms   ← ISP's first hop
 3  203.0.113.1      12.1 ms   11.8 ms   11.9 ms   ← ISP backbone
 4  * * *                                           ← Router blocking ICMP
 5  74.125.242.33    13.2 ms   12.9 ms   13.0 ms   ← Google's network
 6  142.250.80.46    13.4 ms   13.2 ms   13.1 ms   ← Destination!

# Reading traceroute:
* * *         → Router doesn't respond to traceroute (firewall/configured to not respond)
              This is NORMAL — doesn't mean packets aren't flowing
High latency spike at one hop → That router (or link to it) is the bottleneck
Latency then drops → Asymmetric routing (return path is different — common)

# mtr — combines ping + traceroute (real-time, continuous)
sudo apt install mtr
mtr google.com
mtr --report --report-cycles 100 api.example.com   # 100 probes, then report
```

### dig — DNS Deep Dive (Prefer Over nslookup)

**dig** (Domain Information Groper) is the professional DNS debugging tool. More detail, more control, scriptable output.

```bash
# Basic A record lookup
dig google.com

# Output anatomy:
; <<>> DiG 9.18 <<>> google.com
;; QUESTION SECTION:
;google.com.           IN  A           ← What we asked for

;; ANSWER SECTION:
google.com.     300    IN  A  142.250.80.46  ← Answer: IP + TTL (300s)

;; AUTHORITY SECTION:                  ← Name servers for this domain
ns1.google.com. 172800 IN  A  216.239.32.10

;; ADDITIONAL SECTION:                 ← Extra useful info

;; Query time: 12 msec                 ← How long DNS lookup took
;; SERVER: 8.8.8.8#53                  ← Which DNS server answered
;; WHEN: Mon Jun 23 2026

# Short output (just the answer)
dig +short google.com
# 142.250.80.46

# Specific record types
dig google.com A          # IPv4 address
dig google.com AAAA       # IPv6 address
dig google.com MX         # Mail servers
dig google.com TXT        # Text records (SPF, DKIM, verification)
dig google.com NS         # Name servers
dig google.com CNAME      # Canonical name (alias)
dig google.com SOA        # Start of Authority (zone info)

# Short + type
dig +short google.com MX
# 10 smtp.google.com.
# 20 alt1.aspmx.l.google.com.

# Query a SPECIFIC DNS server
dig @8.8.8.8 google.com       # Use Google's DNS
dig @1.1.1.1 google.com       # Use Cloudflare's DNS
dig @ns1.example.com google.com  # Query specific nameserver directly

# Reverse DNS lookup (PTR record)
dig -x 142.250.80.46
dig +short -x 142.250.80.46
# lax17s55-in-f14.1e100.net.

# Check if DNS is propagated to authoritative nameserver
dig +short google.com @ns1.google.com   # Query the authoritative NS directly

# Trace FULL resolution path (see every step: root → TLD → authoritative)
dig +trace google.com
# Shows: . (root) → com. (TLD) → google.com (authoritative)
# Invaluable for debugging DNS delegation issues

# Check TTL remaining (will it be cached?)
dig google.com | grep -A1 "ANSWER SECTION"
# google.com.  47  IN  A  142.250.80.46
#              ↑ 47 seconds remaining in cache

# Check ALL records at once
dig google.com ANY

# Check for DNSSEC
dig +dnssec google.com

# Batch lookup from file
dig -f domains.txt +short

# Time the lookup (benchmark DNS performance)
time dig google.com
```

**Real debugging scenarios with dig:**
```bash
# SCENARIO 1: Is my DNS change propagated?
# Check local DNS server:
dig +short api.myapp.com
# → 1.2.3.4 (old IP — not propagated yet)

# Check authoritative nameserver directly:
dig +short api.myapp.com @ns1.myapp.com
# → 5.6.7.8 (new IP — change is there, just propagating)

# SCENARIO 2: My email is going to spam — check SPF record
dig +short TXT myapp.com
# → "v=spf1 include:sendgrid.net ~all"
# If blank → no SPF → emails will likely be marked spam

# SCENARIO 3: Which server is actually authoritative for my domain?
dig NS myapp.com +short
# → ns1.myapp.com.
# → ns2.myapp.com.

# SCENARIO 4: Debug DNS propagation (check multiple DNS servers)
for dns in 8.8.8.8 1.1.1.1 9.9.9.9 208.67.222.222; do
  echo -n "DNS $dns: "
  dig +short api.myapp.com @$dns
done

# SCENARIO 5: Is my CNAME pointing correctly?
dig +short www.myapp.com CNAME
# → myapp.netlify.app.   ← CNAME target
dig +short myapp.netlify.app
# → 75.2.60.5             ← Final IP
```

---

### Interview-Ready Summary

> "ping tests basic ICMP connectivity and measures RTT — but 'ping fails' doesn't mean 'server is down' since many firewalls block ICMP. Use `curl` to test application-level connectivity. traceroute maps each router hop to a destination and shows latency at each — I use `traceroute -T -p 443` for TCP-based tracing which bypasses more firewalls. mtr combines ping and traceroute for continuous monitoring. dig is my DNS tool of choice over nslookup — `dig +short google.com` gives just the IP, `dig +trace` follows the full resolution chain from root to authoritative, `dig @8.8.8.8` queries a specific server to test propagation, and `dig +short TXT myapp.com` checks SPF records for email debugging. For verifying a DNS change before propagation completes, I query the authoritative nameserver directly with `dig api.myapp.com @ns1.myapp.com` — if the change is there, it's the TTL causing the delay."

## 5. Distributed Tracing

### The Problem — Debugging in Microservices

In a monolith, you have one log file. In microservices:

```
User request: POST /checkout

Your logs:
  API Gateway:     2026-06-23 14:23:45 POST /checkout 500 ERROR
  Order Service:   (no log — it never received the request?)
  Payment Service: 2026-06-23 14:23:45.123 Charge request received
                   2026-06-23 14:23:45.891 TIMEOUT connecting to fraud-service

Which service caused the error?
Where did the 500ms latency come from?
What happened between the gateway and order service?

Without tracing: You grep 5 different log files with different timestamps
                 and try to reconstruct what happened manually. 😱

With distributed tracing: You see the entire request as one unified timeline.
```

**Analogy:**
> Distributed tracing is like a parcel's tracking history — you see every warehouse it passed through, how long it spent at each stop, and exactly where it got delayed or lost. Without tracking, you just know it hasn't arrived.

---

### Core Concepts

**Trace:**
A trace represents ONE complete end-to-end request as it flows through your system. It's a collection of spans.

**Span:**
A span represents ONE operation within a trace — a single service call, DB query, or function call. Spans have:
```
Span:
  trace_id:   abc-123-xyz      ← Same across entire request (what links spans together)
  span_id:    def-456          ← Unique to this span
  parent_id:  null or span_id  ← Which span created this one
  operation:  "POST /checkout"
  service:    "api-gateway"
  start_time: 2026-06-23 14:23:45.000
  end_time:   2026-06-23 14:23:46.234
  duration:   1234ms
  status:     ERROR
  tags:       { http.method: "POST", http.status_code: 500, user.id: "42" }
  logs:       [ { time: ..., message: "Timeout waiting for payment service" } ]
```

**Context Propagation:**
The trace_id travels with the request through HTTP headers:
```
Service A → Service B:
  HTTP Request:
    traceparent: 00-abc123xyz-def456-01
    #                ↑           ↑     ↑
    #            trace_id    span_id  flags

  Service B creates a child span with parent_id = Service A's span_id
  Sends its own traceparent header to Service C
  ...and so on

W3C Trace Context (standard): traceparent, tracestate headers
Zipkin B3 (older):  X-B3-TraceId, X-B3-SpanId, X-B3-ParentSpanId headers
```

**A complete trace visualized:**
```
Trace: abc-123-xyz  (total: 1234ms)

api-gateway  [==========================================] 1234ms   POST /checkout
  order-svc    [======================]  678ms    POST /orders
    db-query     [====]  45ms            INSERT INTO orders
    payment-svc  [===========]  521ms   POST /charge
      fraud-svc  [XXXXXXXXXXXXXXXX]  500ms  TIMEOUT ← HERE IS THE PROBLEM!
  email-svc  [===]  89ms             POST /send-confirmation
```

### OpenTelemetry — The Standard

**OpenTelemetry (OTel)** is the vendor-neutral, open-source standard for distributed tracing, metrics, and logs. You instrument once with OTel, and can export to Jaeger, Zipkin, Datadog, Honeycomb, or any backend.

```
Your App (OTel SDK)
    │ OTLP (OpenTelemetry Protocol)
    ▼
OTel Collector (optional aggregator)
    │
    ├──► Jaeger (open-source visualizer)
    ├──► Zipkin (open-source visualizer)
    ├──► Datadog APM (commercial)
    └──► Honeycomb (commercial)
```

**Node.js OTel instrumentation:**
```javascript
// tracing.js — load this FIRST before any other imports
const { NodeSDK } = require('@opentelemetry/sdk-node');
const { getNodeAutoInstrumentations } = require('@opentelemetry/auto-instrumentations-node');
const { OTLPTraceExporter } = require('@opentelemetry/exporter-trace-otlp-http');
const { Resource } = require('@opentelemetry/resources');
const { SemanticResourceAttributes } = require('@opentelemetry/semantic-conventions');

const sdk = new NodeSDK({
  resource: new Resource({
    [SemanticResourceAttributes.SERVICE_NAME]: 'order-service',
    [SemanticResourceAttributes.SERVICE_VERSION]: '2.1.0',
    'deployment.environment': 'production',
  }),
  traceExporter: new OTLPTraceExporter({
    url: 'http://otel-collector:4318/v1/traces',
  }),
  instrumentations: [
    getNodeAutoInstrumentations()  // Auto-instruments: http, express, pg, redis, axios!
  ],
});

sdk.start();
// Now ALL http calls, DB queries, Redis ops are automatically traced!
// No manual instrumentation needed for common libraries.
```

**Manual instrumentation for custom operations:**
```javascript
const { trace, context, SpanStatusCode } = require('@opentelemetry/api');
const tracer = trace.getTracer('order-service');

async function processOrder(orderId, userId) {
  // Create a manual span for business logic
  return tracer.startActiveSpan('processOrder', async (span) => {
    try {
      // Add attributes to the span (searchable/filterable in Jaeger)
      span.setAttribute('order.id', orderId);
      span.setAttribute('user.id', userId);
      span.setAttribute('business.operation', 'checkout');

      const order = await db.orders.findById(orderId);
      span.setAttribute('order.total', order.total);
      span.setAttribute('order.items_count', order.items.length);

      // Nested span for sub-operation
      const payment = await tracer.startActiveSpan('chargePayment', async (paySpan) => {
        try {
          paySpan.setAttribute('payment.amount', order.total);
          const result = await paymentService.charge(order);
          paySpan.setAttribute('payment.transaction_id', result.transactionId);
          return result;
        } catch (err) {
          paySpan.recordException(err);
          paySpan.setStatus({ code: SpanStatusCode.ERROR, message: err.message });
          throw err;
        } finally {
          paySpan.end();
        }
      });

      span.setStatus({ code: SpanStatusCode.OK });
      return { order, payment };

    } catch (err) {
      span.recordException(err);                          // Record the exception
      span.setStatus({ code: SpanStatusCode.ERROR, message: err.message });
      throw err;
    } finally {
      span.end();   // ALWAYS end the span (even on error!)
    }
  });
}
```

### Sampling Strategies

Tracing EVERY request is expensive. Sampling sends only a fraction.

```
Head-based sampling (decide at start of request):
  Always sample: 1% of all requests (fixed rate)
  → Simple but might miss important rare errors

  Always sample errors: error requests always traced
  Always sample slow: requests > 1s always traced

Tail-based sampling (decide AFTER request completes):
  Look at the full trace → decide to keep or discard
  → Can always keep errors and slow requests
  → More complex — need to buffer traces before deciding
  → Jaeger and OTel Collector support this

Common strategy:
  1% random sampling (baseline)
  + 100% of errors (never miss an error trace)
  + 100% of requests > 2s (always trace slow requests)
```

```javascript
// OTel sampler configuration
const { ParentBasedSampler, TraceIdRatioBasedSampler } = require('@opentelemetry/sdk-trace-base');

const sdk = new NodeSDK({
  sampler: new ParentBasedSampler({
    // Sample 10% of new traces (when no parent span exists)
    root: new TraceIdRatioBasedSampler(0.1),
    // Always follow parent's sampling decision (propagate through services)
    remoteParentSampled: new AlwaysOnSampler(),
    remoteParentNotSampled: new AlwaysOffSampler(),
  }),
  // ...
});
```

### Interview-Ready Summary

> "Distributed tracing tracks a request as it flows through multiple microservices by assigning a unique trace_id and propagating it via HTTP headers (W3C traceparent). Each service creates spans — child operations with the same trace_id — recording timing, metadata, and errors. OpenTelemetry is the standard instrumentation layer — I instrument once and export to any backend. Auto-instrumentation handles common libraries (HTTP, Express, PostgreSQL, Redis) with zero code changes. Jaeger is the most popular open-source backend — the UI lets me search traces by service, operation, or tags like `error=true`, see the full timeline of a request across services, and immediately spot which service introduced latency or caused an error. For production, I use tail-based sampling: 1% random + 100% of errors + 100% of requests over 2s — so I never miss errors but don't pay for tracing every request."

## 6. Logging & Metrics

### Logging — Structured Logs

**Logging** records what happened. In production, logs need to be **structured** (JSON), **leveled**, **searchable**, and **centralized**.

#### Structured Logging

```javascript
// ❌ Unstructured log (hard to parse, search, or alert on)
console.log("User 42 placed order 7 for $99.00 at 2026-06-23 14:23:45");

// ✅ Structured log (JSON — every field is searchable)
logger.info("Order placed", {
  event:    "order.placed",
  userId:   42,
  orderId:  7,
  amount:   99.00,
  currency: "USD",
  duration: 234,         // ms to process
  traceId:  "abc-123",  // Link to distributed trace!
});

// Output:
{
  "level":    "info",
  "message":  "Order placed",
  "event":    "order.placed",
  "userId":   42,
  "orderId":  7,
  "amount":   99.00,
  "currency": "USD",
  "duration": 234,
  "traceId":  "abc-123",
  "timestamp": "2026-06-23T14:23:45.123Z",
  "service":  "order-service",
  "version":  "2.1.0"
}
```

**Log levels — when to use each:**
```
ERROR:  Something broke that needs immediate attention.
        An exception was thrown, a request failed, data was corrupted.
        → Alert / wake someone up
        logger.error("Payment charge failed", { orderId, error: err.message, stack: err.stack });

WARN:   Something unexpected but recoverable. May become an error if ignored.
        Retry succeeded, deprecated feature used, approaching rate limit.
        → Monitor and trend
        logger.warn("Payment retry succeeded", { orderId, attempt: 3 });

INFO:   Normal application events. Business events.
        User logged in, order placed, job completed.
        → Index and search when debugging
        logger.info("User logged in", { userId, ip, userAgent });

DEBUG:  Detailed internal state for developers.
        SQL queries, HTTP requests/responses, function args.
        → Enable in dev/staging only, disable in production (too verbose/costly)
        logger.debug("DB query executed", { sql, params, duration });

TRACE:  Even more granular than debug.
        Rarely used in practice. Framework internals.
```

#### Production-Grade Logger (Node.js with Pino)

```javascript
const pino = require('pino');

const logger = pino({
  level: process.env.LOG_LEVEL || 'info',   // 'debug' in dev, 'info' in prod
  formatters: {
    level: (label) => ({ level: label }),   // Use string not number for level
  },
  base: {
    service: 'order-service',
    version: process.env.APP_VERSION,
    env:     process.env.NODE_ENV,
  },
  redact: [
    'req.headers.authorization',   // Never log auth tokens
    'body.password',                // Never log passwords
    'body.creditCard',              // Never log card numbers
    '*.secret',                     // Any field named "secret"
  ],
  // In production: output to stdout (let log aggregator handle it)
  // In development: pretty-print
  transport: process.env.NODE_ENV !== 'production'
    ? { target: 'pino-pretty' }
    : undefined,
});

// Request logging middleware (Express)
app.use((req, res, next) => {
  const startTime = Date.now();

  // Bind request context to logger (trace_id, request_id)
  req.log = logger.child({
    requestId: req.headers['x-request-id'] || crypto.randomUUID(),
    traceId:   req.headers['traceparent'],
    method:    req.method,
    path:      req.path,
    ip:        req.ip,
  });

  res.on('finish', () => {
    const duration = Date.now() - startTime;
    const level = res.statusCode >= 500 ? 'error'
                : res.statusCode >= 400 ? 'warn'
                : 'info';

    req.log[level]('HTTP request', {
      statusCode: res.statusCode,
      duration,
      contentLength: res.get('Content-Length'),
    });
  });

  next();
});

// Business event logging
async function placeOrder(userId, items) {
  req.log.info("Processing order", { userId, itemCount: items.length });

  try {
    const order = await db.orders.create({ userId, items });
    req.log.info("Order created", {
      orderId: order.id,
      total:   order.total,
    });
    return order;
  } catch (err) {
    req.log.error("Order creation failed", {
      userId,
      error:   err.message,
      code:    err.code,
      stack:   err.stack,
    });
    throw err;
  }
}
```

### Metrics — The Numbers That Matter

**Metrics** are numeric measurements over time. Unlike logs (events), metrics are aggregatable — you can compute averages, percentiles, rates.

#### The Four Golden Signals (Google SRE)

```
1. LATENCY — How long requests take
   → P50, P95, P99 response times
   → "99% of requests complete in < 200ms"

2. TRAFFIC — How much demand there is
   → Requests per second
   → "We handle 10,000 req/s at peak"

3. ERRORS — How often things fail
   → Error rate (% of 5xx responses)
   → "Error rate < 0.1%"

4. SATURATION — How full your service is
   → CPU %, memory %, DB connection pool usage
   → Queue depth
   → "DB connection pool 80% utilized"
```

#### Prometheus — Metrics Collection

**Prometheus** scrapes metrics from your services and stores them as time series.

**Metric types:**
```
Counter:   Always increasing. Track total events.
           http_requests_total, errors_total, bytes_sent_total
           Rate of change = requests per second

Gauge:     Can go up or down. Current state.
           active_connections, memory_bytes, queue_depth, cpu_usage

Histogram: Samples observations in buckets. Latency distribution.
           http_request_duration_seconds_bucket{le="0.1"} 9800
           http_request_duration_seconds_bucket{le="0.5"} 9950
           http_request_duration_seconds_bucket{le="1.0"} 9990
           http_request_duration_seconds_count 10000
           http_request_duration_seconds_sum 485.3
           → P99 = 99th percentile latency

Summary:   Like histogram but pre-calculated quantiles.
           Less flexible than histogram (can't aggregate across instances).
```

**Instrumenting Node.js with Prometheus:**
```javascript
const promClient = require('prom-client');

// Auto-collect Node.js default metrics (event loop lag, heap, GC, etc.)
promClient.collectDefaultMetrics({ prefix: 'nodejs_' });

// Custom metrics
const httpRequestDuration = new promClient.Histogram({
  name:    'http_request_duration_seconds',
  help:    'HTTP request duration in seconds',
  labelNames: ['method', 'route', 'status_code'],
  buckets: [0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5],
});

const httpRequestTotal = new promClient.Counter({
  name:       'http_requests_total',
  help:       'Total HTTP requests',
  labelNames: ['method', 'route', 'status_code'],
});

const activeConnections = new promClient.Gauge({
  name:    'active_connections',
  help:    'Number of active connections',
});

const dbPoolUsage = new promClient.Gauge({
  name:    'db_pool_connections_used',
  help:    'DB connection pool: connections in use',
});

// Middleware to track metrics
app.use((req, res, next) => {
  const end = httpRequestDuration.startTimer();
  activeConnections.inc();

  res.on('finish', () => {
    const route = req.route?.path || req.path;
    const labels = {
      method:      req.method,
      route:       route,
      status_code: res.statusCode,
    };

    end(labels);                            // Record duration
    httpRequestTotal.inc(labels);           // Increment counter
    activeConnections.dec();
  });

  next();
});

// Expose metrics endpoint for Prometheus to scrape
app.get('/metrics', async (req, res) => {
  res.set('Content-Type', promClient.register.contentType);
  res.end(await promClient.register.metrics());
});
```

**PromQL — Prometheus Query Language:**
```promql
# Request rate (requests per second, last 5 min)
rate(http_requests_total[5m])

# Error rate (percentage of 5xx responses)
sum(rate(http_requests_total{status_code=~"5.."}[5m]))
/
sum(rate(http_requests_total[5m]))
* 100

# P99 latency across all routes
histogram_quantile(0.99,
  sum(rate(http_request_duration_seconds_bucket[5m])) by (le)
)

# P99 latency per route
histogram_quantile(0.99,
  sum(rate(http_request_duration_seconds_bucket[5m])) by (le, route)
)

# Active DB connections per service
db_pool_connections_used{service="order-service"}

# Alert: error rate > 1% for 5 minutes
ALERT HighErrorRate
  IF sum(rate(http_requests_total{status_code=~"5.."}[5m]))
     / sum(rate(http_requests_total[5m])) > 0.01
  FOR 5m
  LABELS { severity = "critical" }
  ANNOTATIONS { summary = "Error rate above 1%" }
```

### Alerting

Metrics without alerting are useless. You need to know when something breaks.

```
Alert tiers:
  P1 (Critical — wake someone up NOW):
    Error rate > 1% for 5 minutes
    P99 latency > 5 seconds for 5 minutes
    Service is completely down (0 req/s but traffic expected)
    DB connection pool 100% used

  P2 (Warning — investigate soon):
    Error rate > 0.5%
    P99 latency > 2 seconds
    DB connection pool > 80%
    Disk usage > 80%

  P3 (Info — investigate next business day):
    Unusual traffic patterns
    Memory growing steadily (potential leak)
    Increased 4xx rate (client errors — maybe API changes breaking clients)

Alert routing:
  P1 → PagerDuty → on-call engineer phone call + SMS
  P2 → Slack #alerts channel
  P3 → Email digest

Avoid alert fatigue:
  Alerts must be actionable (someone can actually fix it)
  Alert on symptoms (high latency, errors) not causes (high CPU)
  Use "for 5m" — don't alert on brief spikes
  Tune thresholds — if alert fires constantly, it gets ignored
```

### Interview-Ready Summary

> "Structured logging in JSON is non-negotiable in production — every field becomes searchable in Elasticsearch/Kibana. I use Pino or Winston with request context (requestId, traceId, userId) bound to every log from a request. Log levels: ERROR for breaking things needing immediate attention, WARN for recoverable issues, INFO for business events, DEBUG only in dev. Never log sensitive data — use redaction for passwords, tokens, card numbers. The four golden signals (Google SRE) are what every service should monitor: Latency (P50/P95/P99), Traffic (req/s), Errors (5xx rate), and Saturation (CPU, memory, connection pool). I instrument with Prometheus: Histograms for latency (gives P99 with `histogram_quantile`), Counters for totals (use `rate()` for per-second), Gauges for current state. Grafana visualizes these with dashboards. Alerting is the final piece — P1 alerts (> 1% error rate, > 5s P99) go to PagerDuty. I link logs to traces via traceId in log fields — if I see an error in Jaeger, I can grep all logs for that traceId and see the full context."
