## 1. REST Principles

### What is REST?

**REST** (Representational State Transfer) is an **architectural style** for designing networked APIs. It's not a protocol or a standard — it's a set of constraints and principles that, when followed, make APIs predictable, scalable, and easy to use.

REST was defined by Roy Fielding in his 2000 PhD dissertation. When people say "REST API" or "RESTful API", they mean an HTTP API that follows these principles.

**Analogy:**
> Think of a library. Every book has a unique location (shelf + number). You can borrow (GET), return (DELETE), add new (POST), or replace a book (PUT). The librarian doesn't need to remember who you are between visits — you show your card each time. That's REST: resources have addresses, operations are standard, and the server is stateless.

---

### The 6 REST Constraints

**1. Statelessness (Most Important)**

The server stores **no session state** between requests. Every request must contain all information needed to process it.

```
❌ Stateful (BAD):
  Request 1: POST /login  → Server stores "User Alice is logged in"
  Request 2: GET /profile → Server remembers Alice from session

✅ Stateless (GOOD - REST way):
  Request 1: POST /login  → Server returns JWT token to client
  Request 2: GET /profile
             Authorization: Bearer eyJhbGc...  ← client sends token every time
             Server reads the token, identifies Alice, responds
```

**Why statelessness matters:**
```
Scalability: Any server in a cluster can handle any request
             (no need for "sticky sessions")
Reliability: If a server crashes, requests just go to another server
Simplicity:  Each request is self-contained and easy to debug
```

**2. Client-Server Separation**
- Client handles UI/UX. Server handles data and business logic.
- They evolve independently. Your React frontend and Go backend don't need to be deployed together.

**3. Uniform Interface**
- Consistent, predictable API design (covered in resource design section below).
- Resources identified by URLs, manipulated through representations (JSON/XML).

**4. Layered System**
- Client doesn't know if it's talking to the actual server, a load balancer, a cache, or a CDN.
- Each layer only knows about the layer directly next to it.

```
Client → CDN → Load Balancer → API Gateway → Your Server → Database
          ↑          ↑               ↑
     Client doesn't know these exist — just sees the response
```

**5. Cacheable**
- Responses must declare whether they're cacheable.
- Proper caching reduces server load massively.

```
Cache-Control: max-age=3600        → Cache for 1 hour
Cache-Control: no-store            → Never cache (sensitive data)
ETag: "abc123"                     → Version identifier for conditional requests
```

**6. Code on Demand (Optional)**
- Server can send executable code to the client (e.g., JavaScript).
- Rarely discussed in REST APIs today.

### Resource Design

The most visible part of REST — how you design your URLs and what they represent.

**Core rule: URLs identify RESOURCES (nouns), not actions (verbs).**

```
❌ NON-RESTful (RPC style):
  POST /getUser
  POST /createUser
  POST /deleteUser
  POST /getUserOrders

✅ RESTful (Resource style):
  GET    /users         → collection of users
  GET    /users/42      → specific user
  POST   /users         → create a user
  PUT    /users/42      → replace user 42
  PATCH  /users/42      → update fields of user 42
  DELETE /users/42      → delete user 42
```

**Nested resources (relationships):**
```
GET    /users/42/orders          → all orders of user 42
GET    /users/42/orders/7        → order 7 of user 42
POST   /users/42/orders          → create order for user 42
DELETE /users/42/orders/7        → cancel order 7 of user 42
```

**Rule of thumb for nesting:**
> Only nest when the child resource **can't exist independently** or when you need it **in context of the parent**. Max 2 levels deep. Beyond that, use query parameters.

```
✅ /users/42/orders             → makes sense, orders belong to user
✅ /orders/7/items              → makes sense, items belong to order
❌ /users/42/orders/7/items/3  → too deep, use /order-items/3 instead
```

**Naming conventions:**
```
✅ Plural nouns:          /users, /orders, /products
✅ Lowercase + hyphens:   /order-items, /shipping-addresses
✅ No verbs in URL:        /users (not /getUsers)
✅ IDs in path:            /users/42 (not /users?id=42)
✅ Filters in query param: /users?status=active&role=admin
✅ Pagination:             /users?page=2&limit=20
✅ Sorting:                /users?sort=created_at&order=desc
✅ Search:                 /users?q=alice
```

**Special actions that don't fit CRUD:**
```
# When you must use a verb, treat it as a sub-resource action
POST /orders/42/cancel          → cancel an order
POST /users/42/activate         → activate account
POST /payments/42/refund        → refund a payment
POST /auth/refresh              → refresh a token (auth is a special resource)
```

---

### Idempotency

**Idempotent** = calling the operation **multiple times** has the **same effect** as calling it once.

This is critical for reliability — network failures cause retries, and retries must be safe.

```
GET    /users/42    → Idempotent ✅  (fetching doesn't change state)
DELETE /users/42    → Idempotent ✅  (deleting twice = same result, user is gone)
PUT    /users/42    → Idempotent ✅  (replacing with same data = same result)
POST   /users       → NOT Idempotent ❌ (each call creates a new user!)
PATCH  /users/42    → Depends on implementation
```

**Why idempotency matters for POST:**
```
Client sends: POST /orders (create order)
Network error! Did the server receive it?
Client retries: POST /orders
Result: TWO orders created! 😱

Solution: Idempotency Key
Client sends: POST /orders
              Idempotency-Key: client-uuid-abc123

Server checks: "Have I seen key abc123 before?"
  → No: Process the order, store the key + result
  → Yes: Return the stored result without processing again

Client retries: POST /orders
               Idempotency-Key: client-uuid-abc123
Server: "I've seen this key. Here's the original response."
Result: Only ONE order created ✅ (Stripe and many payment APIs do this)
```

---

### REST API Versioning

APIs evolve. Old clients break when you change the API. Versioning prevents this.

**3 common strategies:**

```
1. URL Versioning (most common, most explicit):
   GET /v1/users
   GET /v2/users

2. Header Versioning:
   GET /users
   Accept: application/vnd.myapi.v2+json

3. Query Parameter:
   GET /users?version=2
```

**Best practice:**
> Use **URL versioning** (`/v1/`, `/v2/`) — it's explicit, easy to test in a browser, easy to route at the load balancer, and visible in logs.

### REST Response Design

**Consistent response envelope:**
```json
// Success (single resource)
{
  "data": {
    "id": 42,
    "name": "Alice",
    "email": "alice@example.com"
  }
}

// Success (collection with pagination)
{
  "data": [
    { "id": 1, "name": "Alice" },
    { "id": 2, "name": "Bob" }
  ],
  "meta": {
    "total": 100,
    "page": 1,
    "per_page": 20,
    "total_pages": 5
  },
  "links": {
    "self":  "/users?page=1",
    "next":  "/users?page=2",
    "prev":  null,
    "first": "/users?page=1",
    "last":  "/users?page=5"
  }
}

// Error response (consistent error format)
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Request validation failed",
    "details": [
      { "field": "email", "message": "Invalid email format" },
      { "field": "age",   "message": "Must be at least 18" }
    ]
  }
}
```

### REST vs Other Styles

| Feature | REST | GraphQL | gRPC |
|---------|------|---------|------|
| Protocol | HTTP | HTTP | HTTP/2 |
| Format | JSON/XML | JSON | Protobuf (binary) |
| Schema | Optional | Required | Required (.proto) |
| Fetching | Fixed endpoints | Flexible queries | Defined methods |
| Over-fetching | Common | Eliminated | N/A |
| Browser support | ✅ Native | ✅ Native | ❌ Needs proxy |
| Learning curve | Low | Medium | High |
| Best for | Public APIs | Complex UIs | Microservices |

---

### Interview-Ready Summary

> "REST is an architectural style for APIs built on HTTP. Its 6 constraints are: statelessness (every request is self-contained — no server-side sessions), client-server separation, uniform interface, layered system, cacheability, and optional code on demand. The most important is statelessness — it enables horizontal scaling because any server can handle any request. Resource design means using nouns in URLs (`/users/42`), not verbs, with HTTP methods defining the action. Idempotency means an operation can be repeated safely — GET, PUT, DELETE are idempotent; POST is not, which is why payment APIs use idempotency keys. I always version APIs with `/v1/` in the URL and return consistent JSON envelopes with `data`, `meta`, and `error` fields."

## 2. GraphQL

### What is GraphQL?

**GraphQL** is a **query language for APIs** and a runtime for executing those queries. Developed by Facebook (2012, open-sourced 2015).

Instead of fixed endpoints that return fixed data, GraphQL exposes a **single endpoint** where the client specifies **exactly what data it needs**.

**The core problem it solves:**

```
REST Problem 1: OVER-FETCHING
  Client needs: user's name and email only
  REST returns: { id, name, email, phone, address, created_at, updated_at, ... }
  → Client downloads data it doesn't need

REST Problem 2: UNDER-FETCHING (N+1 problem)
  Client needs: user + their orders + each order's products
  REST requires:
    GET /users/42          → 1 request
    GET /users/42/orders   → 1 request  (returns 10 orders)
    GET /orders/1/products → 1 request  ↑
    GET /orders/2/products → 1 request  |
    ... × 10               → 10 requests (one per order!)
  Total: 12 requests!

GraphQL:
  1 request → get exactly what you need
```

---

### GraphQL Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│                    GraphQL Layer                                 │
│                                                                  │
│   Single Endpoint: POST /graphql                                │
│                                                                  │
│   Schema (Type System) ──────────────────────────────────────  │
│   Resolvers (Business Logic) ────────────────────────────────  │
│                                                                  │
│   Data Sources:                                                  │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────────┐  │
│   │ Database │  │ REST API │  │  Cache   │  │ Microservice │  │
│   └──────────┘  └──────────┘  └──────────┘  └──────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

### The Schema — The Contract

The **schema** defines every type and operation available in your API. It's the contract between frontend and backend.

```graphql
# Scalar types (built-in)
# Int, Float, String, Boolean, ID

# Object types
type User {
  id: ID!              # ! means non-nullable (required)
  name: String!
  email: String!
  age: Int             # nullable (optional)
  orders: [Order!]!    # array of orders, array itself non-null, elements non-null
  createdAt: String!
}

type Order {
  id: ID!
  total: Float!
  status: OrderStatus!
  items: [OrderItem!]!
  user: User!          # back-reference to user
}

type OrderItem {
  id: ID!
  product: Product!
  quantity: Int!
  price: Float!
}

type Product {
  id: ID!
  name: String!
  price: Float!
  category: String!
}

# Enum type
enum OrderStatus {
  PENDING
  PROCESSING
  SHIPPED
  DELIVERED
  CANCELLED
}

# Input type (used for mutations - like a request body DTO)
input CreateUserInput {
  name: String!
  email: String!
  password: String!
}

input UpdateUserInput {
  name: String
  email: String
}
```

### Queries — Reading Data

A **query** is for fetching data (like GET in REST).

**Basic query:**
```graphql
# Ask for exactly what you need
query {
  user(id: "42") {
    name
    email
  }
}
```

**Response:**
```json
{
  "data": {
    "user": {
      "name": "Alice",
      "email": "alice@example.com"
    }
  }
}
```

**Nested query (replaces 12 REST requests with 1):**
```graphql
query GetUserWithOrders {
  user(id: "42") {
    name
    email
    orders {
      id
      total
      status
      items {
        quantity
        price
        product {
          name
          category
        }
      }
    }
  }
}
```

**Query with variables (always use variables, never string interpolation):**
```graphql
# Query definition (sent once)
query GetUser($userId: ID!, $orderStatus: OrderStatus) {
  user(id: $userId) {
    name
    orders(status: $orderStatus) {
      id
      total
    }
  }
}
```

```json
// Variables (sent separately — safe, no injection risk)
{
  "userId": "42",
  "orderStatus": "DELIVERED"
}
```

**Aliases (fetch same field twice with different args):**
```graphql
query {
  recentOrders: orders(status: DELIVERED, limit: 5) {
    id
    total
  }
  pendingOrders: orders(status: PENDING) {
    id
    total
  }
}
```

**Fragments (reuse field selections):**
```graphql
fragment UserFields on User {
  id
  name
  email
}

query {
  alice: user(id: "1") {
    ...UserFields          # spread the fragment
    orders { id }
  }
  bob: user(id: "2") {
    ...UserFields
  }
}
```

### Mutations — Writing Data

A **mutation** is for creating, updating, or deleting data (like POST/PUT/PATCH/DELETE in REST).

```graphql
# Define mutations in schema
type Mutation {
  createUser(input: CreateUserInput!): User!
  updateUser(id: ID!, input: UpdateUserInput!): User!
  deleteUser(id: ID!): Boolean!
  createOrder(userId: ID!, items: [OrderItemInput!]!): Order!
  cancelOrder(orderId: ID!): Order!
}
```

**Calling a mutation:**
```graphql
mutation CreateUser($input: CreateUserInput!) {
  createUser(input: $input) {
    id
    name
    email
  }
}
```

```json
// Variables
{
  "input": {
    "name": "Alice",
    "email": "alice@example.com",
    "password": "secret123"
  }
}
```

**Response:**
```json
{
  "data": {
    "createUser": {
      "id": "42",
      "name": "Alice",
      "email": "alice@example.com"
    }
  }
}
```

### Subscriptions — Real-Time Data

A **subscription** is for real-time updates (uses WebSocket under the hood).

```graphql
# Schema definition
type Subscription {
  orderStatusChanged(orderId: ID!): Order!
  newMessage(roomId: ID!): Message!
  userTyping(roomId: ID!): User!
}
```

**Client subscribes:**
```graphql
subscription WatchOrder($orderId: ID!) {
  orderStatusChanged(orderId: $orderId) {
    id
    status
    updatedAt
  }
}
```

```
Client connects via WebSocket to /graphql
Server pushes update whenever order status changes:
  → { "data": { "orderStatusChanged": { "id": "7", "status": "SHIPPED" } } }
  → { "data": { "orderStatusChanged": { "id": "7", "status": "DELIVERED" } } }
```

### Resolvers — The Business Logic

A **resolver** is a function that fetches the data for a field. Every field in the schema has a resolver.

```javascript
// Node.js with Apollo Server
const resolvers = {
  Query: {
    // Resolver for: query { user(id: "42") { ... } }
    user: async (parent, args, context) => {
      const { id } = args;
      return await db.users.findById(id);
    },

    orders: async (parent, { status, limit }, context) => {
      return await db.orders.findAll({ where: { status }, limit });
    }
  },

  Mutation: {
    createUser: async (parent, { input }, context) => {
      // context usually has auth user, db connection, etc.
      if (!context.user) throw new Error('Not authenticated');
      const hashed = await bcrypt.hash(input.password, 10);
      return await db.users.create({ ...input, password: hashed });
    }
  },

  // Field resolvers (run when the field is requested)
  User: {
    // This runs when someone requests user.orders
    orders: async (parent, args, context) => {
      // parent = the user object already fetched
      return await db.orders.findAll({ where: { userId: parent.id } });
    }
  },

  Subscription: {
    orderStatusChanged: {
      subscribe: (parent, { orderId }, { pubsub }) =>
        pubsub.asyncIterator(`ORDER_${orderId}`),
    }
  }
};
```

### The N+1 Problem in GraphQL

GraphQL's nested resolvers create a classic performance issue:

```
Query:
{
  users {           ← 1 DB query: SELECT * FROM users (returns 100 users)
    orders {        ← 100 DB queries! SELECT * FROM orders WHERE userId = ?
      id
    }
  }
}
Total: 101 queries! 😱
```

**Solution: DataLoader (batching + caching)**

```javascript
const DataLoader = require('dataloader');

// Instead of 100 queries, DataLoader batches them into 1
const orderLoader = new DataLoader(async (userIds) => {
  // Called ONCE with all userIds: [1, 2, 3, ..., 100]
  const orders = await db.orders.findAll({
    where: { userId: userIds }           // SELECT * FROM orders WHERE userId IN (1,2,...,100)
  });

  // Group by userId for DataLoader to map back
  return userIds.map(id => orders.filter(o => o.userId === id));
});

// Resolver
const resolvers = {
  User: {
    orders: (parent, args, { loaders }) => {
      return loaders.orderLoader.load(parent.id);  // Batched automatically!
    }
  }
};

// Result: 2 DB queries instead of 101 ✅
```

### Error Handling in GraphQL

GraphQL always returns **200 OK** even for errors (unlike REST). Errors are in the response body:

```json
// Partial success — some data, some errors
{
  "data": {
    "user": {
      "name": "Alice",
      "orders": null        ← this field errored
    }
  },
  "errors": [
    {
      "message": "Failed to fetch orders",
      "path": ["user", "orders"],
      "extensions": {
        "code": "INTERNAL_SERVER_ERROR"
      }
    }
  ]
}
```

**This is powerful** — you get partial data even when some fields fail.

---

### Interview-Ready Summary

> "GraphQL is a query language for APIs that solves REST's over-fetching and under-fetching problems. Instead of multiple fixed endpoints, you have a single `/graphql` endpoint. The client specifies exactly what fields it needs. The schema is the contract — it defines types, queries (read), mutations (write), and subscriptions (real-time via WebSocket). Resolvers are functions that fetch data for each field. The main gotcha is the N+1 problem — nested resolvers fire a DB query per parent record. The solution is DataLoader, which batches and caches these queries. GraphQL always returns 200 OK — errors come in the response body alongside partial data. I'd choose GraphQL when the frontend has complex, varied data needs; REST for simple public APIs."

## 3. Message Queues

### What is a Message Queue?

A **message queue** is a form of **asynchronous communication** between services. Instead of Service A calling Service B directly and waiting for a response, A puts a message in a queue and B processes it whenever it's ready.

**Analogy:**
> Think of a restaurant kitchen. Waiters (producers) don't cook the food themselves — they put orders on a ticket rail (queue). Cooks (consumers) pick up tickets and cook when they're ready. The waiter doesn't wait at the kitchen — they go serve other customers. The kitchen processes orders at its own pace.

**Why message queues exist:**

```
Without Message Queue (Synchronous):
  Order Service → Payment Service (waits 2s) → Inventory Service (waits 1s) → Email Service (waits 1s)
  Total: 4 seconds blocked!
  If Email Service is down → Order fails! 😱

With Message Queue (Asynchronous):
  Order Service → publishes "OrderCreated" event → Queue
  → Payment Service consumes + processes independently
  → Inventory Service consumes + processes independently
  → Email Service consumes + processes independently
  Order Service returns immediately! ✅
  If Email Service is down → message waits in queue, processed when it comes back ✅
```

---

### Core Concepts

```
Producer   → Service that creates and sends messages
Consumer   → Service that receives and processes messages
Queue      → Buffer that holds messages until consumed
Message    → The data being sent (JSON, bytes, etc.)
Broker     → The server that manages queues (RabbitMQ, Kafka)
Exchange   → (RabbitMQ) Routes messages to queues based on rules
Topic      → (Kafka) Named channel/category for messages
Partition  → (Kafka) Subdivision of a topic for parallelism
Offset     → (Kafka) Position of a message in a partition
```

---

### AMQP & RabbitMQ

**AMQP** (Advanced Message Queuing Protocol) is a standard protocol for message queuing. **RabbitMQ** is the most popular broker that implements AMQP.

#### RabbitMQ Architecture

```
Producer
   │
   │ publish(exchange, routingKey, message)
   ▼
┌──────────────────────────────────────────────────────┐
│                    RabbitMQ Broker                    │
│                                                       │
│   Exchange                                            │
│   (routes messages)                                   │
│       │                                               │
│       ├── Binding (routingKey: "order.created") ──► Queue A │
│       ├── Binding (routingKey: "order.*") ────────► Queue B │
│       └── Binding (routingKey: "#") ──────────────► Queue C │
│                                                       │
└──────────────────────────────────────────────────────┘
                    │           │           │
                    ▼           ▼           ▼
              Consumer 1   Consumer 2   Consumer 3
```

#### Exchange Types (How messages are routed)

**1. Direct Exchange**
Message goes to queues whose binding key **exactly matches** the routing key.
```
Producer → Exchange (routingKey: "order.created")
  → Queue "orders" (bound with key "order.created") ✅ receives it
  → Queue "payments" (bound with key "payment.processed") ❌ doesn't
```

**2. Fanout Exchange**
Message goes to **all** bound queues. Routing key is ignored.
```
Producer → Fanout Exchange
  → Queue "email-service"      ✅ all get a copy
  → Queue "sms-service"        ✅
  → Queue "push-notification"  ✅
```

**3. Topic Exchange**
Routing key is a pattern with wildcards.
```
* = one word
# = zero or more words

Producer → Exchange (routingKey: "order.us.created")
  → Queue bound with "order.*.created"  ✅ matches
  → Queue bound with "order.#"          ✅ matches
  → Queue bound with "payment.#"        ❌ doesn't match
```

**4. Headers Exchange**
Routes based on message header attributes, not routing key.

---

#### RabbitMQ Key Features

**Acknowledgments (ACK/NACK):**
```
Consumer receives message
  → Process successfully → ACK (acknowledge)
     RabbitMQ removes message from queue ✅

  → Processing fails → NACK (negative acknowledge)
     RabbitMQ can: requeue it, discard it, or send to Dead Letter Queue
```

**Dead Letter Queue (DLQ):**
```
Messages that fail (NACK'd, expired, or queue full) go to a Dead Letter Queue.
→ You can inspect failures, fix bugs, and re-process.
→ Critical for production systems — never lose a message!

Normal Queue ──(failed)──► Dead Letter Exchange ──► Dead Letter Queue
                                                          │
                                              Ops team inspects & replays
```

**Message Durability:**
```
# For messages to survive broker restart:
1. Mark queue as durable (persisted to disk)
2. Mark messages as persistent (delivery_mode=2)
3. Publisher confirms (wait for broker to confirm it persisted)

Without durability: messages live in memory → crash = lost messages!
```

**Node.js RabbitMQ Example:**
```javascript
const amqp = require('amqplib');

// PRODUCER: publish an order event
async function publishOrder(order) {
  const conn = await amqp.connect('amqp://localhost');
  const channel = await conn.createChannel();

  const exchange = 'orders';
  await channel.assertExchange(exchange, 'topic', { durable: true });

  const routingKey = 'order.created';
  const message = JSON.stringify(order);

  channel.publish(exchange, routingKey, Buffer.from(message), {
    persistent: true       // message survives broker restart
  });

  console.log(`Published: ${routingKey}`, order);
  await conn.close();
}

// CONSUMER: process order events
async function startConsumer() {
  const conn = await amqp.connect('amqp://localhost');
  const channel = await conn.createChannel();

  await channel.assertExchange('orders', 'topic', { durable: true });
  const q = await channel.assertQueue('email-service', { durable: true });
  await channel.bindQueue(q.queue, 'orders', 'order.*');

  channel.prefetch(1);  // Process one message at a time (fair dispatch)

  channel.consume(q.queue, async (msg) => {
    const order = JSON.parse(msg.content.toString());
    try {
      await sendOrderConfirmationEmail(order);
      channel.ack(msg);           // ✅ success — remove from queue
    } catch (err) {
      channel.nack(msg, false, true);  // ❌ failure — requeue
    }
  });
}
```

### Apache Kafka

**Kafka** is a distributed **event streaming platform** built for **high throughput, fault tolerance, and replay**. It's fundamentally different from RabbitMQ.

**Core difference:**
```
RabbitMQ: Messages are consumed and deleted. Broker routes to consumers.
Kafka:    Messages are persisted on disk for configurable retention.
          Consumers pull at their own pace. Old messages can be replayed.
```

#### Kafka Architecture

```
┌──────────────────────────────────────────────────────────────────────┐
│                         Kafka Cluster                                 │
│                                                                       │
│  Topic: "orders"                                                      │
│  ┌─────────────────────────────────────────────────────────────────┐ │
│  │ Partition 0: [msg0] [msg1] [msg2] [msg3] [msg4] [msg5]         │ │
│  │              offset=0  1     2     3     4     5               │ │
│  ├─────────────────────────────────────────────────────────────────┤ │
│  │ Partition 1: [msg0] [msg1] [msg2] [msg3]                       │ │
│  ├─────────────────────────────────────────────────────────────────┤ │
│  │ Partition 2: [msg0] [msg1] [msg2] [msg3] [msg4]               │ │
│  └─────────────────────────────────────────────────────────────────┘ │
│                                                                       │
└──────────────────────────────────────────────────────────────────────┘
         ↑ Producers write                     ↑ Consumers read
         (appended to end)                     (from their offset)
```

#### Key Kafka Concepts

**Topics and Partitions:**
```
Topic = a named category (like "orders", "payments", "user-events")
Partition = a topic is split into N partitions

Why partition?
1. Parallelism — multiple consumers read different partitions simultaneously
2. Ordering — messages within ONE partition are ordered
              Messages across partitions are NOT ordered
3. Scalability — more partitions = more throughput

Partition key = determines which partition a message goes to
  Same key → always same partition → ordering guaranteed for that key

Example:
  publishMessage("orders", key=userId, value=orderEvent)
  → All orders for user 42 go to same partition → processed in order
  → Orders for different users go to different partitions → parallelism
```

**Consumer Groups:**
```
Topic "orders" (3 partitions)

Consumer Group "email-service":
  Consumer A reads Partition 0
  Consumer B reads Partition 1
  Consumer C reads Partition 2
  → 3 consumers process in parallel!

Consumer Group "inventory-service":
  Consumer X reads ALL 3 partitions (if only 1 consumer in group)
  → Completely independent from email-service!
  → Both groups get ALL messages (unlike RabbitMQ where one consumer gets it)
```

**Offsets (Kafka's superpower):**
```
Offset = position of a message in a partition (starts at 0)

Consumer tracks "I've processed up to offset 42"
→ Consumer crashes and restarts → resumes from offset 43
→ Consumer can REPLAY from offset 0 → reprocess all historical events!
→ New service deployed → can read all past events to build its state!
```

**Kafka Node.js Example:**
```javascript
const { Kafka } = require('kafkajs');

const kafka = new Kafka({
  clientId: 'my-app',
  brokers: ['kafka:9092']
});

// PRODUCER
async function publishEvent(topic, key, value) {
  const producer = kafka.producer();
  await producer.connect();

  await producer.send({
    topic,
    messages: [{
      key: String(key),        // partition key
      value: JSON.stringify(value),
      headers: { source: 'order-service' }
    }]
  });

  await producer.disconnect();
}

// CONSUMER
async function startConsumer() {
  const consumer = kafka.consumer({ groupId: 'email-service' });
  await consumer.connect();
  await consumer.subscribe({ topic: 'orders', fromBeginning: false });

  await consumer.run({
    eachMessage: async ({ topic, partition, message }) => {
      const order = JSON.parse(message.value.toString());
      const offset = message.offset;

      console.log(`Processing order from partition ${partition}, offset ${offset}`);

      try {
        await sendEmail(order);
        // Kafka commits offset automatically after eachMessage returns
      } catch (err) {
        console.error('Failed to process:', err);
        // Message won't be committed → will be retried
        throw err;
      }
    }
  });
}
```

### RabbitMQ vs Kafka

| Feature | RabbitMQ | Kafka |
|---------|----------|-------|
| Primary use | Task queues, routing, RPC | Event streaming, log aggregation |
| Message lifetime | Deleted after consumption | Retained (days/forever) |
| Ordering | Per-queue | Per-partition |

| Replay | ❌ No | ✅ Yes (by offset) |
| Routing | Flexible (exchanges) | Simple (topic/partition) |
| Consumer model | Push (broker pushes to consumer) | Pull (consumer polls) |
| Protocol | AMQP | Custom (Kafka protocol) |
| Complexity | Medium | High |
| Best for | Background jobs, email sending, RPC | Event sourcing, audit logs, analytics pipelines |

### Pub/Sub Pattern

**Publish-Subscribe** is a messaging pattern where producers (publishers) send messages to a **topic/channel**, and multiple subscribers independently receive copies.

```
Publisher: "OrderCreated" event
     │
     ▼
  [Topic / Channel]
     │
     ├──────────────► Email Service (sends confirmation)
     ├──────────────► Inventory Service (reserves stock)
     ├──────────────► Analytics Service (records event)
     └──────────────► Notification Service (push notification)

All 4 services get the SAME event independently.
Publisher doesn't know or care who's subscribed.
```

**Pub/Sub vs Queue:**
```
Queue (Point-to-Point):
  Message → Queue → ONE consumer gets it (work queue, task distribution)
  Used for: background jobs where only one worker should process each task

Pub/Sub:
  Message → Topic → ALL subscribers get a copy
  Used for: event broadcasting where multiple services need to react

RabbitMQ Fanout Exchange = Pub/Sub
Kafka Consumer Groups = each GROUP gets all messages (Pub/Sub across groups)
                        within a group, messages are distributed (queue-like)
```

---

### When to Use What

```
Use RabbitMQ when:
  ✅ Task distribution (send 1 email to exactly 1 worker)
  ✅ Complex routing rules (route by type, priority, pattern)
  ✅ Request-reply patterns (RPC over messaging)
  ✅ Message TTL and priority queues
  ✅ Simpler setup / smaller scale

Use Kafka when:
  ✅ High-throughput event streaming (millions of events/sec)
  ✅ Event replay / event sourcing
  ✅ Multiple independent services consuming same events
  ✅ Audit logs / activity feeds
  ✅ Stream processing (with Kafka Streams / Flink)
  ✅ Data pipelines to data warehouses

Use Redis Pub/Sub when:
  ✅ Simple real-time pub/sub (WebSocket broadcasting)
  ✅ Already using Redis in your stack
  ⚠️  No persistence — messages lost if no subscriber or Redis restarts
```

---

### Interview-Ready Summary

> "Message queues enable asynchronous, decoupled communication between services. Instead of Service A calling Service B synchronously, A publishes a message to a broker, and B consumes it at its own pace. RabbitMQ implements AMQP — it has exchanges that route messages to queues via routing keys. Exchange types are: direct (exact match), fanout (broadcast to all), topic (wildcard patterns), and headers. Messages should be persistent, queues durable, and failures handled with Dead Letter Queues. Kafka is an event streaming platform — messages persist to disk, consumers track their own offset and can replay history. Kafka partitions enable parallel consumption within a consumer group, with ordering guaranteed per partition. I'd use RabbitMQ for task queues and complex routing, Kafka for high-throughput event streaming, event sourcing, and when multiple services need to independently consume the same events."

## 4. Webhooks

### What is a Webhook?

A **webhook** is a way for a service to **notify your server in real-time** when an event happens — by making an HTTP POST request to a URL you provide.

**Analogy:**
> Polling is like calling a pizza shop every 5 minutes: "Is my pizza ready?" Webhooks are like giving the shop your phone number so they call YOU when it's ready. You don't have to check — they notify you.

**Webhook vs Polling:**
```
POLLING (your server asks repeatedly):
  Your Server: "Any new payments?" → Stripe: "No"   (1,000 wasted requests)
  Your Server: "Any new payments?" → Stripe: "No"
  Your Server: "Any new payments?" → Stripe: "Yes! $99 from Alice"

WEBHOOK (Stripe calls you):
  Stripe: POST https://yourserver.com/webhooks/stripe
          Body: { "event": "payment.succeeded", "amount": 99, "customer": "Alice" }

→ Zero wasted requests. Instant notification. 🎯
```

---

### How Webhooks Work

```
1. You register your URL with the provider:
   Stripe Dashboard → Webhooks → Add endpoint: https://api.myapp.com/webhooks/stripe
   Events: payment_intent.succeeded, customer.subscription.deleted

2. Event happens on their side (e.g., payment succeeds)

3. Provider sends HTTP POST to your URL:
   POST https://api.myapp.com/webhooks/stripe
   Content-Type: application/json
   Stripe-Signature: t=1614556800,v1=abc123...   ← signature for verification

   {
     "id": "evt_abc123",
     "type": "payment_intent.succeeded",
     "data": {
       "object": {
         "id": "pi_xyz",
         "amount": 9900,
         "currency": "usd",
         "customer": "cus_alice"
       }
     }
   }

4. Your server processes the event and returns 200 OK

5. If you return 4xx/5xx, provider will retry (usually with exponential backoff)
```

---

### Building a Webhook Receiver

**The most important rule: respond fast, process async.**

```javascript
const express = require('express');
const crypto = require('crypto');
const app = express();

// CRITICAL: Use raw body for signature verification (NOT json parsed)
app.post('/webhooks/stripe',
  express.raw({ type: 'application/json' }),
  async (req, res) => {

    // Step 1: Verify signature IMMEDIATELY (security!)
    const sig = req.headers['stripe-signature'];
    const webhookSecret = process.env.STRIPE_WEBHOOK_SECRET;

    let event;
    try {
      event = stripe.webhooks.constructEvent(req.body, sig, webhookSecret);
    } catch (err) {
      console.error('Webhook signature verification failed:', err.message);
      return res.status(400).send(`Webhook Error: ${err.message}`);
    }

    // Step 2: Respond FAST with 200 (before processing!)
    // Provider times out if you take too long (usually 5-30 seconds)
    res.status(200).json({ received: true });

    // Step 3: Process asynchronously (after responding)
    setImmediate(async () => {
      try {
        await processWebhookEvent(event);
      } catch (err) {
        console.error('Failed to process webhook:', err);
        // Log to error tracking (Sentry, Datadog)
      }
    });
  }
);

async function processWebhookEvent(event) {
  switch (event.type) {
    case 'payment_intent.succeeded':
      const paymentIntent = event.data.object;
      await db.orders.update(
        { status: 'PAID' },
        { where: { stripePaymentId: paymentIntent.id } }
      );
      await sendOrderConfirmationEmail(paymentIntent.customer);
      break;

    case 'customer.subscription.deleted':
      const subscription = event.data.object;
      await db.users.update(
        { plan: 'FREE' },
        { where: { stripeCustomerId: subscription.customer } }
      );
      break;

    default:
      console.log(`Unhandled event type: ${event.type}`);
  }
}
```

### Webhook Security — Signature Verification

**Anyone on the internet can POST to your webhook URL.**
You must verify that the request actually came from the expected provider.

**How it works:**
```
Provider knows a secret key (you set it, only you and provider know it).

When sending webhook:
  1. Provider creates: payload = timestamp + "." + body
  2. Signs it:         signature = HMAC-SHA256(payload, secret_key)
  3. Sends header:     Stripe-Signature: t=1614556800,v1=<signature>

Your server:
  1. Reads timestamp from header
  2. Creates:     expected_payload = timestamp + "." + raw_body
  3. Computes:    expected_sig = HMAC-SHA256(expected_payload, your_secret)
  4. Compares:    expected_sig === received_sig?
     → Match:    Legitimate webhook ✅
     → No match: Reject with 400 ❌
```

**Manual signature verification (understanding the concept):**
```javascript
function verifyWebhookSignature(rawBody, signatureHeader, secret) {
  // Parse "t=1614556800,v1=abc123def456..."
  const parts = signatureHeader.split(',');
  const timestamp = parts.find(p => p.startsWith('t=')).split('=')[1];
  const receivedSig = parts.find(p => p.startsWith('v1=')).split('=')[1];

  // Reconstruct the signed payload
  const signedPayload = `${timestamp}.${rawBody}`;

  // Compute expected signature
  const expectedSig = crypto
    .createHmac('sha256', secret)
    .update(signedPayload)
    .digest('hex');

  // Use timingSafeEqual to prevent timing attacks
  const match = crypto.timingSafeEqual(
    Buffer.from(receivedSig, 'hex'),
    Buffer.from(expectedSig, 'hex')
  );

  // Also check timestamp to prevent replay attacks (reject if >5 min old)
  const age = Math.floor(Date.now() / 1000) - parseInt(timestamp);
  if (age > 300) throw new Error('Webhook too old — possible replay attack');

  return match;
}
```

### Idempotency in Webhooks

Providers **retry** failed webhooks. Your handler will receive the **same event multiple times**. You must handle this safely.

```
Stripe sends: payment.succeeded (evt_abc123)
Your server crashes before responding
Stripe retries: payment.succeeded (evt_abc123)  ← same event ID!
Your server processes it again
Result: Customer charged twice? Order fulfilled twice? 😱

Solution: Idempotency check using event ID

async function processWebhookEvent(event) {
  // Check if we've already processed this event
  const existing = await db.processedWebhooks.findOne({
    where: { eventId: event.id }
  });

  if (existing) {
    console.log(`Event ${event.id} already processed, skipping`);
    return;  // Idempotent! Safe to ignore duplicates.
  }

  // Process the event
  await handlePaymentSuccess(event.data.object);

  // Mark as processed
  await db.processedWebhooks.create({
    eventId: event.id,
    processedAt: new Date()
  });
}
```

---

### Webhook Retry Strategies (Provider Side)

Most providers retry failed webhooks with **exponential backoff**:

```
Attempt 1: immediately
Attempt 2: after 5 minutes
Attempt 3: after 30 minutes
Attempt 4: after 2 hours
Attempt 5: after 5 hours
...
After 3 days: give up, mark as failed

This means:
→ Your endpoint must be idempotent (same event may arrive multiple times)
→ A 2xx response = success (remove from retry queue)
→ 4xx/5xx response = failure (add to retry queue)
→ Timeout (no response in 5-30s) = failure (add to retry queue)
```

### Sending Your Own Webhooks (Outbound)

If you're building a platform and want to notify your customers' systems:

```javascript
const crypto = require('crypto');
const axios = require('axios');

class WebhookSender {
  constructor(secret) {
    this.secret = secret;
  }

  // Sign the payload
  sign(payload) {
    const timestamp = Math.floor(Date.now() / 1000);
    const body = JSON.stringify(payload);
    const signedPayload = `${timestamp}.${body}`;
    const signature = crypto
      .createHmac('sha256', this.secret)
      .update(signedPayload)
      .digest('hex');
    return { timestamp, signature };
  }

  // Send with retry logic
  async send(url, event, payload, retries = 3) {
    const { timestamp, signature } = this.sign(payload);

    for (let attempt = 1; attempt <= retries; attempt++) {
      try {
        const response = await axios.post(url, payload, {
          headers: {
            'Content-Type': 'application/json',
            'X-Webhook-Event': event,
            'X-Webhook-Signature': `t=${timestamp},v1=${signature}`,
            'X-Webhook-ID': crypto.randomUUID()
          },
          timeout: 10000  // 10 second timeout
        });

        console.log(`Webhook delivered to ${url} on attempt ${attempt}`);
        await this.logDelivery(url, event, 'SUCCESS', response.status);
        return;

      } catch (err) {
        console.error(`Attempt ${attempt} failed:`, err.message);
        if (attempt < retries) {
          const delay = Math.pow(2, attempt) * 1000;  // 2s, 4s, 8s
          await new Promise(resolve => setTimeout(resolve, delay));
        } else {
          await this.logDelivery(url, event, 'FAILED', null);
        }
      }
    }
  }

  async logDelivery(url, event, status, httpStatus) {
    await db.webhookLogs.create({ url, event, status, httpStatus, sentAt: new Date() });
  }
}
```

### Webhook Best Practices Checklist

```
Receiving webhooks:
  ✅ Verify signature before processing
  ✅ Check timestamp to prevent replay attacks
  ✅ Respond 200 immediately, process async
  ✅ Implement idempotency using event ID
  ✅ Use raw body for signature verification (not parsed JSON)
  ✅ Log all incoming webhooks (event ID, type, timestamp)
  ✅ Use a queue (RabbitMQ/Kafka) to buffer processing

Sending webhooks:
  ✅ Sign payloads with HMAC-SHA256
  ✅ Include unique event ID in every payload
  ✅ Implement retry with exponential backoff
  ✅ Respect receiver's response time (timeout after 10-30s)
  ✅ Provide a dashboard for customers to see delivery logs
  ✅ Allow customers to re-send failed webhooks manually
  ✅ Include event type in header (X-Webhook-Event)
```

### Interview-Ready Summary

> "Webhooks are HTTP callbacks — instead of polling a third-party API repeatedly, you register a URL and they POST to it when events happen. The receiver must do three critical things: verify the signature (using HMAC-SHA256 with a shared secret to prove the request is genuine), respond with 200 immediately (before processing, since providers have short timeouts), and implement idempotency using the event ID (because providers retry on failure, and you'll receive duplicates). I always use raw body for signature verification — if you parse JSON first, the body differs slightly and verification fails. For outbound webhooks, I sign payloads, implement exponential backoff retry, and log all delivery attempts."
