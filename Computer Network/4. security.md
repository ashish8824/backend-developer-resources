## 1. Authentication Protocols

### Authentication vs Authorization — Get This Right First

These two words sound similar but mean completely different things:

```
Authentication (AuthN) → WHO are you?
  "Prove your identity"
  → Login with username + password
  → Present a JWT token
  → Show your Google account

Authorization (AuthZ) → WHAT are you allowed to do?
  "What permissions do you have?"
  → Can you access /admin?
  → Can you delete other users' posts?
  → Can you read this file?
```

**Analogy:**
> At an airport: showing your passport is **authentication** (proving who you are). The boarding pass that says which seat you can sit in is **authorization** (what you're allowed to do). Both are needed.

---

### JWT — JSON Web Token

**JWT** is a compact, self-contained way to transmit information between parties as a digitally signed token. It's the most common token format used in modern APIs.

**JWT is NOT a protocol — it's a token format.**

#### Structure of a JWT

A JWT has 3 parts separated by dots:

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiI0MiIsIm5hbWUiOiJBbGljZSIsInJvbGUiOiJhZG1pbiIsImlhdCI6MTcwMDAwMDAwMCwiZXhwIjoxNzAwMDAzNjAwfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

HEADER . PAYLOAD . SIGNATURE
  ↑          ↑          ↑
Base64    Base64    HMAC-SHA256
encoded   encoded   of header+payload
```

**Header:**
```json
{
  "alg": "HS256",    // Algorithm used to sign (HS256, RS256, ES256)
  "typ": "JWT"       // Token type
}
```

**Payload (Claims):**
```json
{
  "sub": "42",              // Subject — user ID (standard claim)
  "name": "Alice",          // Custom claim
  "email": "alice@ex.com",  // Custom claim
  "role": "admin",          // Custom claim
  "iat": 1700000000,        // Issued At (Unix timestamp)
  "exp": 1700003600,        // Expiry (1 hour later)
  "iss": "api.myapp.com",   // Issuer
  "aud": "myapp.com"        // Audience
}
```

**Signature:**
```
HMAC-SHA256(
  base64url(header) + "." + base64url(payload),
  secret_key
)
```

#### How JWT Verification Works

```
Client sends: Authorization: Bearer eyJhbGc...

Server:
1. Split token into header.payload.signature
2. Recompute: expected_sig = HMAC-SHA256(header + "." + payload, secret)
3. Compare:   expected_sig === received_sig?
   → Match:   Token not tampered with ✅
   → No match: Reject 401 ❌
4. Check exp: Is current time < exp?
   → Yes: Token valid ✅
   → No:  Token expired, reject 401 ❌
5. Extract claims and use: userId = payload.sub
```

**Critical: JWT payload is Base64 encoded, NOT encrypted.**
```
Anyone can decode the payload and read it!
base64decode("eyJzdWIiOiI0MiIsI...") → { "sub": "42", "role": "admin" }

Rules:
✅ Store userId, role, permissions in JWT
❌ NEVER store passwords, credit cards, SSNs in JWT
```

#### JWT in Practice

```javascript
const jwt = require('jsonwebtoken');

const SECRET = process.env.JWT_SECRET;  // Store in env, NEVER hardcode

// Generate token (at login)
function generateToken(user) {
  return jwt.sign(
    {
      sub: user.id,
      email: user.email,
      role: user.role
    },
    SECRET,
    {
      expiresIn: '1h',         // Short-lived access token
      issuer: 'api.myapp.com',
      audience: 'myapp.com'
    }
  );
}

// Verify token (middleware)
function authenticateToken(req, res, next) {
  const authHeader = req.headers['authorization'];
  const token = authHeader && authHeader.split(' ')[1]; // "Bearer <token>"

  if (!token) return res.status(401).json({ error: 'No token provided' });

  try {
    const payload = jwt.verify(token, SECRET, {
      issuer: 'api.myapp.com',
      audience: 'myapp.com'
    });
    req.user = payload;    // { sub: "42", email: "...", role: "admin" }
    next();
  } catch (err) {
    if (err.name === 'TokenExpiredError') {
      return res.status(401).json({ error: 'Token expired' });
    }
    return res.status(403).json({ error: 'Invalid token' });
  }
}

// Usage
app.get('/profile', authenticateToken, (req, res) => {
  res.json({ userId: req.user.sub, email: req.user.email });
});
```

#### Access Token + Refresh Token Pattern

Short-lived access tokens expire quickly (15 min–1 hour). Refresh tokens let users stay logged in without re-entering credentials.

```
LOGIN:
  Client → POST /login { email, password }
  Server validates credentials
  Server returns:
    accessToken:  expires in 15 minutes  (stored in memory/variable)
    refreshToken: expires in 30 days     (stored in HttpOnly cookie)

NORMAL REQUEST:
  Client → GET /profile
            Authorization: Bearer <accessToken>
  Server → 200 OK (token valid)

ACCESS TOKEN EXPIRES:
  Client → GET /profile
            Authorization: Bearer <expiredAccessToken>
  Server → 401 Token Expired

CLIENT REFRESHES:
  Client → POST /auth/refresh
            Cookie: refreshToken=<refreshToken>  (sent automatically)
  Server validates refreshToken
  Server returns: new accessToken (15 min)
  Client retries original request with new access token

LOGOUT:
  Client → POST /auth/logout
  Server → invalidates refreshToken in DB (blacklists it)
  Client → clears accessToken from memory
```

```
Why store refresh token in HttpOnly cookie?
→ JavaScript cannot read HttpOnly cookies
→ XSS attack (malicious JS injected into your page) can't steal it
→ Access token in memory (JS variable) — lost on page refresh, never hits disk
```

### OAuth 2.0

**OAuth 2.0** is an **authorization framework** (not authentication!) that lets a user grant a third-party application limited access to their resources — without sharing their password.

**Analogy:**
> You want a parking valet to park your car. You give them a **valet key** — it opens the car door and starts the engine, but doesn't open the trunk or glove box. You didn't give them your master key. OAuth lets apps get a "valet key" to your data.

**Real-world example:**
> "Sign in with Google" on a third-party app. You don't give the app your Google password — Google issues a token that lets the app access only what you approved (e.g., your email and profile).

#### OAuth 2.0 Roles

```
Resource Owner   → The user (you) who owns the data
Client           → The third-party app that wants access
Authorization Server → Issues tokens (Google, GitHub, your own auth server)
Resource Server  → The API that holds the protected data (Google APIs)
```

#### The Authorization Code Flow (Most Secure, Most Common)

Used when there's a browser-based redirect. This is what "Login with Google" uses.

```
USER                  YOUR APP (Client)           GOOGLE (Auth Server)
  |                         |                              |
  | clicks "Login w/ Google"|                              |
  |─────────────────────────►                              |
  |                         |                              |
  |                         |── Redirect to Google ────────►
  |                         |  /authorize?                 |
  |                         |  client_id=abc               |
  |                         |  redirect_uri=myapp/callback |
  |                         |  scope=email profile         |
  |                         |  response_type=code          |
  |                         |  state=random_csrf_token     |
  |                         |                              |
  |◄──────────────────── Google Login Page ────────────────|
  | User logs in + consents |                              |
  |─────────────────────── User Approves ─────────────────►|
  |                         |                              |
  |                         |◄── Redirect to callback ─────|
  |                         |   ?code=AUTH_CODE            |
  |                         |   &state=random_csrf_token   |
  |                         |                              |
  |    [Verify state matches to prevent CSRF]              |
  |                         |                              |
  |                         |── POST /token ───────────────►|
  |                         |   code=AUTH_CODE             |
  |                         |   client_secret=xyz          |
  |                         |   grant_type=authorization_code
  |                         |                              |
  |                         |◄── access_token + refresh ───|
  |                         |                              |
  |                         |── GET /userinfo ─────────────►|
  |                         |   Authorization: Bearer token|
  |                         |◄── { email, name, ... } ─────|
  |                         |                              |
  |◄──── Logged In! ────────|                              |
```

**The `state` parameter is critical** — it prevents CSRF attacks. Your app sets a random value, Google echoes it back, you verify it matches.

#### OAuth 2.0 Grant Types

```
Authorization Code      → Browser-based apps (most secure, uses redirect)
Authorization Code+PKCE → Mobile/SPA apps (no client secret, uses code challenge)
Client Credentials      → Machine-to-machine (no user involved)
                          e.g., your backend calling another backend API
Device Code             → TVs, CLIs (device shows a code, user enters on phone)
Implicit                → DEPRECATED ❌ (use Auth Code + PKCE instead)
```

**Client Credentials (Machine-to-Machine):**
```javascript
// Your service calling another internal service
const response = await fetch('https://auth.example.com/token', {
  method: 'POST',
  body: new URLSearchParams({
    grant_type: 'client_credentials',
    client_id: process.env.CLIENT_ID,
    client_secret: process.env.CLIENT_SECRET,
    scope: 'read:orders write:inventory'
  })
});
const { access_token } = await response.json();
// Now use access_token to call the protected API
```

### OpenID Connect (OIDC)

**OpenID Connect** = OAuth 2.0 + **Authentication layer**.

OAuth 2.0 only handles authorization (access to resources). It doesn't tell you WHO the user is. OIDC adds identity.

```
OAuth 2.0:   "This token lets you access the user's Gmail"
              → Says NOTHING about who the user is

OIDC:        "This token lets you access the user's Gmail,
              AND here's a signed ID Token proving the user is alice@gmail.com"
```

**What OIDC adds:**
```
1. ID Token — a JWT containing user identity information
   {
     "sub": "1234567890",        // Unique user ID at the provider
     "name": "Alice Smith",
     "email": "alice@gmail.com",
     "email_verified": true,
     "picture": "https://...",
     "iss": "https://accounts.google.com",
     "aud": "your-client-id",
     "iat": 1700000000,
     "exp": 1700003600
   }

2. UserInfo Endpoint — GET /userinfo with access token → user details

3. Standard Scopes:
   openid   → Required for OIDC (enables ID token)
   profile  → name, picture, etc.
   email    → email, email_verified
```

**OIDC Flow (same as OAuth Auth Code, just add `openid` scope):**
```
/authorize?
  client_id=abc
  redirect_uri=myapp/callback
  scope=openid email profile     ← "openid" triggers OIDC
  response_type=code

Token endpoint returns:
  access_token   → For calling APIs
  refresh_token  → For refreshing access
  id_token       → JWT with user identity ← NEW in OIDC
```

---

### OAuth 2.0 vs OIDC vs JWT Summary

```
JWT         → A TOKEN FORMAT. Used by OAuth and OIDC to carry data.
OAuth 2.0   → An AUTHORIZATION FRAMEWORK. "App gets access to resources."
OIDC        → An AUTHENTICATION LAYER on top of OAuth. "Who is the user?"

Analogy:
JWT     = the format of an ID card (plastic card with fields)
OAuth   = the system that issues access passes
OIDC    = the system that also verifies and certifies your identity
```

---

### Interview-Ready Summary

> "Authentication answers WHO you are; authorization answers WHAT you can do. JWT is a token format — three Base64-encoded parts: header (algorithm), payload (claims like userId, role, expiry), and signature (HMAC to detect tampering). The payload is readable by anyone, so never put secrets in it. Access tokens are short-lived (15 min) and stored in memory; refresh tokens are long-lived (30 days) and stored in HttpOnly cookies to prevent XSS theft. OAuth 2.0 is an authorization framework — it lets users grant third-party apps limited access without sharing passwords. The Authorization Code flow redirects the user, gets an auth code, then exchanges it server-side for tokens. OIDC is OAuth 2.0 + identity — it adds an ID Token (a signed JWT with user info) and a UserInfo endpoint. For machine-to-machine communication, I use the Client Credentials grant."

## 2. CORS

### What is CORS?

**CORS** (Cross-Origin Resource Sharing) is a **browser security mechanism** that controls which websites can make HTTP requests to your API from a different origin.

**What is an "origin"?**
```
Origin = Protocol + Domain + Port

https://myapp.com        → one origin
https://api.myapp.com    → DIFFERENT origin (different subdomain)
http://myapp.com         → DIFFERENT origin (different protocol)
https://myapp.com:3000   → DIFFERENT origin (different port)
https://otherdomain.com  → DIFFERENT origin (different domain)
```

**The Same-Origin Policy (SOP):**
> By default, browsers block JavaScript from making requests to a **different origin** than the page it's on. This is a security feature — it prevents evil.com from using your logged-in browser to make API calls to yourbank.com and steal your data.

**CORS is the mechanism that RELAXES this restriction** — allowing specific origins you trust.

---

### How CORS Works

**Simple request (no preflight):**
```
Browser (on https://myapp.com):
  JavaScript: fetch('https://api.myapp.com/users')

Request:
  GET /users HTTP/1.1
  Host: api.myapp.com
  Origin: https://myapp.com      ← Browser automatically adds this

Server response:
  HTTP/1.1 200 OK
  Access-Control-Allow-Origin: https://myapp.com    ← CORS header
  Content-Type: application/json

  [{"id": 1, "name": "Alice"}]

Browser checks:
  Does "Access-Control-Allow-Origin" include my origin "https://myapp.com"?
  → Yes: Allow the JS to read the response ✅
  → No:  Block the response, throw CORS error ❌
         (request DID reach the server — CORS blocks the RESPONSE from JS)
```

**Important: CORS errors are browser-enforced.**
> The request reaches your server. The server responds. But the browser blocks your JavaScript from reading the response if CORS headers are wrong. This is why server-side code (curl, Postman) never has CORS issues — only browsers enforce CORS.

---

### Preflight Requests

For "non-simple" requests, the browser first sends an **OPTIONS preflight** to ask: "Will you allow this?"

**When is a preflight triggered?**
```
Simple requests (NO preflight):
  Methods:  GET, HEAD, POST
  Headers:  Only simple headers (Accept, Content-Type: text/plain etc.)

Non-simple requests (PREFLIGHT required):
  Methods:  PUT, PATCH, DELETE, or any custom method
  Headers:  Authorization, Content-Type: application/json, X-Custom-Header
            (basically anything you use in a real API)
```

**Preflight flow:**
```
Browser (https://myapp.com)                    Server (api.myapp.com)
         |                                              |
         |────── OPTIONS /users ───────────────────────►|
         |  Origin: https://myapp.com                  |
         |  Access-Control-Request-Method: DELETE       |
         |  Access-Control-Request-Headers: Authorization, Content-Type
         |                                              |
         |◄────── 204 No Content ──────────────────────|
         |  Access-Control-Allow-Origin: https://myapp.com
         |  Access-Control-Allow-Methods: GET, POST, PUT, DELETE, PATCH
         |  Access-Control-Allow-Headers: Authorization, Content-Type
         |  Access-Control-Max-Age: 86400    ← Cache preflight for 24 hours
         |                                              |
         | (preflight approved — now send real request) |
         |                                              |
         |────── DELETE /users/42 ─────────────────────►|
         |  Authorization: Bearer eyJ...               |
         |                                              |
         |◄────── 200 OK ──────────────────────────────|
```

**Access-Control-Max-Age** is important for performance — it tells the browser to cache the preflight result. Without it, every request triggers a preflight (doubles your request count).

---

### CORS Headers Deep Dive

**Response headers your server must set:**

```
Access-Control-Allow-Origin: https://myapp.com
  → Which origin is allowed. Use specific origins, not * for credentialed requests.
  → * means ANY origin (use only for truly public APIs with no auth)

Access-Control-Allow-Methods: GET, POST, PUT, DELETE, PATCH, OPTIONS
  → Which HTTP methods are allowed

Access-Control-Allow-Headers: Authorization, Content-Type, X-Request-ID
  → Which request headers the client can send

Access-Control-Expose-Headers: X-Total-Count, X-RateLimit-Remaining
  → Which response headers JS can read (by default, only basic headers are exposed)

Access-Control-Allow-Credentials: true
  → Allow cookies and Authorization headers to be sent with the request
  → REQUIRES Access-Control-Allow-Origin to be a specific origin (not *)

Access-Control-Max-Age: 86400
  → Cache preflight response for 86400 seconds (24 hours)
```

---

### Proper CORS Configuration

**Node.js / Express:**
```javascript
const cors = require('cors');

// Option 1: Allow specific origins (recommended for production)
const allowedOrigins = [
  'https://myapp.com',
  'https://www.myapp.com',
  'https://admin.myapp.com',
  process.env.NODE_ENV === 'development' ? 'http://localhost:3000' : null
].filter(Boolean);

app.use(cors({
  origin: (origin, callback) => {
    // Allow requests with no origin (curl, mobile apps, Postman)
    if (!origin) return callback(null, true);

    if (allowedOrigins.includes(origin)) {
      callback(null, true);
    } else {
      callback(new Error(`Origin ${origin} not allowed by CORS`));
    }
  },
  methods: ['GET', 'POST', 'PUT', 'PATCH', 'DELETE', 'OPTIONS'],
  allowedHeaders: ['Authorization', 'Content-Type', 'X-Request-ID'],
  exposedHeaders: ['X-Total-Count', 'X-RateLimit-Remaining'],
  credentials: true,          // Allow cookies + Authorization header
  maxAge: 86400               // Cache preflight for 24 hours
}));

// Option 2: Handle preflight explicitly
app.options('*', cors());     // Enable pre-flight for all routes
```

**Nginx CORS headers:**
```nginx
location /api/ {
    # Handle preflight
    if ($request_method = 'OPTIONS') {
        add_header 'Access-Control-Allow-Origin' 'https://myapp.com';
        add_header 'Access-Control-Allow-Methods' 'GET, POST, PUT, DELETE, OPTIONS';
        add_header 'Access-Control-Allow-Headers' 'Authorization, Content-Type';
        add_header 'Access-Control-Max-Age' 86400;
        add_header 'Content-Length' 0;
        return 204;
    }

    add_header 'Access-Control-Allow-Origin' 'https://myapp.com';
    add_header 'Access-Control-Allow-Credentials' 'true';

    proxy_pass http://backend:8080;
}
```

### Common CORS Mistakes

```
❌ Mistake 1: Using * with credentials
  Access-Control-Allow-Origin: *
  Access-Control-Allow-Credentials: true
  → BROWSER REJECTS THIS. * and credentials cannot be combined.

❌ Mistake 2: Setting CORS headers only on 200 responses
  → Preflight expects headers on 204/200 OPTIONS response
  → Also set headers on error responses (4xx, 5xx)

❌ Mistake 3: Forgetting OPTIONS method
  Access-Control-Allow-Methods: GET, POST, DELETE
  → Preflight itself uses OPTIONS — if OPTIONS is missing, preflight fails

❌ Mistake 4: Missing Access-Control-Allow-Headers for custom headers
  Client sends: X-Request-ID: abc123
  Server doesn't list it in Access-Control-Allow-Headers
  → Browser blocks the request

✅ Correct approach:
  - Specific origins (not *)
  - credentials: true only when needed
  - Include OPTIONS in allowed methods
  - List ALL custom headers
  - Set maxAge to reduce preflight overhead
```

---

### Interview-Ready Summary

> "CORS is a browser security mechanism based on the Same-Origin Policy — by default, JavaScript can't read responses from a different origin. CORS relaxes this by having the server include `Access-Control-Allow-Origin` headers. For non-simple requests (anything with Authorization header, JSON Content-Type, PUT/DELETE), the browser first sends an OPTIONS preflight to check permissions before the real request. Key headers are: `Access-Control-Allow-Origin` (specific origin or *), `Access-Control-Allow-Methods`, `Access-Control-Allow-Headers`, `Access-Control-Allow-Credentials`, and `Access-Control-Max-Age` (cache the preflight). Critical gotcha: you can't combine `*` with `credentials: true` — when using cookies or Authorization headers, you must specify the exact origin."

## 3. Rate Limiting & Throttling

### What is Rate Limiting?

**Rate limiting** controls how many requests a client can make to your API in a given time window. Without it, a single bad actor (or a buggy client) can bring down your server.

**Analogy:**
> A nightclub has a maximum capacity. The bouncer only lets in a certain number of people per hour. It doesn't matter if you're a VIP or a regular — the limit exists to keep everyone safe inside.

**Why you need it:**
```
Without rate limiting:
  → One client sends 10,000 requests/second → server overloaded → everyone suffers
  → Brute-force attack: tries 1M password combinations in minutes
  → Web scraping: downloads your entire product catalog in seconds
  → DDoS amplification: abuses your API as part of an attack
```

---

### Rate Limiting Algorithms

**1. Fixed Window Counter (Simplest)**
```
Window: 1 minute
Limit:  100 requests

|-------- minute 1 --------|-------- minute 2 --------|
[req 1-100 allowed]        [req 1-100 allowed]
[req 101+ blocked]         [req 101+ blocked]

Problem: Boundary burst attack
  Client sends 100 requests at 11:59:55
  Window resets at 12:00:00
  Client sends 100 more at 12:00:05
  = 200 requests in 10 seconds! The "limit" was bypassed.
```

**2. Sliding Window Log**
```
Keep a log of timestamps of all requests in the last N seconds.
On each request:
  - Remove timestamps older than window
  - Count remaining timestamps
  - If count < limit: allow and add current timestamp
  - If count >= limit: reject

Accurate but memory-intensive (stores every timestamp).
```

**3. Sliding Window Counter (Best Balance)**
```
Combines fixed windows with weighted overlap:

Current window count + (previous window count × overlap %) ≤ limit

Example (limit=100, 1-min window):
  Previous minute: 80 requests
  Current minute (30s elapsed, 50% of window): 40 requests so far
  Estimated count: 40 + (80 × 0.5) = 40 + 40 = 80 → Allow

  If someone sent 60 more:
  60 + (80 × 0.5) = 60 + 40 = 100 → At limit, start blocking
```

**4. Token Bucket (Burst-Friendly)**
```
Imagine a bucket that holds tokens:
  → Bucket capacity: 100 tokens (max burst)
  → Refill rate: 10 tokens per second

Each request consumes 1 token.
If bucket is empty: reject the request.

Advantage: Allows short bursts (use all 100 tokens instantly)
           but long-term rate is controlled by refill rate.

Used by: AWS API Gateway, many traffic shapers
```

**5. Leaky Bucket (Smooth Output)**
```
Requests go IN at any rate.
Requests come OUT at a fixed rate.
If bucket overflows: requests are dropped.

Like a bucket with a small hole at the bottom.
Output is smooth regardless of bursty input.

Used for: Traffic shaping, ensuring smooth load on backend.
```

---

### Implementing Rate Limiting in Redis

Redis is perfect for rate limiting — it's fast, atomic, and supports expiry.

```javascript
const redis = require('redis');
const client = redis.createClient();

// Sliding Window Counter with Redis
async function rateLimiter(req, res, next) {
  const identifier = req.ip;           // or req.user?.id for authenticated users
  const limit = 100;
  const windowSeconds = 60;
  const key = `ratelimit:${identifier}`;

  const now = Date.now();
  const windowStart = now - (windowSeconds * 1000);

  // Use Redis sorted set: score = timestamp, member = unique request ID
  const pipeline = client.multi();

  pipeline.zRemRangeByScore(key, 0, windowStart);    // Remove old entries
  pipeline.zCard(key);                                // Count remaining
  pipeline.zAdd(key, { score: now, value: `${now}-${Math.random()}` });
  pipeline.expire(key, windowSeconds);                // Auto-cleanup

  const results = await pipeline.exec();
  const requestCount = results[1];   // Count before adding current request

  // Set rate limit headers (good practice)
  res.set({
    'X-RateLimit-Limit': limit,
    'X-RateLimit-Remaining': Math.max(0, limit - requestCount - 1),
    'X-RateLimit-Reset': Math.ceil((now + windowSeconds * 1000) / 1000)
  });

  if (requestCount >= limit) {
    return res.status(429).json({
      error: 'Too Many Requests',
      message: `Rate limit exceeded. Try again in ${windowSeconds} seconds.`,
      retryAfter: windowSeconds
    });
  }

  next();
}

app.use(rateLimiter);
```

### Rate Limiting Strategies

**Different limits for different contexts:**

```javascript
// Tiered rate limiting
const rateLimits = {
  // By authentication status
  anonymous:     { limit: 20,   window: 60 },   // 20 req/min
  authenticated: { limit: 200,  window: 60 },   // 200 req/min
  premium:       { limit: 1000, window: 60 },   // 1000 req/min

  // By endpoint sensitivity
  '/auth/login':         { limit: 5,  window: 60  },  // Brute force protection
  '/auth/forgot-password': { limit: 3, window: 300 }, // Prevent abuse
  '/api/send-email':     { limit: 10, window: 3600 }, // Prevent spam
  '/api/search':         { limit: 30, window: 60  },  // Expensive operation
};

function getLimit(req) {
  // Check endpoint-specific limits first
  const endpointLimit = rateLimits[req.path];
  if (endpointLimit) return endpointLimit;

  // Then by user tier
  if (req.user?.plan === 'premium') return rateLimits.premium;
  if (req.user) return rateLimits.authenticated;
  return rateLimits.anonymous;
}
```

**What to use as the identifier (key):**
```
For anonymous users:   IP address (req.ip)
For authenticated:     User ID (req.user.id) — better, handles proxies
For APIs:              API key

Multi-dimensional limiting (combine both):
  key1 = `ratelimit:ip:${req.ip}`          → Global IP limit
  key2 = `ratelimit:user:${req.user.id}`   → Per-user limit
  key3 = `ratelimit:ip:${req.ip}:${req.path}` → Per-IP per-endpoint
```

---

### Rate Limiting vs Throttling

```
Rate Limiting:   Hard stop. "You've hit your limit. Come back later."
                 Returns 429 Too Many Requests.

Throttling:      Slow down. "You're going too fast. I'll process your
                 requests slower, not reject them."
                 Adds delay/queuing rather than rejecting.

Throttling example:
  → Normal: respond in 50ms
  → Throttled: queue the request, respond in 2s
  → Heavily throttled: queue with 10s delay
```

---

### Interview-Ready Summary

> "Rate limiting controls how many requests a client can make in a time window, protecting against abuse, brute-force attacks, and accidental flooding. The four main algorithms are: fixed window (simple but vulnerable to boundary bursts), sliding window log (accurate but memory-heavy), sliding window counter (best balance), and token bucket (allows bursts while controlling long-term rate). I implement rate limiting with Redis — using sorted sets for sliding windows or increment+expire for fixed windows. I return HTTP 429 with `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `Retry-After` headers. I apply different limits by context: anonymous users get 20 req/min, authenticated 200, and sensitive endpoints like login get 5 attempts/min to prevent brute force."

## 4. Firewalls & Security Groups

### What is a Firewall?

A **firewall** is a network security system that monitors and controls incoming and outgoing network traffic based on predetermined security rules. It acts as a barrier between trusted internal networks and untrusted external networks.

**Analogy:**
> A firewall is the security guard at a building's entrance. They check everyone's ID against a list — certain people can enter, others are turned away. You can also have a rule: "No one from this building can go to that sketchy neighborhood" (outbound rules).

---

### Types of Firewalls

```
1. Network Firewall (traditional)
   → Hardware or software at the network perimeter
   → Filters by IP address and port
   → Examples: iptables (Linux), pfSense, Cisco ASA

2. Security Groups (cloud)
   → Virtual firewall for cloud instances (AWS EC2, GCP VMs)
   → Stateful (tracks connection state)
   → Applied at the instance level

3. Network ACLs (cloud)
   → Applied at the subnet level (another layer)
   → Stateless (each packet evaluated independently)
   → AWS VPC Network ACLs

4. WAF (Web Application Firewall)
   → Layer 7 — understands HTTP
   → Blocks SQL injection, XSS, bot traffic
   → Examples: AWS WAF, Cloudflare WAF, ModSecurity
```

---

### Inbound vs Outbound Rules

```
INBOUND rules  → Control traffic coming INTO your server
OUTBOUND rules → Control traffic going OUT from your server

Inbound example (web server):
┌────────────────────────────────────────────────────────┐
│ Type      Protocol  Port    Source          Action      │
├────────────────────────────────────────────────────────┤
│ HTTPS     TCP       443     0.0.0.0/0       ALLOW       │ ← Anyone
│ HTTP      TCP       80      0.0.0.0/0       ALLOW       │ ← Anyone (redirect)
│ SSH       TCP       22      10.0.0.0/8      ALLOW       │ ← Only internal IPs
│ Custom    TCP       8080    10.0.0.0/8      ALLOW       │ ← Only internal
│ All       All       All     0.0.0.0/0       DENY        │ ← Default deny
└────────────────────────────────────────────────────────┘

Outbound example (web server):
┌────────────────────────────────────────────────────────┐
│ Type      Protocol  Port    Destination     Action      │
├────────────────────────────────────────────────────────┤
│ HTTPS     TCP       443     0.0.0.0/0       ALLOW       │ ← External APIs
│ HTTP      TCP       80      0.0.0.0/0       ALLOW       │
│ PostgreSQL TCP      5432    10.0.2.0/24     ALLOW       │ ← DB subnet only
│ Redis     TCP       6379    10.0.3.0/24     ALLOW       │ ← Cache subnet
│ DNS       UDP       53      0.0.0.0/0       ALLOW       │ ← DNS resolution
│ SMTP      TCP       587     0.0.0.0/0       ALLOW       │ ← Email sending
│ All       All       All     0.0.0.0/0       DENY        │ ← Default deny
└────────────────────────────────────────────────────────┘
```

---

### AWS Security Groups in Depth

Security Groups in AWS are the most common firewall mechanism backend devs work with.

**Key properties:**
```
1. STATEFUL:
   If you allow inbound HTTP on port 80,
   the response is automatically allowed outbound.
   (You don't need a separate outbound rule for return traffic)

2. DEFAULT DENY:
   All traffic is denied unless explicitly allowed.
   There are NO deny rules — only allow rules.
   "If it's not in the allow list, it's blocked."

3. ATTACHED TO INSTANCES:
   Security groups are attached to EC2 instances, RDS, Lambda, etc.
   Not to subnets (that's Network ACLs).

4. MULTIPLE SECURITY GROUPS:
   An instance can have multiple security groups.
   Rules are combined (union of all allow rules).

5. SOURCE CAN BE ANOTHER SECURITY GROUP:
   Instead of an IP, use a security group as the source.
   → "Allow port 5432 from instances with the 'web-servers' security group"
   → Automatically applies to all instances in that group (even new ones!)
```

**Typical AWS security group setup for a 3-tier architecture:**

```
Internet
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│ Load Balancer Security Group (lb-sg)                         │
│  Inbound:  80, 443 from 0.0.0.0/0 (internet)               │
│  Outbound: 8080 to web-sg                                    │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│ Web/App Server Security Group (web-sg)                       │
│  Inbound:  8080 from lb-sg  ← source is security group!    │
│  Inbound:  22 from bastion-sg (SSH only from bastion host)  │
│  Outbound: 5432 to db-sg                                    │
│  Outbound: 6379 to cache-sg                                 │
│  Outbound: 443 to 0.0.0.0/0 (external APIs)                │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│ Database Security Group (db-sg)                              │
│  Inbound:  5432 from web-sg ONLY                            │
│  Outbound: (none needed — stateful)                         │
└─────────────────────────────────────────────────────────────┘
```

---

### Network ACLs vs Security Groups

| Feature | Security Groups | Network ACLs |
|---------|----------------|--------------|
| Level | Instance level | Subnet level |
| State | Stateful | Stateless |
| Rules | Allow only | Allow AND Deny |
| Rule evaluation | All rules evaluated | Rules evaluated in order (numbered) |
| Default | Deny all | Allow all (default VPC) |
| Use case | Primary firewall per instance | Extra subnet-level protection |

```
Stateless means:
  You allow inbound HTTP on port 80.
  The RESPONSE goes out on a random ephemeral port (e.g., 54231).
  You must ALSO allow outbound 1024-65535 for return traffic!
  (Security Groups handle this automatically; ACLs don't)
```

---

### Principle of Least Privilege

**The golden rule of firewall configuration:**

> Give each component **only the access it needs** — nothing more.

```
❌ Bad:
  Database inbound: 5432 from 0.0.0.0/0
  → Anyone on the internet can try to connect to your DB!

✅ Good:
  Database inbound: 5432 from web-sg
  → Only your web servers can connect to DB

❌ Bad:
  All outbound: Allow all traffic
  → If your server is compromised, attacker can reach anywhere

✅ Good:
  Outbound: Only allow specific needed ports/destinations
  → Compromised server can't easily call out to attacker's server
```

---

### Interview-Ready Summary

> "A firewall controls network traffic based on rules — inbound rules control what can reach your server, outbound rules control what your server can reach. In AWS, Security Groups are stateful virtual firewalls at the instance level — if you allow inbound traffic, the return traffic is automatically allowed without an outbound rule. They're allow-only (no deny rules); anything not explicitly allowed is blocked. I design security groups using the principle of least privilege: the load balancer accepts internet traffic on 443/80, app servers only accept traffic from the load balancer's security group, and the database only accepts connections from the app server's security group — using security group references instead of IPs so it scales automatically. Network ACLs are stateless subnet-level filters used as an additional layer."

## 5. DDoS Basics

### What is a DDoS Attack?

**DDoS** (Distributed Denial of Service) is an attack where thousands of compromised machines (a **botnet**) simultaneously flood your server with traffic, overwhelming it so legitimate users can't access it.

```
Normal traffic:
  User 1 ──────────────────────►
  User 2 ──────────────────────►  Your Server (handles fine)
  User 3 ──────────────────────►

DDoS attack:
  Bot 1  ──────────────────────►
  Bot 2  ──────────────────────►
  Bot 3  ──────────────────────►
  ...
  Bot 10,000 ──────────────────►  Your Server 💥 OVERWHELMED
  Legitimate User ─────────────►  (Can't get through)
```

**DoS vs DDoS:**
```
DoS  = One attacker, one machine. Easier to block (just block that IP).
DDoS = Distributed: thousands of IPs. Much harder to block.
```

---

### DDoS Attack Types

**Layer 3/4 Attacks (Network/Transport layer — Volume-based)**

These attacks overwhelm your bandwidth or connection capacity:

```
1. UDP Flood
   → Attacker sends massive amounts of UDP packets to random ports
   → Your server tries to process each one, gets overwhelmed
   → Response: ICMP "Destination Unreachable" floods your bandwidth

2. ICMP Flood (Ping Flood)
   → Floods target with ICMP Echo Request (ping) packets
   → Target spends all resources replying to pings

3. SYN Flood (TCP connection exhaustion)
   → Attacker sends thousands of TCP SYN packets (connection requests)
   → Never completes the 3-way handshake (no ACK sent)
   → Server keeps half-open connections in memory waiting for ACK
   → Connection table fills up → legitimate connections rejected
   
   Normal:  SYN → SYN-ACK → ACK (connection established)
   Attack:  SYN → SYN-ACK → (silence — attacker never sends ACK)
                              Server waits... memory fills up...

4. Amplification Attacks (traffic multiplication)
   → Attacker sends small request with SPOOFED source IP (victim's IP)
   → Server sends large response TO THE VICTIM
   → DNS Amplification: 40-byte query → 3,000-byte response (75x amplification)
   → NTP Amplification: 8-byte request → 468-byte response (60x amplification)
   → Attacker sends 1 Gbps of requests → victim gets 75 Gbps of traffic!
```

**Layer 7 Attacks (Application layer — Sophisticated)**

These look like legitimate traffic:

```
5. HTTP Flood
   → Thousands of bots send legitimate-looking HTTP GET/POST requests
   → Hard to distinguish from real users
   → Your app server runs out of threads/connections

6. Slowloris
   → Opens many connections to your server
   → Sends HTTP headers very slowly (never completes the request)
   → Holds each connection open for minutes
   → Server runs out of connections, legitimate users can't connect
   → Works even with low bandwidth!

7. Application-specific attacks
   → Request a resource that's expensive to compute (complex search query)
   → Exploit inefficient database query via API
   → Upload maximum-size file repeatedly
```

---

### DDoS Mitigation Strategies

**1. Anycast Network Diffusion (CDN-level)**
```
Without CDN:
  All DDoS traffic hits your single origin server → overwhelmed

With Cloudflare/AWS CloudFront:
  DDoS traffic is absorbed across 200+ global data centers
  Each PoP absorbs a fraction of the attack
  Your origin server sees only clean traffic
```

**2. Rate Limiting (covered in Section 3)**
```
Limit requests per IP:
  → 100 requests/min per IP from the internet
  → If exceeded: block at the edge (before reaching your server)

SYN cookies (kernel-level SYN flood protection):
  → Server doesn't store half-open connections
  → Encodes connection info in the SYN-ACK sequence number
  → Only allocates resources when the ACK arrives with correct value
```

**3. IP Reputation & Blocking**
```
Block known bad IPs:
  → Threat intelligence feeds (known botnet IPs)
  → Geo-blocking (block countries you don't serve)
  → Block IPs that exceed rate limits

AWS Shield: automatically detects and mitigates L3/L4 DDoS
AWS Shield Advanced: includes L7 protection, DDoS cost protection, 24/7 DRT team
Cloudflare: filters traffic before it reaches your infrastructure
```

**4. Web Application Firewall (WAF)**
```
For Layer 7 attacks, a WAF analyzes HTTP traffic:
  → Block requests matching attack patterns (SQLi, XSS, scanner signatures)
  → Challenge suspicious IPs with CAPTCHA
  → Block bad User-Agent strings (known bot/scanner tools)
  → Rate limit by URL pattern (/login endpoint gets 5 req/min)

AWS WAF rules:
  → AWS Managed Rules (OWASP Top 10 protection)
  → Bot Control (identifies and blocks bots)
  → Custom rules (your specific patterns)
```

**5. Connection Limiting & Timeouts**
```
Nginx config against Slowloris and connection floods:
  worker_connections 1024;
  client_header_timeout 10s;    # Max time to receive headers
  client_body_timeout 10s;      # Max time between body reads
  keepalive_timeout 30s;        # Close idle connections
  send_timeout 10s;             # Max time to send response

  # Limit connections per IP
  limit_conn_zone $binary_remote_addr zone=conn_limit:10m;
  limit_conn conn_limit 20;     # Max 20 concurrent connections per IP

  # Limit request rate
  limit_req_zone $binary_remote_addr zone=req_limit:10m rate=10r/s;
  limit_req zone=req_limit burst=20 nodelay;
```

**6. Auto-scaling (absorb the traffic)**
```
If you can't block it, scale to handle it:
  → AWS Auto Scaling: automatically add EC2 instances under load
  → Load balancer distributes traffic across instances
  → Expensive but buys time while mitigation kicks in

Combined with Spot Instances for cost control during attacks.
```

---

### DDoS Mitigation Architecture

```
Internet Traffic (including DDoS)
           │
           ▼
┌─────────────────────────────────────────────────────┐
│         Cloudflare / AWS CloudFront (Edge)           │
│  - Anycast absorption (200+ PoPs globally)           │
│  - DDoS detection + blocking at edge                 │
│  - Rate limiting per IP                              │
│  - WAF rules (OWASP, bot control)                   │
└─────────────────────────────────────────────────────┘
           │ (only clean traffic passes through)
           ▼
┌──────────────────────────────────────┐
│     AWS Shield + Load Balancer        │
│  - L3/L4 DDoS mitigation (Shield)    │
│  - Connection limits                  │
│  - Health checks + auto-failover      │
└──────────────────────────────────────┘
           │
           ▼
┌──────────────────────────────────────┐
│     Auto-Scaling App Servers          │
│  - Nginx connection limits            │
│  - Application-level rate limiting    │
│  - Request validation                 │
└──────────────────────────────────────┘
```

---

### Interview-Ready Summary

> "A DDoS attack uses thousands of bots to overwhelm a target. Layer 3/4 attacks are volume-based: UDP floods, ICMP floods, SYN floods (exhausting TCP connection tables), and amplification attacks (spoofed source IP causes large responses to hit the victim). Layer 7 attacks are more sophisticated: HTTP floods, Slowloris (holding connections open slowly), and expensive query attacks. Mitigation is layered: use a CDN like Cloudflare to absorb traffic across global PoPs, AWS Shield for automatic L3/L4 mitigation, a WAF for L7 attack patterns (SQL injection, bot signatures, rate limiting by URL), Nginx connection limits and timeouts to defend against Slowloris, and auto-scaling to absorb traffic while mitigations kick in. The key principle is defense in depth — no single layer stops everything."

## 6. HTTPS Enforcement

### Why Just Having HTTPS Isn't Enough

Having an HTTPS endpoint doesn't mean users are protected:

```
Scenario 1: User types "mybank.com" in browser
  → Browser first tries http://mybank.com (HTTP!)
  → Server redirects to https://mybank.com
  → During that first HTTP request, attacker can intercept (MITM)
  → Attacker returns a fake HTTP page instead of redirecting

Scenario 2: User clicks http://mybank.com link in old email
  → Goes to HTTP, gets downgraded, attacker intercepts

Solution: HSTS — tell browsers to NEVER use HTTP for your domain.
```

---

### HSTS — HTTP Strict Transport Security

**HSTS** is a response header that tells browsers: "This site is HTTPS only. For the next N seconds, never send HTTP requests to me — always upgrade to HTTPS automatically."

```
Server sends:
HTTP/1.1 200 OK
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```

**Breakdown:**
```
max-age=31536000        → Remember this for 1 year (31,536,000 seconds)
includeSubDomains       → Also apply to ALL subdomains
                          (api.myapp.com, cdn.myapp.com, etc.)
preload                 → Include in browser's built-in HSTS preload list
                          (never sends HTTP even on very first visit)
```

**How HSTS protects:**
```
BEFORE HSTS:
  User visits http://myapp.com
  → HTTP request goes over the wire → MITM can intercept!

AFTER HSTS (browser has seen the header):
  User visits http://myapp.com
  → Browser converts to HTTPS internally (307 Internal Redirect)
  → HTTPS request goes out → never exposes HTTP!
  → MITM can't intercept because there's no HTTP to intercept
```

**HSTS Preload List:**
```
Problem with HSTS: On the VERY FIRST visit, there's no HTTP→HTTPS upgrade.
                   The first visit is still vulnerable.

Solution: HSTS Preload List
  → Submit your domain to hstspreload.org
  → Google, Mozilla, Apple include your domain in their browser builds
  → Users get HSTS even on their VERY FIRST visit, ever!
  → Baked into Chrome, Firefox, Safari, Edge

Requirements to preload:
  1. Serve a valid HTTPS certificate
  2. Redirect HTTP to HTTPS
  3. Serve HSTS header with max-age >= 1 year
  4. Include includeSubDomains AND preload directives
  5. ALL subdomains must also support HTTPS
```

**WARNING: HSTS is hard to undo.**
```
If you set max-age=31536000 and then need to go back to HTTP:
  → Users' browsers will refuse HTTP for 1 year!
  → Even if you remove the header, browsers ignore that (security feature)

Strategy:
  1. Start with short max-age: max-age=300 (5 minutes) for testing
  2. Gradually increase: 3600, 86400, 604800, 2592000, 31536000
  3. Add includeSubDomains only when ALL subdomains support HTTPS
  4. Add preload last (it's permanent once submitted)
```

**Nginx HSTS configuration:**
```nginx
server {
    listen 80;
    server_name myapp.com www.myapp.com;

    # Redirect ALL HTTP to HTTPS
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl http2;
    server_name myapp.com www.myapp.com;

    ssl_certificate /etc/letsencrypt/live/myapp.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/myapp.com/privkey.pem;

    # HSTS — 1 year, include subdomains, preload
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;

    # Additional security headers
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-Frame-Options "DENY" always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    add_header Content-Security-Policy "default-src 'self'; script-src 'self'" always;
}
```

---

### Certificate Pinning

**Certificate pinning** is a technique where the client (mobile app, IoT device) hard-codes the expected certificate or public key and **refuses connections** if the server presents a different certificate.

**Why it exists:**
```
Normal HTTPS:
  Client trusts ANY certificate signed by a trusted CA.
  If an attacker compromises or tricks a CA into issuing a certificate for your domain,
  they can MITM your HTTPS traffic — the certificate appears valid!

With Certificate Pinning:
  Client hard-codes your specific certificate (or public key hash).
  Even if attacker has a "valid" CA-signed certificate for your domain,
  the client rejects it because it doesn't match the pinned value.
```

**Types of pinning:**

```
1. Certificate Pinning (pin the full certificate)
   → Most restrictive
   → Certificate renewal = pin update = app update required!
   → If certificate expires and pin not updated: APP BREAKS for all users

2. Public Key Pinning (pin the public key only) ← Recommended
   → Pin the public key hash (SPKI hash)
   → You can renew the certificate but keep the same key pair
   → No app update needed for routine renewals

3. CA Pinning (pin the intermediate or root CA)
   → Least restrictive
   → Trusts any certificate from your chosen CA
   → Easier to manage, less protection
```

**Public key pinning in an iOS app:**
```swift
// Extract the public key hash from your certificate:
// openssl x509 -in cert.pem -pubkey -noout | openssl pkey -pubin -outform der | openssl dgst -sha256 -binary | base64

let pinnedPublicKeyHash = "sha256/BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB="

// In URLSessionDelegate
func urlSession(_ session: URLSession,
                didReceive challenge: URLAuthenticationChallenge,
                completionHandler: @escaping (URLSession.AuthChallengeDisposition, URLCredential?) -> Void) {

    guard let serverCert = challenge.protectionSpace.serverTrust,
          let publicKey = SecTrustCopyKey(serverCert) else {
        completionHandler(.cancelAuthenticationChallenge, nil)
        return
    }

    // Hash the server's public key and compare
    let serverKeyHash = hashPublicKey(publicKey)

    if serverKeyHash == pinnedPublicKeyHash {
        completionHandler(.useCredential, URLCredential(trust: serverCert))
    } else {
        // Certificate mismatch — possible MITM attack!
        completionHandler(.cancelAuthenticationChallenge, nil)
    }
}
```

**Android (OkHttp):**
```kotlin
val client = OkHttpClient.Builder()
    .certificatePinner(
        CertificatePinner.Builder()
            .add("api.myapp.com", "sha256/BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB=")
            .add("api.myapp.com", "sha256/BACKUP_KEY_HASH==")  // Always pin backup!
            .build()
    )
    .build()
```

**Certificate Pinning Risks and Best Practices:**
```
Risk 1: App breaks if pin expires / key rotated without app update
  → Always pin at least 2 keys (current + backup)
  → Backup key: generate a new key pair, keep it secure, use its hash as backup pin

Risk 2: Hard to debug (connection silently fails)
  → Implement clear error messages for pin failures
  → Log pin failures to your monitoring system

Risk 3: Dynamic pinning bypassed by attackers
  → Don't fetch pins from a server (defeats the purpose!)
  → Pins must be baked into the app binary

When to use:
  ✅ High-security mobile apps (banking, healthcare, financial)
  ✅ IoT devices with long lifetimes
  ✅ Apps in regulated industries
  ❌ Regular websites (use HSTS instead — pinning in browsers was deprecated)
  ❌ Apps with frequent backend cert changes
```

---

### Other HTTPS Enforcement Measures

**Redirect HTTP to HTTPS (always):**
```nginx
# Nginx — redirect all HTTP to HTTPS
server {
    listen 80;
    return 301 https://$host$request_uri;
}
```

```javascript
// Express.js middleware
app.use((req, res, next) => {
  if (req.headers['x-forwarded-proto'] !== 'https') {
    return res.redirect(301, `https://${req.hostname}${req.url}`);
  }
  next();
});
```

**Security headers that complement HTTPS:**
```
Content-Security-Policy (CSP):
  → Defines what resources can be loaded (scripts, images, styles)
  → Prevents XSS by blocking inline scripts and unauthorized sources
  Content-Security-Policy: default-src 'self'; script-src 'self' cdn.example.com

X-Content-Type-Options: nosniff
  → Prevents browser from MIME-sniffing (guessing file type)
  → Forces browser to use declared Content-Type
  → Prevents XSS via uploaded files

X-Frame-Options: DENY
  → Prevents your page from being embedded in an iframe
  → Protects against clickjacking attacks

Referrer-Policy: strict-origin-when-cross-origin
  → Controls how much URL info is sent in the Referer header
  → Prevents leaking sensitive URL parameters to third parties

Permissions-Policy: camera=(), microphone=(), geolocation=()
  → Disables browser features you don't use
  → Reduces attack surface if XSS occurs
```

**Check your security headers:**
```
https://securityheaders.com — scan your site and get a grade
https://www.ssllabs.com/ssltest/ — test your TLS configuration
```

---

### Interview-Ready Summary

> "Just having HTTPS isn't enough — users can still make that initial HTTP request which can be intercepted. HSTS (HTTP Strict Transport Security) solves this with the `Strict-Transport-Security` header — it tells browsers to always use HTTPS for your domain for `max-age` seconds, upgrading HTTP requests internally before they leave the browser. Adding `includeSubDomains` extends protection to all subdomains, and `preload` gets you included in browsers' built-in lists so even the very first visit is protected. Start with a short max-age and gradually increase it — it's hard to undo. Certificate pinning takes it further for mobile apps — the app hard-codes the expected public key hash and rejects connections if the server presents anything different, protecting against compromised CAs. Always pin at least two keys (current + backup) to avoid locking users out during key rotation. For websites, pair HSTS with a CSP header, X-Frame-Options, and X-Content-Type-Options for defense in depth."
