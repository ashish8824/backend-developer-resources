## 1. OSI & TCP/IP Models

### What is it?

When two computers talk to each other over a network, a LOT of things happen behind the scenes — your data needs to be broken into packets, addressed, routed, checked for errors, and finally reassembled. The **OSI model** and **TCP/IP model** are frameworks that organize all these steps into clean **layers**, each with a specific job.

Think of it like sending a physical parcel:
> You write the letter → put it in an envelope → address it → hand it to the courier → it travels through sorting centers → arrives at the destination → gets opened → read.

Each step is a "layer."

---

### The OSI Model (7 Layers)

OSI stands for **Open Systems Interconnection**. It has 7 layers. Think of it as the **theoretical reference model** — it's mostly used for learning and troubleshooting.

```
+-----+-----------------------+---------------------------------------------+
| No. | Layer Name            | Job                                         |
+-----+-----------------------+---------------------------------------------+
|  7  | Application Layer     | What the user/app sees (HTTP, FTP, SMTP)    |
|  6  | Presentation Layer    | Data format, encryption, compression        |
|  5  | Session Layer         | Manages sessions/connections                |
|  4  | Transport Layer       | Reliable delivery, ports (TCP/UDP)          |
|  3  | Network Layer         | Routing, IP addresses                       |
|  2  | Data Link Layer       | MAC addresses, frames, switches             |
|  1  | Physical Layer        | Raw bits, cables, Wi-Fi signals             |
+-----+-----------------------+---------------------------------------------+
```

**Easy way to remember (top to bottom):**
> **"All People Seem To Need Data Processing"**
> Application → Presentation → Session → Transport → Network → Data Link → Physical

---

### Deep Dive: Each OSI Layer

**Layer 7 — Application Layer**
- This is where your apps live.
- Protocols: `HTTP`, `HTTPS`, `FTP`, `SMTP`, `WebSocket`
- Example: When you open `https://google.com` in your browser, the browser uses HTTP at this layer to form the request.

**Layer 6 — Presentation Layer**
- Converts data into a format both sides understand.
- Handles encryption (TLS/SSL), compression, encoding (UTF-8, ASCII).
- Example: When HTTPS encrypts your password before sending, that's this layer.

**Layer 5 — Session Layer**
- Manages start, maintain, and end of communication sessions.
- Example: Logging into a website creates a session. This layer manages it.

**Layer 4 — Transport Layer**
- Breaks data into segments and ensures they reach the destination reliably.
- Key protocols: **TCP** (reliable) and **UDP** (fast but unreliable).
- Introduces **port numbers**.
- Example: When you download a file, TCP ensures every chunk arrives and reassembles correctly.

**Layer 3 — Network Layer**
- Handles routing of data from source to destination across networks.
- Uses **IP addresses**.
- Key protocol: **IP (Internet Protocol)**, also routers work here.
- Example: Your request to Google travels through multiple routers. Each router uses this layer to decide where to forward it next.

**Layer 2 — Data Link Layer**
- Handles communication within a local network (between devices on the same network).
- Uses **MAC addresses**.
- Example: Your laptop sends data to your home router using MAC addresses. Switches work at this layer.

**Layer 1 — Physical Layer**
- Raw transmission of bits (0s and 1s) over physical medium.
- Cables (Ethernet), Wi-Fi signals, fiber optics.
- Example: The actual electrical pulses sent over a network cable.

---

### The TCP/IP Model (4 Layers)

The TCP/IP model is the **practical, real-world model** used on the Internet. It collapses the 7 OSI layers into 4.

```
+----------------------------+-------------------------------------------+
| TCP/IP Layer               | Maps to OSI Layers                        |
+----------------------------+-------------------------------------------+
| Application Layer          | OSI 5 + 6 + 7 (Session+Presentation+App) |
| Transport Layer            | OSI 4 (TCP / UDP)                         |
| Internet Layer             | OSI 3 (IP, Routing)                       |
| Network Access Layer       | OSI 1 + 2 (Physical + Data Link)          |
+----------------------------+-------------------------------------------+
```

---

### How Data Flows: Encapsulation & Decapsulation

**Sending side (Encapsulation) — data goes DOWN the layers:**

```
Your Message: "Hello Google!"
       ↓ Application Layer adds HTTP headers
[HTTP Request: GET /]
       ↓ Transport Layer adds TCP header (port 80, sequence number)
[TCP Header | HTTP Request]
       ↓ Network Layer adds IP header (source IP, destination IP)
[IP Header | TCP Header | HTTP Request]
       ↓ Data Link Layer adds MAC address frame
[MAC Frame | IP Header | TCP Header | HTTP Request]
       ↓ Physical Layer converts to bits
01001010101010101...
```

**Receiving side (Decapsulation) — data goes UP the layers:**
Each layer strips its header and hands the remaining data up to the next layer.

---

### Interview-Ready Summary

> "The OSI model is a 7-layer theoretical framework for understanding how network communication works. Each layer has a specific responsibility — from physical bit transmission at Layer 1 to user-facing apps at Layer 7. The TCP/IP model simplifies this into 4 layers and is what the actual Internet is built on. When data is sent, each layer wraps data with its own header (encapsulation), and when received, each layer unwraps it (decapsulation)."

## 2. IP Addressing

### What is an IP Address?

An **IP (Internet Protocol) address** is a unique identifier assigned to every device on a network — just like a home address tells the postal service where to deliver a letter, an IP address tells the network where to send data.

**Real-world analogy:**
> Every house on a street has a unique address. On the Internet, every device (laptop, phone, server) has a unique IP address so data knows where to go.

---

### IPv4

**IPv4** is the most widely used IP version. It's a **32-bit number** written as four groups of numbers separated by dots.

```
Example: 192.168.1.25

Breakdown:
192  .  168  .  1  .  25
 ↑        ↑      ↑    ↑
8 bits  8 bits  8 bits  8 bits  = 32 bits total
```

Each group (called an **octet**) ranges from **0 to 255**.

Total possible IPv4 addresses: **~4.3 billion** (2³²)

---

### IPv6

Because the Internet grew massively, IPv4 addresses started running out. **IPv6** was created with **128 bits**, giving us astronomically more addresses.

```
Example: 2001:0db8:85a3:0000:0000:8a2e:0370:7334

Written in 8 groups of 4 hexadecimal digits separated by colons.

Shorthand: 2001:db8:85a3::8a2e:370:7334
(consecutive zeros can be replaced with ::, but only once)
```

Total possible IPv6 addresses: **~340 undecillion** (3.4 × 10³⁸) — practically unlimited.

**Key difference:**
| Feature | IPv4 | IPv6 |
|---------|------|------|
| Length | 32-bit | 128-bit |
| Format | Dotted decimal (192.168.1.1) | Hexadecimal (2001:db8::1) |
| Total addresses | ~4.3 billion | ~340 undecillion |
| Header size | 20 bytes | 40 bytes (simpler, fixed) |
| NAT needed? | Yes (due to shortage) | No |

---

### Public vs Private IP Addresses

Not all IP addresses are for the Internet. Some are reserved for **private networks** (your home or office LAN).

**Private IP ranges (reserved, not routable on the Internet):**
```
Class A:  10.0.0.0    –  10.255.255.255    (for large networks)
Class B:  172.16.0.0  –  172.31.255.255    (medium networks)
Class C:  192.168.0.0 –  192.168.255.255   (home/small office)
```

**Public IPs** are globally unique and assigned by ISPs. Your home router has one public IP facing the Internet.

**Analogy:**
> Private IPs are like room numbers inside an office building — they're only meaningful inside the building. The public IP is the building's street address that the outside world uses.

---

### Subnets & CIDR Notation

#### What is a Subnet?

A **subnet** (sub-network) divides a large network into smaller, manageable parts. It's like splitting a large apartment complex into floors — each floor is a subnet.

#### Why Subnet?
- Better security (isolate departments)
- Better performance (less broadcast traffic)
- Efficient IP allocation

#### Subnet Mask

A **subnet mask** tells you which part of an IP address is the **network** and which part is the **host**.

```
IP Address:    192.168.1.25
Subnet Mask:   255.255.255.0

Network part:  192.168.1   (first 3 octets — identifies the network)
Host part:     .25         (last octet — identifies the device)
```

All devices with `192.168.1.x` are on the same network.

#### CIDR Notation (Classless Inter-Domain Routing)

CIDR is a **shorthand** way to write the subnet mask using a slash and the number of network bits.

```
192.168.1.0/24

/24 means: first 24 bits are the network part
         = subnet mask of 255.255.255.0
         = 256 total addresses (0–255)
         = 254 usable host addresses (2 are reserved)
```

**Common CIDR examples:**
```
/32  → 1 address        (single host, e.g., a specific server)
/30  → 4 addresses      (2 usable — point-to-point links)
/24  → 256 addresses    (254 usable — typical home/small network)
/16  → 65,536 addresses (65,534 usable — large corporate network)
/8   → 16.7 million     (huge networks)
```

**Quick formula:**
```
Usable hosts = 2^(32 - prefix) - 2
For /24: 2^(32-24) - 2 = 2^8 - 2 = 256 - 2 = 254 hosts
```
*(The 2 reserved addresses are: network address and broadcast address)*

**Real-world example:**
```
Your AWS VPC:     10.0.0.0/16    (65,534 IPs for the whole VPC)
  ├── Subnet A:   10.0.1.0/24    (254 IPs for web servers)
  ├── Subnet B:   10.0.2.0/24    (254 IPs for app servers)
  └── Subnet C:   10.0.3.0/24    (254 IPs for databases)
```

---

### Special IP Addresses

```
127.0.0.1         → Loopback / localhost (refers to your own machine)
0.0.0.0           → All interfaces / any IP (used in server binding)
255.255.255.255   → Broadcast (send to all devices on the network)
```

---

### Interview-Ready Summary

> "An IP address is a unique identifier for a device on a network. IPv4 uses 32-bit addresses like `192.168.1.1`, while IPv6 uses 128-bit addresses to solve IPv4 exhaustion. Private IPs (like `192.168.x.x`) are used inside local networks and aren't routable on the Internet. CIDR notation like `/24` defines how many bits represent the network — `192.168.1.0/24` means 254 usable host addresses. In cloud setups like AWS, we use subnets to isolate different layers — web, app, and database."

## 3. DNS — Domain Name System

### What is DNS?

**DNS** is the Internet's **phone book**. Humans remember names like `google.com`, but computers communicate using IP addresses like `142.250.80.46`. DNS translates domain names into IP addresses.

**Analogy:**
> You want to call your friend "Alice". You don't memorize her phone number — you search her name in your contacts and it finds the number. DNS does the same: you type `google.com`, DNS finds its IP address `142.250.80.46`, and your browser connects to it.

---

### How DNS Resolution Works (Step by Step)

When you type `www.google.com` in your browser:

```
Step 1: Browser Cache
  → "Have I visited google.com recently? Do I know its IP?"
  → If yes: use cached IP. If no: ask OS.

Step 2: OS Cache / hosts file
  → OS checks /etc/hosts file and its own DNS cache.
  → If found: use it. If not: ask the Recursive Resolver.

Step 3: Recursive Resolver (your ISP or 8.8.8.8)
  → This is the DNS server configured on your device (e.g., Google's 8.8.8.8).
  → It does the heavy lifting of finding the answer.

Step 4: Root Name Server
  → Recursive resolver asks: "Who handles .com domains?"
  → Root server replies: "Ask the .com TLD server."

Step 5: TLD (Top-Level Domain) Name Server
  → Resolver asks: "Who handles google.com?"
  → TLD server replies: "Ask Google's authoritative name server."

Step 6: Authoritative Name Server
  → Resolver asks Google's DNS: "What is the IP of www.google.com?"
  → Google's DNS replies: "142.250.80.46"

Step 7: Response
  → Recursive resolver returns IP to your browser.
  → Browser caches it (for TTL duration) and connects to 142.250.80.46.
```

**Visual flow:**
```
Browser → Recursive Resolver → Root NS → TLD NS → Authoritative NS
                ↑_______________________________________________↓
                              IP Address returned
```

---

### DNS Record Types

DNS uses **records** stored in a zone file to answer queries. Each record type serves a different purpose.

#### A Record (Address Record)
- Maps a domain to an **IPv4** address.
```
google.com.    300    IN    A    142.250.80.46
```
- `300` is the TTL (in seconds).
- When you ping `google.com`, an A record is what gets resolved.

#### AAAA Record
- Maps a domain to an **IPv6** address.
```
google.com.    300    IN    AAAA    2607:f8b0:4004:c09::6a
```

#### CNAME Record (Canonical Name)
- Creates an **alias** — maps one domain name to another domain name (not an IP directly).
```
www.example.com.    300    IN    CNAME    example.com.
blog.example.com.   300    IN    CNAME    myblog.wordpress.com.
```
- Useful for: CDN setups, pointing subdomains to external services.
- **Important:** CNAME cannot be set on the root domain (you can't CNAME `example.com` itself).

#### MX Record (Mail Exchange)
- Specifies the **mail server** responsible for receiving emails for a domain.
```
example.com.    300    IN    MX    10    mail.example.com.
example.com.    300    IN    MX    20    mail2.example.com.
```
- The number (`10`, `20`) is **priority** — lower means higher priority.
- When someone sends email to `user@example.com`, the MX record tells where to deliver it.

#### TXT Record (Text Record)
- Stores arbitrary **text data**. Used for verification, anti-spam, etc.
```
example.com.  300  IN  TXT  "v=spf1 include:_spf.google.com ~all"
example.com.  300  IN  TXT  "google-site-verification=abc123xyz"
```
- Common uses:
  - **SPF** (Sender Policy Framework) — which servers can send email for your domain
  - **DKIM** — email signature verification
  - **Domain ownership verification** for Google, AWS, etc.

#### NS Record (Name Server)
- Specifies the **authoritative name servers** for the domain.
```
example.com.    86400    IN    NS    ns1.example.com.
example.com.    86400    IN    NS    ns2.example.com.
```

#### PTR Record (Pointer Record)
- **Reverse DNS** — maps an IP address back to a domain name.
```
46.80.250.142.in-addr.arpa.    300    IN    PTR    google.com.
```
- Used in email security (mail servers verify sender IP maps back to a domain).

---

### TTL (Time to Live)

**TTL** is the number of **seconds** a DNS response should be cached before it's considered stale and needs to be re-queried.

```
example.com.    300    IN    A    93.184.216.34
                 ↑
                TTL = 300 seconds (5 minutes)
```

**Low TTL (e.g., 60 seconds):**
- DNS changes propagate quickly.
- More DNS queries → slightly higher latency.
- Use when: planning IP changes, deployments, failovers.

**High TTL (e.g., 86400 = 24 hours):**
- Faster responses (cached longer).
- Changes take longer to propagate.
- Use when: stable infrastructure, rarely changing IPs.

**Best practice:**
> Before a planned migration, lower TTL to 300 seconds a day in advance. This lets caches expire quickly. After migration, raise TTL back up.

---

### DNS in Backend Development

As a backend developer you'll deal with DNS in:

```
1. Custom domains for your API     → A or CNAME records
2. Email sending setup             → MX + SPF + DKIM TXT records
3. Load balancer DNS               → CNAME to AWS ELB / CloudFront
4. SSL certificate verification    → TXT record for Let's Encrypt / ACM
5. Service discovery               → Internal DNS in Kubernetes / AWS VPC
6. Debugging slow requests         → Check DNS resolution time with dig
```

**Useful commands:**
```bash
# Look up A record
dig google.com

# Look up specific record type
dig google.com MX
dig google.com TXT

# Trace full DNS resolution path
dig google.com +trace

# Check what IP a domain resolves to (simple)
nslookup google.com

# Check DNS propagation using a specific DNS server
dig @8.8.8.8 google.com
```

---

### Interview-Ready Summary

> "DNS is the Internet's phone book — it translates human-readable domain names like `google.com` into IP addresses. Resolution works through a chain: browser cache → OS cache → recursive resolver → root server → TLD server → authoritative server. Key record types are: **A** (domain to IPv4), **CNAME** (alias to another domain), **MX** (email routing), **TXT** (verification and anti-spam), and **NS** (authoritative name servers). TTL controls how long responses are cached. As a backend dev, I often configure A/CNAME records for APIs, TXT records for domain verification, and MX records for email."

## 4. Ports & Sockets

### What is a Port?

An IP address identifies a **device** on a network. But a single device runs many services simultaneously — a web server, a database, an SSH server, etc. A **port** identifies which specific **service** (process) on that device the data should go to.

**Analogy:**
> An IP address is like a building's street address. A port is like the apartment number inside that building. The postman (network) delivers to the building (IP), and then the specific apartment (port) receives it.

```
192.168.1.10 : 80   → Web server (HTTP)
192.168.1.10 : 443  → Web server (HTTPS)
192.168.1.10 : 5432 → PostgreSQL database
192.168.1.10 : 22   → SSH
```

Ports are **16-bit numbers** ranging from **0 to 65535**.

---

### Port Categories

**1. Well-Known Ports (0–1023)**
Reserved for standard services. Requires root/admin to bind on Linux.

```
Port  20/21  → FTP (File Transfer Protocol) — data/control
Port  22     → SSH (Secure Shell)
Port  25     → SMTP (Email sending)
Port  53     → DNS
Port  80     → HTTP
Port  443    → HTTPS
Port  3306   → MySQL
Port  5432   → PostgreSQL
Port  6379   → Redis
Port  27017  → MongoDB
```

**2. Registered Ports (1024–49151)**
Used by applications and middleware. Not reserved but widely recognized.

**3. Dynamic/Ephemeral Ports (49152–65535)**
Assigned **temporarily** by the OS to the **client side** of a connection. These are discarded after the connection closes.

```
Example: When YOUR browser connects to google.com:80
  - Google's server listens on port 80 (well-known)
  - Your OS assigns YOUR browser a random ephemeral port (e.g., 54231)
  - Connection: your_ip:54231 → google_ip:80
```

---

### What is a Socket?

A **socket** is a combination of an IP address + port number that uniquely identifies a communication **endpoint**.

```
Socket = IP Address + Port Number

Example:
  Client socket: 192.168.1.5 : 54231
  Server socket: 142.250.80.46 : 443
```

A **connection** (TCP) is defined by a **4-tuple**:
```
(Source IP, Source Port, Destination IP, Destination Port)

Example:
(192.168.1.5, 54231, 142.250.80.46, 443)
```

This means a single server on port 443 can handle **millions of simultaneous connections** — each has a unique 4-tuple.

---

### Binding, Listening, Accepting (Server Socket Lifecycle)

Here's what happens on the **server side** when it starts up:

```
1. CREATE socket
   → OS creates a socket (file descriptor)

2. BIND to an address and port
   → "I want to receive traffic on 0.0.0.0:8080"
   → 0.0.0.0 means "listen on all available network interfaces"

3. LISTEN
   → "I'm ready to accept connections"
   → OS starts queuing incoming connection requests

4. ACCEPT (in a loop)
   → For each incoming client, accept() returns a NEW socket
   → The original socket keeps listening; the new socket handles the client

5. READ / WRITE
   → Exchange data with the client over the accepted socket

6. CLOSE
   → Terminate the connection
```

**Code example (Node.js — to see the concept):**
```javascript
const net = require('net');

// Create server (steps 1-3 combined)
const server = net.createServer((socket) => {   // step 4: accept
  console.log('Client connected:', socket.remoteAddress, socket.remotePort);

  socket.on('data', (data) => {                 // step 5: read
    console.log('Received:', data.toString());
    socket.write('Hello from server!\n');       // step 5: write
  });

  socket.on('end', () => {
    console.log('Client disconnected');         // step 6: close
  });
});

server.listen(8080, '0.0.0.0', () => {         // step 2+3: bind + listen
  console.log('Server listening on port 8080');
});
```

---

### TCP vs UDP Socket Behavior

**TCP Socket (Reliable, Connection-Oriented)**
```
Client                          Server
  |                               |
  |------- SYN ------------------>|   Step 1: Client initiates
  |<------ SYN-ACK ---------------|   Step 2: Server acknowledges
  |------- ACK ------------------>|   Step 3: Connection established
  |                               |   (This is the TCP 3-way handshake)
  |===== DATA EXCHANGE ========== |
  |------- FIN ------------------>|   Graceful connection close
  |<------ FIN-ACK ---------------|
```
- Data arrives in **order**, **no loss**, **error-checked**.
- Slightly more overhead due to handshake and acknowledgments.
- Used for: HTTP, HTTPS, databases, SSH, email.

**UDP Socket (Fast, Connectionless)**
```
Client                          Server
  |                               |
  |--- Data Packet 1 ------------>|   Sent — no guarantee of delivery
  |--- Data Packet 2 ------------>|   May arrive out of order
  |--- Data Packet 3 ------------>|   No acknowledgment
```
- **No handshake**, no reliability guarantee.
- Faster, lower latency.
- Used for: DNS, video streaming, gaming, VoIP.

---

### Common Backend Scenarios

**Checking what's running on which port:**
```bash
# List all open ports and the process using them
lsof -i -P -n | grep LISTEN

# Or with netstat
netstat -tulnp

# Check if port 8080 is in use
lsof -i :8080
```

**Connecting to a remote socket manually:**
```bash
# Connect to a server's port (like a client)
telnet google.com 80

# Or with netcat (more powerful)
nc google.com 80
nc -l 9090  # Listen on port 9090 (act as server)
```

**Firewall / Security Groups:**
- In cloud environments (AWS, GCP), you control which ports are open via **Security Groups** or **firewall rules**.
- As a backend dev, you often need to **open** a port for your API (e.g., 443 for HTTPS) and **close** everything else.

---

### Socket Programming Concepts

| Concept | Meaning |
|---------|---------|
| **Binding** | Assigning an IP+port to a socket (server side) |
| **Listening** | Putting the socket in a state ready to accept connections |
| **Accepting** | Taking an incoming connection from the queue |
| **Connecting** | Client initiating a connection to a server |
| **Blocking socket** | Operations wait (block the thread) until complete |
| **Non-blocking socket** | Operations return immediately; check later for results |
| **Socket timeout** | How long to wait before giving up on an operation |
| **Connection pooling** | Reusing existing sockets instead of creating new ones (for performance) |

---

### Interview-Ready Summary

> "A port is a 16-bit number (0–65535) that identifies a specific service on a device. Well-known ports (0–1023) are for standard services — HTTP is 80, HTTPS is 443, SSH is 22. A socket is the combination of IP + port that defines a communication endpoint. A TCP connection is uniquely identified by the 4-tuple: source IP, source port, destination IP, destination port. On the server side, the lifecycle is: create → bind → listen → accept → read/write → close. TCP is reliable and ordered (used for web traffic, databases), while UDP is faster but unreliable (used for DNS, video streaming, gaming)."
