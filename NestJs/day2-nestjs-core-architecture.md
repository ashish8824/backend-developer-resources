## 1. What is NestJS — The Real Answer

### Simple explanation
NestJS is a framework for building Node.js backend applications. It sits on top of Express (or Fastify) and adds structure, conventions, and powerful features that Express alone does not have.

### Technical explanation

Express is **unopinionated** — it gives you routing and middleware, and that's it. As your app grows, teams end up inventing their own folder structures, patterns, and conventions. This leads to inconsistent codebases that are hard to maintain.

NestJS is **opinionated** — it enforces a specific architecture inspired by **Angular** (same author — Kamil Myśliwiec) and heavily influenced by enterprise patterns from the Java/Spring world:

```
Problem Express has:              NestJS solution:
─────────────────────────────────────────────────────────
No standard folder structure   →  Module-based architecture
No built-in DI container       →  Full dependency injection
No type safety by default      →  TypeScript first-class
No built-in validation         →  Pipes + class-validator
Hard to unit test              →  DI makes mocking easy
No standard for auth, guards   →  Guards, Interceptors, Filters
```

### What NestJS actually IS under the hood

```
                    Your NestJS Code
                         │
                    NestJS Framework
                    ┌────┴────┐
              IoC Container   Request Pipeline
              (Dependency     (Middleware → Guards
               Injection)      → Interceptors → Pipes
                               → Controllers → Services)
                         │
              HTTP Adapter (Express or Fastify)
                         │
              Node.js (libuv + V8)
                         │
              Operating System
```

NestJS is essentially three things:
1. An **IoC (Inversion of Control) container** — manages object creation and dependencies
2. A **request pipeline** — processes HTTP requests through a series of layers
3. A **module system** — organizes code into feature-based units

### Why companies use NestJS for 3+ year engineers

In teams with multiple developers, you need:
- **Predictability** — any developer can open any module and know where to find things
- **Testability** — DI makes it easy to swap real services with mocks in tests
- **Scalability** — modules can be extracted into microservices when needed
- **TypeScript** — catches bugs at compile time, not at 3am in production

## 2. Installing NestJS CLI & Project Structure

### Installation

```bash
# Install the NestJS CLI globally
npm install -g @nestjs/cli

# Create a new project — CLI prompts for package manager
nest new my-app
# Choose: npm / yarn / pnpm

# Move into the project
cd my-app

# Start development server
npm run start:dev
# App runs on http://localhost:3000
```

## 3. The main.ts Bootstrap File

### What and why
`main.ts` is the **entry point** of your application. It creates the NestJS application, configures it, and starts the HTTP server. This file runs once when your app starts.

```typescript
// src/main.ts
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module';
import { ValidationPipe } from '@nestjs/common';

async function bootstrap() {
  // ─── Step 1: Create the NestJS application ──────────────────────────
  // NestFactory.create() does several things:
  // 1. Creates the IoC container (dependency injection container)
  // 2. Scans AppModule and all its imports recursively
  // 3. Instantiates all providers (services, repositories, etc.)
  // 4. Wires dependencies together
  // 5. Creates the HTTP adapter (Express by default)
  const app = await NestFactory.create(AppModule);
  // At this point, ALL your services are already created and ready

  // ─── Step 2: Global configuration ──────────────────────────────────

  // Global API prefix — all routes become /api/v1/...
  // Instead of: /users → /api/v1/users
  app.setGlobalPrefix('api/v1');

  // Global validation pipe — automatically validates all incoming DTOs
  // We cover this deeply on Day 4, but always set this up
  app.useGlobalPipes(
    new ValidationPipe({
      whitelist: true,     // Strip properties not in DTO
      forbidNonWhitelisted: true,  // Throw error if extra props sent
      transform: true,     // Auto-convert strings to numbers/booleans
    }),
  );

  // CORS — allow cross-origin requests (needed for frontend on different domain)
  app.enableCors({
    origin: process.env.ALLOWED_ORIGINS?.split(',') || '*',
    methods: ['GET', 'POST', 'PUT', 'PATCH', 'DELETE', 'OPTIONS'],
    credentials: true,  // Allow cookies to be sent
  });

  // ─── Step 3: Start the HTTP server ──────────────────────────────────
  const PORT = process.env.PORT || 3000;
  await app.listen(PORT);
  // app.listen() starts the underlying Express/Fastify server
  // Returns a Promise — must be awaited
  
  console.log(`Application running on: http://localhost:${PORT}/api/v1`);
}

// Call the bootstrap function — this starts the whole app
bootstrap();
// bootstrap() returns a Promise — if not handled, Node.js silently swallows errors
// In production, you'd want: bootstrap().catch(console.error)
```

### How NestFactory.create() works internally

```typescript
// NestFactory.create(AppModule) conceptually does this:

// 1. Creates a module scanner
const scanner = new DependencyScanner();

// 2. Recursively scans all modules starting from AppModule
// Finds all controllers, providers, imports, exports
const graph = scanner.scan(AppModule);

// 3. Creates the DI container
const container = new NestContainer();

// 4. Instantiates all providers in dependency order
// If ServiceA depends on ServiceB, ServiceB is created first
container.instantiateAll(graph);

// 5. Creates HTTP adapter (Express by default)
const httpAdapter = new ExpressAdapter();
httpAdapter.init();

// 6. Returns the application instance
return new NestApplication(container, httpAdapter);
```

## 4. Modules — The Building Blocks

### What is a module?
A module is a **class decorated with `@Module()`** that groups related functionality together. Think of it as a self-contained feature unit — all code for "users" lives in the users module.

### Simple analogy
Think of your app as a large company:
- The company itself = AppModule (root module)
- Departments = Feature modules (UsersModule, ProductsModule, OrdersModule)
- Employees in departments = Services and Controllers
- Shared tools = SharedModule (utilities, helpers used by all departments)

### The @Module() decorator — every property explained

```typescript
// src/users/users.module.ts
import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { UsersController } from './users.controller';
import { UsersService } from './users.service';
import { EmailService } from '../email/email.service';
import { User } from './entities/user.entity';

@Module({
  // ─── imports ────────────────────────────────────────────────────────
  imports: [
    TypeOrmModule.forFeature([User]),
    // Makes Repository<User> available for injection in this module
    // forFeature() registers entity-specific providers
    
    // You can also import other feature modules
    // EmailModule,
    // This makes everything EmailModule EXPORTS available here
  ],
  // imports: Which OTHER modules' exported providers do you need?
  // The providers those modules export become available in THIS module

  // ─── controllers ────────────────────────────────────────────────────
  controllers: [UsersController],
  // List all controllers that belong to this module
  // NestJS registers their routes with the HTTP adapter
  // Controllers are NOT injectable — they just handle routes

  // ─── providers ──────────────────────────────────────────────────────
  providers: [
    UsersService,
    // Shorthand for: { provide: UsersService, useClass: UsersService }
    // NestJS creates ONE instance of UsersService for this module
    // That same instance is injected wherever UsersService is requested
    
    // You can also provide custom values:
    {
      provide: 'CONFIG',                    // Token (string or class)
      useValue: { maxUsers: 100 },          // The actual value to inject
    },
    {
      provide: 'MAILER',
      useFactory: () => new EmailService(), // Factory function — called to create provider
    },
  ],
  // providers: What services/helpers exist in this module?
  // Providers listed here are available to controllers AND other providers in this module
  // They are NOT available outside this module unless exported

  // ─── exports ────────────────────────────────────────────────────────
  exports: [
    UsersService,
    // Make UsersService available to OTHER modules that import UsersModule
    // If not exported, UsersService is PRIVATE to this module
  ],
  // exports: What do you want to share with other modules?
  // Think of this as your module's public API
})
export class UsersModule {}
```

### The Root Module — AppModule

```typescript
// src/app.module.ts
// The ROOT module — the top of the entire module tree
// Everything your app needs is imported here (directly or transitively)

import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { ConfigModule } from '@nestjs/config';
import { UsersModule } from './users/users.module';
import { ProductsModule } from './products/products.module';
import { AuthModule } from './auth/auth.module';
import { AppController } from './app.controller';
import { AppService } from './app.service';

@Module({
  imports: [
    // ── Global configuration ──────────────────────────────────────────
    ConfigModule.forRoot({
      isGlobal: true,
      // isGlobal: true means you don't need to import ConfigModule
      // in every feature module — it's available everywhere automatically
      envFilePath: '.env',
    }),

    // ── Database connection ───────────────────────────────────────────
    TypeOrmModule.forRoot({
      type: 'postgres',
      host: process.env.DB_HOST,
      port: parseInt(process.env.DB_PORT),
      username: process.env.DB_USER,
      password: process.env.DB_PASS,
      database: process.env.DB_NAME,
      autoLoadEntities: true,  // Auto-discover entities registered with forFeature()
      synchronize: false,       // NEVER true in production — use migrations
    }),

    // ── Feature modules ───────────────────────────────────────────────
    UsersModule,
    ProductsModule,
    AuthModule,
    // Each module encapsulates its own routes, services, and DB entities
  ],
  controllers: [AppController],  // Only the root-level controller (handles / route)
  providers: [AppService],
})
export class AppModule {}
```

### Global vs Feature modules

```typescript
// ── GLOBAL MODULE ─────────────────────────────────────────────────────
// Available everywhere without importing — use sparingly
// Good for: config, logging, database connection

import { Global, Module } from '@nestjs/common';

@Global()   // ← This makes the module global
@Module({
  providers: [LoggerService, ConfigService],
  exports: [LoggerService, ConfigService],
  // Must still export — @Global just removes the need to import the module
})
export class CoreModule {}

// Any module can now inject LoggerService without importing CoreModule


// ── FEATURE MODULE ────────────────────────────────────────────────────
// Standard — must be imported to use its exports

@Module({
  providers: [UsersService],
  exports: [UsersService],
  controllers: [UsersController],
})
export class UsersModule {}

// To use UsersService in OrdersModule:
@Module({
  imports: [UsersModule],  // Must import it
  providers: [OrdersService],
})
export class OrdersModule {}
// Now OrdersService can inject UsersService


// ── DYNAMIC MODULE ────────────────────────────────────────────────────
// Modules that accept configuration — used by libraries like TypeORM, Config

// How TypeORM.forRoot() works (simplified):
@Module({})
class TypeOrmModule {
  static forRoot(options: TypeOrmModuleOptions): DynamicModule {
    return {
      module: TypeOrmModule,
      providers: [
        {
          provide: 'TYPEORM_OPTIONS',
          useValue: options,  // Pass config into providers
        },
        DataSourceProvider,  // Creates DB connection using the options
      ],
      exports: [DataSourceProvider],
      global: true,  // Make DataSource available app-wide
    };
  }

  // forFeature() registers entity-specific repositories
  static forFeature(entities: EntityClass[]): DynamicModule {
    return {
      module: TypeOrmModule,
      providers: entities.map(entity => ({
        provide: getRepositoryToken(entity),  // Token like 'UserRepository'
        useFactory: (dataSource: DataSource) =>
          dataSource.getRepository(entity),
        inject: [DataSource],
      })),
      exports: entities.map(entity => getRepositoryToken(entity)),
    };
  }
}
```

### Module re-exports — sharing imported modules

```typescript
// Scenario: Multiple modules need HttpModule
// Instead of importing it everywhere, create a SharedModule

@Module({
  imports: [
    HttpModule,      // Import it here
    ConfigModule,
  ],
  exports: [
    HttpModule,      // Re-export it — any module importing SharedModule
    ConfigModule,    // gets HttpModule and ConfigModule too
  ],
})
export class SharedModule {}

// Now in any feature module:
@Module({
  imports: [SharedModule],  // Gets HttpModule AND ConfigModule for free
})
export class UsersModule {}
```

## 5. Controllers — Handling HTTP Requests

### What is a controller?
A controller is responsible for **receiving HTTP requests and returning responses**. It handles routing only — business logic belongs in services.

### Why controllers exist separately from services
This is the **Single Responsibility Principle** in action:
- Controller knows about HTTP (request, response, status codes, routes)
- Service knows about business logic (validation rules, data transformation)
- This makes services **reusable** — the same service can be used by a controller, a CLI command, a WebSocket gateway, or a test

```typescript
// src/users/users.controller.ts
import {
  Controller,
  Get,
  Post,
  Put,
  Patch,
  Delete,
  Body,
  Param,
  Query,
  HttpCode,
  HttpStatus,
  ParseIntPipe,
  UseGuards,
  NotFoundException,
} from '@nestjs/common';
import { UsersService } from './users.service';
import { CreateUserDto } from './dto/create-user.dto';
import { UpdateUserDto } from './dto/update-user.dto';

// @Controller('users') sets the ROUTE PREFIX for all methods in this class
// All routes in this controller start with /users
// Combined with app.setGlobalPrefix('api/v1'): /api/v1/users
@Controller('users')
export class UsersController {
  // Dependency injection via constructor
  // NestJS automatically injects UsersService here
  constructor(private readonly usersService: UsersService) {
    // 'private readonly' is a TypeScript shorthand:
    // - Creates a private class property called usersService
    // - Assigns the injected value to it
    // - readonly: prevents reassignment (immutable reference)
    // Equivalent to:
    // private readonly usersService: UsersService;
    // constructor(usersService: UsersService) { this.usersService = usersService; }
  }

  // ─── GET /users ─────────────────────────────────────────────────────
  @Get()
  // No path argument = matches /users exactly
  findAll(
    @Query('page') page: string = '1',
    // @Query() extracts query string params: GET /users?page=2&limit=10
    // Note: query params are ALWAYS strings — ParseIntPipe converts them
    
    @Query('limit') limit: string = '10',
  ) {
    // Controllers should be thin — delegate to service
    return this.usersService.findAll({
      page: parseInt(page),
      limit: parseInt(limit),
    });
    // NestJS automatically serializes the returned value to JSON
    // Sets Content-Type: application/json
    // Status code is 200 by default for GET
  }

  // ─── GET /users/:id ──────────────────────────────────────────────────
  @Get(':id')
  // :id is a route parameter — matches /users/1, /users/abc, /users/anything
  findOne(
    @Param('id', ParseIntPipe) id: number,
    // @Param('id') extracts route parameter named 'id' → always a string
    // ParseIntPipe transforms string '1' → number 1
    // If transformation fails (e.g., /users/abc), throws 400 Bad Request automatically
  ) {
    return this.usersService.findOne(id);
    // If service throws NotFoundException, NestJS returns 404 response
  }

  // ─── POST /users ─────────────────────────────────────────────────────
  @Post()
  @HttpCode(HttpStatus.CREATED)
  // By default POST returns 200. @HttpCode(201) changes the status code.
  // HttpStatus.CREATED = 201 (semantic: resource was created)
  create(
    @Body() createUserDto: CreateUserDto,
    // @Body() extracts the request body and parses JSON
    // createUserDto: CreateUserDto — ValidationPipe validates this automatically
    // If validation fails (missing fields, wrong types), returns 400 with error details
  ) {
    return this.usersService.create(createUserDto);
  }

  // ─── PATCH /users/:id ────────────────────────────────────────────────
  @Patch(':id')
  // PATCH = partial update (only send fields you want to change)
  // PUT = full replacement (send ALL fields)
  // NestJS convention: use PATCH for partial updates
  update(
    @Param('id', ParseIntPipe) id: number,
    @Body() updateUserDto: UpdateUserDto,
  ) {
    return this.usersService.update(id, updateUserDto);
  }

  // ─── DELETE /users/:id ───────────────────────────────────────────────
  @Delete(':id')
  @HttpCode(HttpStatus.NO_CONTENT)
  // 204 No Content — success, but no body in response
  // Standard REST convention for DELETE
  remove(@Param('id', ParseIntPipe) id: number) {
    return this.usersService.remove(id);
  }
}
```

### Route parameter patterns

```typescript
@Controller('products')
export class ProductsController {

  // ── Simple routes ──────────────────────────────────────────────────

  @Get()                    // GET /products
  @Get(':id')               // GET /products/123
  @Get(':id/reviews')       // GET /products/123/reviews
  @Get(':id/reviews/:reviewId')  // GET /products/123/reviews/456


  // ── Nested routes ─────────────────────────────────────────────────

  // GET /products/featured — MUST come before :id or 'featured' matches as id
  @Get('featured')
  getFeatured() { /* ... */ }

  // GET /products/123
  @Get(':id')
  findOne(@Param('id') id: string) { /* ... */ }


  // ── Wildcards ─────────────────────────────────────────────────────

  @Get('ab*cd')  // Matches: abcd, ab_cd, abanythingcd
  wildcard() { /* ... */ }


  // ── Multiple decorators ───────────────────────────────────────────

  @Get()
  @Post()  // Same method handles both GET and POST
  handleBoth() { /* ... */ }


  // ── Full request/response control ────────────────────────────────

  @Get('raw')
  rawResponse(
    @Req() req: Request,     // Full Express Request object
    @Res() res: Response,    // Full Express Response object
  ) {
    // When you inject @Res(), NestJS DISABLES automatic response handling
    // YOU must manually send the response
    res.status(200).json({ data: 'manual response' });
    // Warning: interceptors won't work when using @Res() like this
    // Use @Res({ passthrough: true }) to keep NestJS interception + manual response
  }
}
```

### Response customization

```typescript
// Method 1: Return value — NestJS handles everything (PREFERRED)
@Get(':id')
findOne(@Param('id', ParseIntPipe) id: number) {
  return this.usersService.findOne(id);
  // NestJS: serializes to JSON, sets Content-Type, uses 200 status
}

// Method 2: Throw exceptions — NestJS converts to HTTP error response
@Get(':id')
async findOne(@Param('id', ParseIntPipe) id: number) {
  const user = await this.usersService.findOne(id);
  if (!user) {
    throw new NotFoundException(`User #${id} not found`);
    // NestJS catches this and returns:
    // HTTP 404 { "statusCode": 404, "message": "User #1 not found", "error": "Not Found" }
  }
  return user;
}

// Available NestJS exceptions:
// BadRequestException      → 400
// UnauthorizedException    → 401
// ForbiddenException       → 403
// NotFoundException        → 404
// ConflictException        → 409
// UnprocessableEntityException → 422
// InternalServerErrorException → 500
// ServiceUnavailableException  → 503
```

## 6. Providers & Services — Business Logic Layer

### What is a provider?
In NestJS, a **provider** is anything that can be **injected** as a dependency. This includes services, repositories, factories, helpers — anything decorated with `@Injectable()`.

**Services** are the most common type of provider. They contain the actual business logic.

### Why separate services from controllers?

```
Without separation (bad):
  Controller → directly queries database, sends emails, processes payments
  Problem: Can't test controller without a real DB and mail server
  Problem: Logic is duplicated when multiple controllers need same operation

With separation (good):
  Controller → UsersService → UserRepository
  Controller handles: routing, request parsing, response formatting
  Service handles: business rules, orchestration, data access
  Result: Service can be tested in isolation with mocked repository
```

```typescript
// src/users/users.service.ts
import { Injectable, NotFoundException, ConflictException } from '@nestjs/common';
import { InjectRepository } from '@nestjs/typeorm';
import { Repository } from 'typeorm';
import { User } from './entities/user.entity';
import { CreateUserDto } from './dto/create-user.dto';
import { UpdateUserDto } from './dto/update-user.dto';
import * as bcrypt from 'bcrypt';

// @Injectable() marks this class as a PROVIDER
// Without this decorator, NestJS cannot manage this class or inject dependencies into it
// It tells NestJS: "Include this in the IoC container"
@Injectable()
export class UsersService {
  constructor(
    // @InjectRepository(User) is how you inject TypeORM repositories
    // It's shorthand for the token 'UserRepository'
    // TypeORM's module registers this provider when you call TypeOrmModule.forFeature([User])
    @InjectRepository(User)
    private readonly userRepository: Repository<User>,
  ) {}

  // ── findAll ─────────────────────────────────────────────────────────
  async findAll(options: { page: number; limit: number }): Promise<{
    data: User[];
    total: number;
    page: number;
    lastPage: number;
  }> {
    const { page, limit } = options;
    
    // findAndCount: returns [entities, total count] in one query
    // skip: how many records to skip (offset)
    // take: how many records to return (limit)
    const [data, total] = await this.userRepository.findAndCount({
      skip: (page - 1) * limit,  // Page 1 = skip 0, Page 2 = skip limit
      take: limit,
      order: { createdAt: 'DESC' },  // Newest first
    });

    return {
      data,
      total,
      page,
      lastPage: Math.ceil(total / limit),
    };
  }

  // ── findOne ──────────────────────────────────────────────────────────
  async findOne(id: number): Promise<User> {
    // findOne returns null if not found — we throw to give proper 404
    const user = await this.userRepository.findOne({
      where: { id },
      // relations: ['orders'] — load related entities if needed
    });

    if (!user) {
      // NestJS converts this exception to HTTP 404 automatically
      throw new NotFoundException(`User with ID ${id} not found`);
    }

    return user;
  }

  // ── create ───────────────────────────────────────────────────────────
  async create(createUserDto: CreateUserDto): Promise<User> {
    // Check for duplicate email before creating
    const existing = await this.userRepository.findOne({
      where: { email: createUserDto.email },
    });

    if (existing) {
      // 409 Conflict — resource already exists
      throw new ConflictException('User with this email already exists');
    }

    // Hash password before storing — NEVER store plain text passwords
    const salt = await bcrypt.genSalt(10);
    const hashedPassword = await bcrypt.hash(createUserDto.password, salt);

    // Create entity instance — does NOT save to DB yet
    const user = this.userRepository.create({
      ...createUserDto,
      password: hashedPassword,
    });

    // save() persists to DB — returns the saved entity with generated id, timestamps
    return this.userRepository.save(user);
  }

  // ── update ───────────────────────────────────────────────────────────
  async update(id: number, updateUserDto: UpdateUserDto): Promise<User> {
    // findOne throws NotFoundException if not found — no need to check again
    const user = await this.findOne(id);

    // Object.assign merges updateUserDto into the user entity instance
    // Only updates fields that are present in updateUserDto
    Object.assign(user, updateUserDto);

    return this.userRepository.save(user);
    // save() on an existing entity (has an id) → UPDATE query, not INSERT
  }

  // ── remove ───────────────────────────────────────────────────────────
  async remove(id: number): Promise<void> {
    const user = await this.findOne(id);  // Throws if not found
    
    await this.userRepository.remove(user);
    // remove() calls DELETE query and removes the entity from DB
    // Returns void — controller returns 204 No Content
  }

  // ── findByEmail ──────────────────────────────────────────────────────
  // Used by AuthService — demonstrates service reusability
  async findByEmail(email: string): Promise<User | null> {
    return this.userRepository.findOne({ where: { email } });
    // Returns null (not exception) — AuthService decides what to do
    // Services should be flexible — let callers decide on error handling
  }
}
```

### Custom providers — different ways to provide values

```typescript
// In your module's providers array, you have several options:

@Module({
  providers: [
    // ── 1. Class provider (most common) ────────────────────────────
    UsersService,
    // Shorthand for:
    { provide: UsersService, useClass: UsersService },
    // NestJS creates an instance of UsersService and injects it when needed

    // ── 2. Value provider ──────────────────────────────────────────
    {
      provide: 'APP_CONFIG',  // Injection token (string)
      useValue: {
        maxRequestsPerMinute: 100,
        defaultPageSize: 20,
      },
    },
    // Inject it with: @Inject('APP_CONFIG') private config: AppConfig

    // ── 3. Factory provider ────────────────────────────────────────
    {
      provide: 'LOGGER',
      useFactory: (configService: ConfigService) => {
        // Factory function — called once to create the provider
        // inject array specifies what to pass to the factory
        const level = configService.get('LOG_LEVEL') || 'info';
        return createLogger({ level });
      },
      inject: [ConfigService],  // Dependencies for the factory function
    },

    // ── 4. Alias provider ──────────────────────────────────────────
    {
      provide: 'PRIMARY_USER_SERVICE',
      useExisting: UsersService,
      // Points to an existing provider — same instance, different token
    },
  ],
})
export class UsersModule {}
```

## 7. Dependency Injection — How NestJS Wires Everything

### What is Dependency Injection (DI)?
DI is a design pattern where **objects receive their dependencies from outside** rather than creating them internally.

### Simple analogy
Without DI, a service "goes to the store to buy ingredients." With DI, ingredients are "delivered to the door." The service doesn't care where they come from — it just uses them.

### The problem DI solves

```typescript
// ❌ WITHOUT Dependency Injection — tight coupling
class UsersService {
  // Creates its own dependency — problem!
  private db = new DatabaseConnection({ host: 'localhost', port: 5432 });
  //                  ↑ Hard to test: test always needs a real database
  //                  ↑ Hard to change: swap PostgreSQL for MySQL? Change every service
  //                  ↑ Hard to configure: connection options are hardcoded

  async findAll() {
    return this.db.query('SELECT * FROM users');
  }
}

// ✅ WITH Dependency Injection — loose coupling
class UsersService {
  // Receives dependency — doesn't create it
  constructor(private readonly db: DatabaseConnection) {
    //                              ↑ Can inject real DB in production
    //                              ↑ Can inject mock DB in tests
    //                              ↑ Can swap implementations without changing service
  }

  async findAll() {
    return this.db.query('SELECT * FROM users');
  }
}
```

### How NestJS DI works step by step

```typescript
// Step 1: You declare a provider with @Injectable()
@Injectable()
class EmailService {
  sendWelcome(email: string) { /* ... */ }
}

// Step 2: You register it in a module
@Module({
  providers: [EmailService, UsersService],
})
class UsersModule {}

// Step 3: You declare it as a dependency in a constructor
@Injectable()
class UsersService {
  constructor(
    private readonly emailService: EmailService,
    // ↑ NestJS reads the TYPE 'EmailService' via Reflect.metadata
    // (because emitDecoratorMetadata: true in tsconfig)
  ) {}
}

// What NestJS does at startup (simplified):
// 1. See UsersService needs EmailService
// 2. Check if EmailService is in the module's providers → yes
// 3. Check if EmailService has its own dependencies → no
// 4. Create EmailService instance: const emailService = new EmailService()
// 5. Create UsersService instance: const usersService = new UsersService(emailService)
// 6. Register both in the container
// 7. Next time UsersService is needed → return the SAME instance (singleton by default)
```

### Provider scopes — singleton vs request vs transient

```typescript
import { Injectable, Scope } from '@nestjs/common';

// ── DEFAULT: Singleton scope ────────────────────────────────────────────
// One instance for the ENTIRE application lifetime
// Created once at startup, shared across all requests
// This is what you want for stateless services (database services, etc.)
@Injectable()  // Scope.DEFAULT is implicit
export class UsersService {}

// ── Request scope ───────────────────────────────────────────────────────
// One NEW instance per HTTP request
// Instance is destroyed when the request completes
// Use when: you need to store request-specific data in the provider
// Downside: more GC pressure, can't be injected into singleton providers easily
@Injectable({ scope: Scope.REQUEST })
export class RequestContextService {
  private userId: number;

  setUserId(id: number) { this.userId = id; }
  getUserId() { return this.userId; }
  // Safe! Each request gets its own instance, no data bleeding between requests
}

// ── Transient scope ─────────────────────────────────────────────────────
// New instance every time it is injected (everywhere it's used)
// Rarely needed — most services should be singletons
@Injectable({ scope: Scope.TRANSIENT })
export class LoggerService {
  // Each consumer gets their own independent logger instance
  // Good when: provider maintains its own state per consumer
}

// IMPORTANT RULE: Scope bubbles up
// If a singleton service injects a request-scoped service,
// the singleton BECOMES request-scoped too (NestJS forces it)
// This is why you rarely use REQUEST scope
```

### Constructor injection vs property injection

```typescript
// ── Constructor injection (RECOMMENDED) ────────────────────────────────
@Injectable()
export class UsersService {
  // TypeScript shorthand creates private property AND assigns in one line
  constructor(
    private readonly userRepository: Repository<User>,
    private readonly emailService: EmailService,
    private readonly configService: ConfigService,
  ) {}
  // Advantages:
  // - Dependencies are explicit (you see exactly what's needed)
  // - Works with readonly (immutable)
  // - Required by default — service can't be created without them
  // - Testing: just pass mocks to the constructor
}

// ── Property injection (use rarely) ────────────────────────────────────
@Injectable()
export class UsersService {
  @Inject(ConfigService)
  private readonly configService: ConfigService;
  // When to use:
  // - Circular dependency resolution (A needs B, B needs A)
  // - Base classes that derived classes want to inherit from
  // Generally avoid — harder to test, less explicit
}

// ── @Inject() with token (for non-class providers) ─────────────────────
@Injectable()
export class UsersService {
  constructor(
    @Inject('APP_CONFIG') private readonly config: AppConfig,
    // 'APP_CONFIG' is the string token from:
    // { provide: 'APP_CONFIG', useValue: { ... } }
    
    @Inject('LOGGER') private readonly logger: Logger,
    // Same for factory providers
  ) {}
}
```

## 8. @Injectable, @Controller, @Module — Under the Hood

### What these decorators actually do

```typescript
// ── @Injectable() — what it really does ────────────────────────────────

// Simplified implementation of @Injectable():
function Injectable(options?: { scope?: Scope }): ClassDecorator {
  return (target: Function) => {
    // 1. Mark this class as injectable (NestJS checks for this)
    Reflect.defineMetadata(INJECTABLE_WATERMARK, true, target);
    
    // 2. Store scope options
    Reflect.defineMetadata(SCOPE_OPTIONS_METADATA, options, target);
    
    // 3. TypeScript already emitted constructor parameter types
    //    because of emitDecoratorMetadata: true
    //    NestJS reads: Reflect.getMetadata('design:paramtypes', target)
    //    → [EmailService, ConfigService, ...]
    //    And knows what to inject
  };
}

// ── @Controller() — what it really does ────────────────────────────────

function Controller(prefix?: string): ClassDecorator {
  return (target: Function) => {
    // 1. Mark as controller
    Reflect.defineMetadata(CONTROLLER_WATERMARK, true, target);
    
    // 2. Store the route prefix
    Reflect.defineMetadata(PATH_METADATA, prefix || '/', target);
    
    // 3. NestJS reads this during app bootstrap to register routes
  };
}

// ── @Get(), @Post() etc — route decorators ──────────────────────────────

function Get(path?: string): MethodDecorator {
  return (target, key, descriptor) => {
    // Store route metadata on the METHOD (not the class)
    Reflect.defineMetadata(METHOD_METADATA, RequestMethod.GET, descriptor.value);
    Reflect.defineMetadata(PATH_METADATA, path || '/', descriptor.value);
    // NestJS iterates over all methods, finds these markers, and registers routes
  };
}

// ── @Module() — what it really does ────────────────────────────────────

function Module(metadata: ModuleMetadata): ClassDecorator {
  return (target: Function) => {
    // Store all module metadata for later reading by the scanner
    Reflect.defineMetadata(METADATA.IMPORTS, metadata.imports, target);
    Reflect.defineMetadata(METADATA.EXPORTS, metadata.exports, target);
    Reflect.defineMetadata(METADATA.CONTROLLERS, metadata.controllers, target);
    Reflect.defineMetadata(METADATA.PROVIDERS, metadata.providers, target);
  };
}
// When NestFactory.create(AppModule) runs, it reads all this metadata
// to build the complete dependency graph
```

## 9. Request Flow — End to End

### How a single HTTP request moves through NestJS

```
  HTTP Request: POST /api/v1/users
  ─────────────────────────────────────────────────────
  
  1. Express/Fastify receives the TCP request
       ↓
  2. NestJS Middleware (if any registered for this route)
     - Runs before everything else
     - Can modify request/response or call next()
       ↓
  3. Guards (e.g., JwtAuthGuard)
     - Returns true (continue) or false/throw (block)
     - Used for authentication and authorization
       ↓
  4. Interceptors — "before" logic
     - Can transform the request
     - Starts a timer, adds logging, etc.
       ↓
  5. Pipes
     - Validates and transforms request data
     - ValidationPipe: validates DTO, throws 400 if invalid
     - ParseIntPipe: converts ':id' string → number
       ↓
  6. Controller method executes
     - Calls UsersService.create(dto)
       ↓
  7. Service executes
     - Business logic, DB queries
     - Returns data or throws exceptions
       ↓
  8. Exception Filter (if exception was thrown)
     - Catches exceptions, formats error response
     - NestJS has a default filter for HttpExceptions
       ↓
  9. Interceptors — "after" logic
     - Can transform the RESPONSE
     - Log response time, wrap response in envelope
       ↓
  10. Response sent back to client
```

```typescript
// Example: seeing this flow in code

// Global middleware (runs first)
app.use((req, res, next) => {
  console.log(`[${new Date().toISOString()}] ${req.method} ${req.url}`);
  next();  // Must call next() to pass control
});

// Guard (runs second)
@Injectable()
export class JwtAuthGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    const request = context.switchToHttp().getRequest();
    const token = request.headers.authorization?.split(' ')[1];
    if (!token) throw new UnauthorizedException();
    // validate token...
    return true;
  }
}

// Interceptor (wraps around handler)
@Injectable()
export class LoggingInterceptor implements NestInterceptor {
  intercept(context: ExecutionContext, next: CallHandler): Observable<any> {
    const now = Date.now();
    console.log('Before handler');
    
    return next.handle().pipe(
      tap(() => console.log(`After handler: ${Date.now() - now}ms`)),
    );
  }
}

// Pipe (runs just before controller method)
// ValidationPipe is configured globally in main.ts

// Controller method (the actual handler)
@Post()
@UseGuards(JwtAuthGuard)
@UseInterceptors(LoggingInterceptor)
create(@Body() createUserDto: CreateUserDto) {
  return this.usersService.create(createUserDto);
}
```

## 11. Interview Questions & Answers

**Q1: What is the difference between a module, controller, and provider in NestJS?**

> "A **module** is the organizational unit — it groups related controllers and providers and declares what the module imports and exports. It's like a feature package. A **controller** handles HTTP routing — it maps HTTP requests to handler methods but contains no business logic. A **provider** is any injectable class — typically a **service** that contains business logic, but also repositories, factories, or plain values. The module wires these together and the DI container manages their lifetimes."

---

**Q2: How does dependency injection work in NestJS?**

> "NestJS has an IoC container — a registry of providers. When the app starts, NestJS scans all modules and builds a dependency graph. When a service has constructor parameters, NestJS reads their TypeScript types via `Reflect.metadata` (enabled by `emitDecoratorMetadata: true` in tsconfig). It looks up those types in the container, creates them in dependency order, and injects them. By default, providers are singletons — one instance shared across the entire app. This makes the system loosely coupled: services don't create their own dependencies, which makes swapping implementations and writing tests easy."

---

**Q3: What is the difference between imports and providers in @Module()?**

> "**providers** lists classes that this module creates and makes available for injection within the module — they are this module's own providers. **imports** lists other modules whose exported providers you want to use in this module. When you import a module, you get access to everything it exports. Think of providers as your inventory and imports as your supply chain."

---

**Q4: What are the different provider scopes and when would you use each?**

> "**Singleton** (default) — one instance for the entire app lifetime, shared by all consumers. Best for stateless services like database services, email services, etc. **Request** — a new instance is created for each incoming HTTP request and destroyed when it ends. Use when the provider needs to store request-specific state. **Transient** — a new instance every time it's injected into a consumer. Rare — use when each consumer needs an independent instance. Scope bubbles up: if a singleton injects a request-scoped provider, the singleton also becomes request-scoped."

---

**Q5: What does @Global() do and when should you use it?**

> "`@Global()` makes a module's exports available to all other modules without needing to import it explicitly. NestJS recommends using it sparingly — only for truly cross-cutting concerns like configuration, logging, and database connections. The reason to be conservative is that global modules make dependencies invisible — you can't tell by looking at a module's imports what it depends on. For most features, explicit imports are better because they make dependencies clear and the module self-documenting."

---

**Q6: What is the order of execution in the NestJS request pipeline?**

> "The order is: Middleware → Guards → Interceptors (before) → Pipes → Controller method → Interceptors (after) → Exception Filters (if an error occurred). Middleware runs first and has no NestJS context — it's pure Express middleware. Guards decide whether to allow the request (authentication/authorization). Interceptors wrap the entire handler and can transform both requests and responses. Pipes transform and validate input data. The controller method executes, calling the service. If anything throws, exception filters catch it and format the error response. Interceptors run again after the handler for response transformation."

---

**Q7: Can you have circular dependencies in NestJS modules? How do you solve them?**

> "Yes, circular dependencies can happen — Module A imports Module B and Module B imports Module A. NestJS provides `forwardRef()` to handle this: `imports: [forwardRef(() => ModuleB)]`. This delays the reference until after both modules are defined. For provider-level circular dependencies (Service A needs Service B and vice versa), you use `@Inject(forwardRef(() => ServiceB))` in the constructor. However, circular dependencies are usually a sign of poor architecture — it's better to extract the shared logic into a third SharedModule that both can import without circularity."

---

## Quick Reference — Day 2 Cheat Sheet

```
Module anatomy:
  imports    → other modules' exports you need
  controllers → HTTP route handlers (no business logic)
  providers  → services, factories, values (injectable)
  exports    → what other modules can use from you

Provider scopes:
  DEFAULT    → singleton (one instance, whole app)
  REQUEST    → one per HTTP request
  TRANSIENT  → one per injection

Request pipeline order:
  Middleware → Guard → Interceptor(before) → Pipe
  → Controller → Service → Interceptor(after) → Filter

DI requires:
  ✓ @Injectable() on the provider class
  ✓ Listed in module's providers array
  ✓ emitDecoratorMetadata: true in tsconfig
  ✓ Correct module imports (or @Global())

CLI shortcuts:
  nest g mo <name>   → module
  nest g co <name>   → controller
  nest g s <name>    → service
  nest g res <name>  → full resource (all of the above)

Key rule: Controllers are thin, Services are fat
  Controller: parse request → call service → return response
  Service: ALL business logic, ALL database operations
```
