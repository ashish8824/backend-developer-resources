# Day 6 — Database with TypeORM
### NestJS Interview Prep | 3-Year Backend Engineer Level

> **Goal for today:** Master TypeORM in NestJS — from basic entity setup to complex relations, QueryBuilder for advanced queries, migrations for production-safe schema changes, and transaction handling. A 3-year backend engineer is expected to write production-quality database code, understand SQL generated by the ORM, and know when to step outside the ORM and write raw queries.

---

## 🧭 First-Timer Guidance — Read This First

**What is an ORM and why TypeORM?**

ORM = Object-Relational Mapper. It lets you work with your database using JavaScript/TypeScript classes and objects instead of raw SQL strings.

```
Without ORM (raw SQL):
  const result = await db.query(
    'SELECT * FROM users WHERE id = $1 AND is_active = $2',
    [userId, true]
  );
  // Unsafe: typos in column names crash at runtime, not compile time
  // No TypeScript types on result
  // Hard to maintain across table schema changes

With TypeORM:
  const user = await userRepository.findOne({
    where: { id: userId, isActive: true }
  });
  // TypeScript knows the shape of 'user'
  // Column name typos caught at compile time
  // Easy to refactor when schema changes
```

**Why TypeORM specifically?**
- First-class NestJS support via `@nestjs/typeorm`
- TypeScript decorators for entity definition
- Supports PostgreSQL, MySQL, SQLite, MongoDB, and more
- Active Record pattern AND Data Mapper pattern (NestJS uses Data Mapper)
- Built-in migrations system
- Most popular ORM in the NestJS ecosystem

**Setup — install packages:**
```bash
npm install @nestjs/typeorm typeorm pg
# pg = PostgreSQL driver
# Use 'mysql2' for MySQL, 'sqlite3' for SQLite
```

**Mental model for today:**
```
Entity Class     → Database Table
Entity Property  → Table Column
@OneToMany()     → One-to-Many Relation
Repository<T>    → Methods to query/save that entity
QueryBuilder     → Raw SQL power with TypeScript safety
Migration        → Version-controlled schema changes
Transaction      → Atomic multi-step database operations
```

## 1. TypeORM Setup — Connecting to Database

### AppModule setup with ConfigService

```typescript
// src/app.module.ts

import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { ConfigModule, ConfigService } from '@nestjs/config';

@Module({
  imports: [
    ConfigModule.forRoot({ isGlobal: true }),

    // TypeOrmModule.forRootAsync: async factory that waits for ConfigService
    // NEVER use forRoot() directly — you can't inject ConfigService there
    TypeOrmModule.forRootAsync({
      inject: [ConfigService],

      useFactory: (config: ConfigService) => ({
        // ── Connection settings ────────────────────────────────────────
        type: 'postgres',                              // Database type
        host: config.getOrThrow('DB_HOST'),
        port: parseInt(config.get('DB_PORT', '5432')),
        username: config.getOrThrow('DB_USERNAME'),
        password: config.getOrThrow('DB_PASSWORD'),
        database: config.getOrThrow('DB_DATABASE'),

        // ── Entity discovery ───────────────────────────────────────────
        autoLoadEntities: true,
        // TypeORM automatically loads entities registered with
        // TypeOrmModule.forFeature([Entity]) in any module.
        // This replaces the 'entities' array and is ALWAYS preferred
        // because you don't have to manually list every entity here.

        // ── Schema management ──────────────────────────────────────────
        synchronize: config.get('NODE_ENV') !== 'production',
        // TRUE in development: TypeORM auto-creates/modifies tables to
        // match your entity classes — convenient but DESTRUCTIVE
        // FALSE in production: NEVER synchronize in production
        // It can DROP columns, lose data, and cause outages
        // Use MIGRATIONS in production (covered in section 7)

        // ── Logging ────────────────────────────────────────────────────
        logging: config.get('NODE_ENV') === 'development',
        // Log every SQL query in development — essential for debugging
        // In production: too noisy, performance overhead
        // Advanced: logging: ['error', 'warn'] — only log problems

        // ── Connection pool ────────────────────────────────────────────
        extra: {
          max: parseInt(config.get('DB_POOL_SIZE', '10')),
          // Max simultaneous connections to PostgreSQL
          // Development: 5-10, Production: 20-100 depending on traffic
          // Each NestJS worker process gets this many connections
          // With 4 workers × 25 connections = 100 total DB connections
          idleTimeoutMillis: 30000,     // Close idle connections after 30s
          connectionTimeoutMillis: 2000, // Fail if can't connect in 2s
        },

        // ── SSL (required for cloud databases) ─────────────────────────
        ssl: config.get('NODE_ENV') === 'production'
          ? { rejectUnauthorized: false }
          // rejectUnauthorized: false = accept self-signed certs (AWS RDS)
          : false,

        // ── Migrations config ──────────────────────────────────────────
        migrations: [__dirname + '/migrations/**/*{.ts,.js}'],
        // Where migration files live — TypeORM runs these on app start
        // OR you run them manually with 'npm run migration:run'
      }),
    }),
  ],
})
export class AppModule {}
```

### Feature module registration

```typescript
// src/users/users.module.ts

import { Module } from '@nestjs/common';
import { TypeOrmModule } from '@nestjs/typeorm';
import { User } from './entities/user.entity';
import { Profile } from './entities/profile.entity';
import { UsersController } from './users.controller';
import { UsersService } from './users.service';

@Module({
  imports: [
    TypeOrmModule.forFeature([User, Profile]),
    // Registers TWO things for each entity:
    // 1. Repository<User> provider — injectable as @InjectRepository(User)
    // 2. Registers entity with TypeORM so it creates the table
    // This is HOW autoLoadEntities works — each forFeature() call
    // tells TypeORM "this entity exists in this module"
  ],
  controllers: [UsersController],
  providers: [UsersService],
  exports: [UsersService, TypeOrmModule],
  // Export TypeOrmModule to allow other modules to inject User repository
  // if they import UsersModule
})
export class UsersModule {}
```

## 2. Entities — Mapping Classes to Tables

### What is an Entity?
An entity is a TypeScript class that maps directly to a database table. Each property maps to a column. TypeORM reads the decorators and creates/manages the table.

### Complete Entity — every important decorator

```typescript
// src/users/entities/user.entity.ts

import {
  Entity,
  PrimaryGeneratedColumn,
  PrimaryColumn,
  Column,
  CreateDateColumn,
  UpdateDateColumn,
  DeleteDateColumn,
  Index,
  Unique,
  OneToOne,
  OneToMany,
  ManyToOne,
  ManyToMany,
  JoinColumn,
  JoinTable,
  BeforeInsert,
  BeforeUpdate,
  AfterLoad,
  VersionColumn,
} from 'typeorm';
import { Exclude } from 'class-transformer';

// @Entity() — marks this class as a database table
// The table name defaults to the class name in lowercase ('user')
// @Entity('users') — explicitly set table name to 'users' (recommended)
// Always use plural table names — SQL convention
@Entity('users')

// @Unique() at class level — creates a UNIQUE CONSTRAINT across MULTIPLE columns
// This is different from @Column({ unique: true }) which is single-column unique
@Unique(['email', 'tenantId'])  // email must be unique PER tenant
// @Index() at class level — creates a composite index
@Index(['firstName', 'lastName'])  // Speeds up searches by full name
export class User {

  // ── Primary Key ────────────────────────────────────────────────────────
  @PrimaryGeneratedColumn()
  // Auto-increment integer: 1, 2, 3, 4...
  // Simple, sequential, small storage, but predictable (security concern)
  id: number;

  // Alternative: UUID primary key
  // @PrimaryGeneratedColumn('uuid')
  // Generates: '6ba7b810-9dad-11d1-80b4-00c04fd430c8'
  // Not predictable (better security), but larger storage, slower joins
  // id: string;

  // Alternative: Manual primary key (you control the value)
  // @PrimaryColumn()
  // id: string;  // You must set this before saving

  // ── Basic columns ──────────────────────────────────────────────────────
  @Column({ length: 50 })
  // length: max characters — creates VARCHAR(50) in PostgreSQL
  // Default without length: TEXT (unlimited) in PostgreSQL
  firstName: string;
  // Column name in DB: 'first_name' (TypeORM converts camelCase → snake_case)

  @Column({ length: 50 })
  lastName: string;

  @Column({
    unique: true,
    // Creates a UNIQUE constraint on this column
    // Faster than a separate @Index for unique columns
    length: 255,
    // Always set a reasonable max for indexed columns
  })
  email: string;

  @Exclude()
  // class-transformer: exclude from JSON serialization
  // Users never see the hashed password in API responses
  @Column({ select: false })
  // select: false — TypeORM NEVER includes this column in SELECT queries
  // You must explicitly ask for it: .addSelect('user.password')
  // Double protection: excluded from TypeORM queries AND JSON serialization
  password: string;

  // ── Nullable columns ───────────────────────────────────────────────────
  @Column({ nullable: true, length: 20 })
  // nullable: true — allows NULL in database (column is optional)
  // In TypeScript: string | null (be explicit about nullable types)
  phone: string | null;

  // ── Enum column ────────────────────────────────────────────────────────
  @Column({
    type: 'enum',
    enum: UserRole,
    // Creates a PostgreSQL ENUM type or VARCHAR with CHECK constraint
    // In PostgreSQL, ENUM is a real database type — efficient
    default: UserRole.USER,
  })
  role: UserRole;

  // ── Boolean column ─────────────────────────────────────────────────────
  @Column({ default: true })
  // default: true — database-level default value
  // Applied when you INSERT without specifying this column
  isActive: boolean;

  // ── Number columns ─────────────────────────────────────────────────────
  @Column({ type: 'decimal', precision: 10, scale: 2, default: 0 })
  // DECIMAL(10, 2): up to 10 digits total, 2 after decimal point
  // For money: ALWAYS use DECIMAL, NEVER float (float has rounding errors!)
  // 1234567.89 — 10 digits total, 2 after decimal
  balance: number;

  @Column({ type: 'int', default: 0 })
  // Explicit int type (same as default for @Column() on number property)
  loginCount: number;

  // ── JSON column ────────────────────────────────────────────────────────
  @Column({ type: 'jsonb', nullable: true })
  // jsonb: PostgreSQL binary JSON — queryable, indexable, stored efficiently
  // 'json': stored as text — simpler but not queryable by key
  // Use jsonb for any structured data you might need to query
  metadata: Record<string, any> | null;

  // ── Text column ────────────────────────────────────────────────────────
  @Column({ type: 'text', nullable: true })
  // TEXT: unlimited string length
  // Use for: long descriptions, HTML content, JSON (use jsonb instead)
  bio: string | null;

  // ── Timestamp columns ──────────────────────────────────────────────────
  @CreateDateColumn()
  // Automatically set to NOW() when entity is first saved
  // TypeORM manages this — you never set it manually
  createdAt: Date;

  @UpdateDateColumn()
  // Automatically updated to NOW() every time entity is saved
  // Tracks when the record was last modified
  updatedAt: Date;

  @DeleteDateColumn()
  // SOFT DELETE support — covered in section 9
  // When you call repository.softDelete(), this is set to NOW()
  // The record stays in the database but is excluded from normal queries
  // NULL means "not deleted", a date means "deleted at this time"
  deletedAt: Date | null;

  // ── Version column (optimistic locking) ───────────────────────────────
  @VersionColumn()
  // Auto-incremented integer: 1, 2, 3...
  // TypeORM uses this for OPTIMISTIC LOCKING:
  // "Save this record, but only if the version is still 5"
  // If another process updated it (version is now 6), the save FAILS
  // Prevents lost updates in concurrent environments
  version: number;

  // ── Virtual column (not in database) ──────────────────────────────────
  fullName: string;  // No @Column() → not saved to database

  @AfterLoad()
  // Lifecycle hook: runs after entity is loaded from database
  // Perfect for computed properties
  setFullName() {
    this.fullName = `${this.firstName} ${this.lastName}`;
  }

  // ── Lifecycle hooks ────────────────────────────────────────────────────
  @BeforeInsert()
  // Runs BEFORE INSERT query — use for transformation/normalization
  normalizeEmail() {
    this.email = this.email.toLowerCase().trim();
    // Normalize email before saving — prevents duplicate accounts
    // with different cases: "Alice@Test.com" vs "alice@test.com"
  }

  @BeforeInsert()
  @BeforeUpdate()
  // Can apply multiple lifecycle decorators to same method
  validateData() {
    if (this.balance < 0) {
      throw new Error('Balance cannot be negative');
    }
  }
}

// The enum used above
export enum UserRole {
  USER = 'user',
  ADMIN = 'admin',
  MODERATOR = 'moderator',
}
```

### Column types reference

```typescript
// TypeScript type → PostgreSQL column type mapping

@Column()                           // string  → VARCHAR(255)
@Column({ length: 100 })            // string  → VARCHAR(100)
@Column({ type: 'text' })           // string  → TEXT
@Column()                           // number  → INTEGER
@Column({ type: 'bigint' })         // string  → BIGINT (returns string!)
@Column({ type: 'float' })          // number  → FLOAT8
@Column({ type: 'decimal',
          precision: 10, scale: 2 }) // number  → DECIMAL(10,2)
@Column()                           // boolean → BOOLEAN
@Column()                           // Date    → TIMESTAMP WITH TIME ZONE
@Column({ type: 'date' })           // string  → DATE (no time)
@Column({ type: 'time' })           // string  → TIME
@Column({ type: 'json' })           // object  → JSON (text)
@Column({ type: 'jsonb' })          // object  → JSONB (binary, queryable)
@Column({ type: 'uuid' })           // string  → UUID
@Column({ type: 'bytea' })          // Buffer  → BYTEA (binary data)
@Column('simple-array')             // string[]→ VARCHAR (comma-separated)
@Column('simple-json')              // object  → TEXT (JSON.stringify'd)
```

## 3. Column Types & Decorators

### Indexes — critical for query performance

```typescript
import { Entity, Column, Index } from 'typeorm';

@Entity('products')
export class Product {
  // ── Single column index ────────────────────────────────────────────────
  @Index()
  // Creates a regular B-tree index on this column
  // Speeds up: WHERE category = 'electronics'
  // Slows down: INSERT/UPDATE/DELETE (index must be updated)
  // Rule: index columns you frequently filter/sort by
  @Column()
  category: string;

  // ── Unique index ───────────────────────────────────────────────────────
  @Index({ unique: true })
  // Equivalent to @Column({ unique: true }) but more explicit
  @Column()
  sku: string;  // Product SKU must be unique

  // ── Named index ────────────────────────────────────────────────────────
  @Index('idx_product_price')
  // Custom index name — useful for referencing in queries or migrations
  @Column({ type: 'decimal', precision: 10, scale: 2 })
  price: number;

  // ── Composite index (at class level) ───────────────────────────────────
  // @Index(['category', 'price']) — on the class, not property
  // Speeds up: WHERE category = 'electronics' ORDER BY price
  // Less effective for: WHERE price = 9.99 (doesn't use composite index)

  // ── Partial index (advanced PostgreSQL feature) ─────────────────────────
  @Index('idx_active_products', { where: '"isActive" = true' })
  // Only indexes ACTIVE products — much smaller index
  // Only helps queries that include WHERE isActive = true
  @Column({ default: true })
  isActive: boolean;
}
```

## 4. Repository Pattern — CRUD Operations

### Injecting and using Repository

```typescript
// src/users/users.service.ts

import { Injectable, NotFoundException, ConflictException } from '@nestjs/common';
import { InjectRepository } from '@nestjs/typeorm';
import { Repository, FindManyOptions, FindOneOptions, Like, Between, In, IsNull, Not } from 'typeorm';
import { User } from './entities/user.entity';
import { CreateUserDto } from './dto/create-user.dto';
import { UpdateUserDto } from './dto/update-user.dto';

@Injectable()
export class UsersService {
  constructor(
    @InjectRepository(User)
    // @InjectRepository(User) injects a Repository<User> instance
    // TypeORM's Repository provides all CRUD methods typed for User
    // This repository is pre-configured to run queries on the 'users' table
    private readonly userRepository: Repository<User>,
  ) {}

  // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  // FIND METHODS
  // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  async findAll(): Promise<User[]> {
    return this.userRepository.find();
    // SELECT * FROM users WHERE deleted_at IS NULL
    // Returns all non-deleted users (soft delete aware)
  }

  async findWithOptions(): Promise<User[]> {
    return this.userRepository.find({
      // ── where: filter conditions ───────────────────────────────────
      where: {
        isActive: true,        // WHERE is_active = true
        role: UserRole.USER,   // AND role = 'user'
      },
      // Multiple conditions in 'where' object = AND

      // ── OR conditions ──────────────────────────────────────────────
      // where: [
      //   { role: UserRole.ADMIN },
      //   { role: UserRole.MODERATOR },
      // ],
      // Array of objects = OR: WHERE role = 'admin' OR role = 'moderator'

      // ── relations: eager load related data ────────────────────────
      relations: {
        profile: true,         // LEFT JOIN users.profile
        orders: true,          // LEFT JOIN users.orders
        orders: { items: true }, // Nested: orders AND their items
      },
      // Without relations, related entities are NOT loaded (lazy by default)

      // ── select: only specific columns ─────────────────────────────
      select: {
        id: true,
        firstName: true,
        email: true,
        // Only these columns are fetched — smaller payload, faster query
        // password: false — excluded (but @Column({ select: false }) is better)
      },

      // ── order: sorting ─────────────────────────────────────────────
      order: {
        createdAt: 'DESC',     // ORDER BY created_at DESC
        firstName: 'ASC',      // Then alphabetically
      },

      // ── pagination ─────────────────────────────────────────────────
      skip: 0,                 // OFFSET — skip first N records
      take: 20,                // LIMIT — return at most N records

      // ── cache ──────────────────────────────────────────────────────
      cache: 5000,             // Cache this query result for 5 seconds
      // Requires cache configuration in TypeOrmModule setup
    });
  }

  async findOne(id: number): Promise<User> {
    const user = await this.userRepository.findOne({
      where: { id },
      // For password: explicitly select it (since select: false on entity)
      // select: { id: true, email: true, password: true },
    });

    if (!user) {
      throw new NotFoundException(`User #${id} not found`);
    }
    return user;
  }

  // Using TypeORM operators for complex WHERE clauses
  async findWithOperators(): Promise<User[]> {
    return this.userRepository.find({
      where: [
        // All TypeORM operators are imported from 'typeorm'

        // Like: pattern matching (% = wildcard)
        { email: Like('%@gmail.com') },          // WHERE email LIKE '%@gmail.com'

        // ILike: case-insensitive LIKE (PostgreSQL specific)
        // { name: ILike('%alice%') }             // WHERE name ILIKE '%alice%'

        // Between: range check
        // { age: Between(18, 65) }               // WHERE age BETWEEN 18 AND 65

        // In: match any value in array
        // { role: In([UserRole.ADMIN, UserRole.MODERATOR]) }  // WHERE role IN (...)

        // IsNull: check for NULL
        // { phone: IsNull() }                    // WHERE phone IS NULL

        // Not: negate a condition
        // { isActive: Not(false) }               // WHERE is_active != false

        // LessThan, MoreThan, LessThanOrEqual, MoreThanOrEqual
        // { loginCount: MoreThan(10) }           // WHERE login_count > 10
      ],
    });
  }

  // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  // CREATE & SAVE
  // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  async create(createUserDto: CreateUserDto): Promise<User> {
    // Check for duplicate email
    const existing = await this.userRepository.findOne({
      where: { email: createUserDto.email },
    });
    if (existing) {
      throw new ConflictException('Email already registered');
    }

    // repository.create(): creates a User CLASS INSTANCE from plain object
    // Does NOT save to database — just instantiates the class
    // This triggers @BeforeInsert() hooks are NOT run yet
    const user = this.userRepository.create(createUserDto);
    // Equivalent to: const user = new User(); Object.assign(user, createUserDto);

    // repository.save(): persists to database
    // - New entity (no id): runs INSERT + @BeforeInsert() hooks
    // - Existing entity (has id): runs UPDATE + @BeforeUpdate() hooks
    // - Returns the saved entity with generated id, timestamps
    return this.userRepository.save(user);
  }

  // Bulk create
  async createMany(dtos: CreateUserDto[]): Promise<User[]> {
    const users = this.userRepository.create(dtos);
    // create() also accepts an array — returns User[]
    
    return this.userRepository.save(users);
    // save() also accepts an array — single transaction, more efficient
    // Better than calling save() in a loop
  }

  // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  // UPDATE
  // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  async update(id: number, updateUserDto: UpdateUserDto): Promise<User> {
    // Method 1: load → modify → save (triggers hooks, events)
    const user = await this.findOne(id);  // Throws if not found
    Object.assign(user, updateUserDto);
    return this.userRepository.save(user);

    // Method 2: repository.update() — direct UPDATE query, no hooks
    // await this.userRepository.update(id, updateUserDto);
    // return this.findOne(id);
    // Faster (no SELECT), but @BeforeUpdate() hooks don't run
    // Use when you need performance over hooks

    // Method 3: repository.preload() — advanced pattern
    // const user = await this.userRepository.preload({ id, ...updateUserDto });
    // Merges id + updateUserDto, loads related entities if relations specified
    // if (!user) throw new NotFoundException();
    // return this.userRepository.save(user);
  }

  // Bulk update
  async bulkUpdate(ids: number[], isActive: boolean): Promise<void> {
    await this.userRepository.update(
      { id: In(ids) },    // WHERE id IN (1, 2, 3)
      { isActive },       // SET is_active = true/false
    );
    // Single UPDATE query — much faster than updating one by one
  }

  // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  // DELETE
  // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  async remove(id: number): Promise<void> {
    const user = await this.findOne(id);
    await this.userRepository.remove(user);
    // Hard DELETE — permanently removes from database
    // @BeforeRemove() and @AfterRemove() hooks run
  }

  async softRemove(id: number): Promise<void> {
    const user = await this.findOne(id);
    await this.userRepository.softRemove(user);
    // SOFT DELETE — sets deleted_at = NOW()
    // Record stays in database, excluded from normal queries
    // Requires @DeleteDateColumn() on entity
  }

  async restore(id: number): Promise<void> {
    await this.userRepository.restore(id);
    // Restore soft-deleted record: sets deleted_at = NULL
  }

  // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  // COUNT & EXISTS
  // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  async count(): Promise<number> {
    return this.userRepository.count({
      where: { isActive: true },
    });
    // SELECT COUNT(*) FROM users WHERE is_active = true
  }

  async exists(email: string): Promise<boolean> {
    return this.userRepository.exists({
      where: { email },
    });
    // Efficient EXISTS check — doesn't load the full entity
    // Better than findOne() when you only need to know if it exists
  }

  // findAndCount: returns data AND total count in one query
  async findPaginated(page: number, limit: number) {
    const [data, total] = await this.userRepository.findAndCount({
      where: { isActive: true },
      order: { createdAt: 'DESC' },
      skip: (page - 1) * limit,
      take: limit,
    });

    return {
      data,
      total,
      page,
      lastPage: Math.ceil(total / limit),
      hasNextPage: page * limit < total,
    };
  }
}
```

```typescript
  Entity, PrimaryGeneratedColumn, Column,
  ManyToOne, OneToMany, JoinColumn, CreateDateColumn
} from 'typeorm';
import { User } from '../../users/entities/user.entity';
import { OrderItem } from './order-item.entity';

@Entity('orders')
export class Order {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ type: 'decimal', precision: 10, scale: 2 })
  total: number;

  @Column({ type: 'enum', enum: OrderStatus, default: OrderStatus.PENDING })
  status: OrderStatus;

  @CreateDateColumn()
  createdAt: Date;

  // Many Orders belong to One User
  @ManyToOne(() => User, (user) => user.orders, {
    onDelete: 'CASCADE',
    // If the User is deleted → also delete all their orders
    // Options: 'CASCADE' | 'SET NULL' | 'RESTRICT' | 'NO ACTION'
    // 'RESTRICT': prevent deleting user if they have orders (safe default)
    // 'CASCADE': delete orders when user is deleted (convenient)
    // 'SET NULL': set user_id = NULL when user is deleted (requires nullable)
    
    nullable: false,
    // order.user can never be null — every order must have a user
    
    eager: false,
    // eager: false (default) — related entity NOT loaded automatically
    // eager: true — ALWAYS loads user when you load an order
    // Avoid eager loading — it can cause N+1 problems
  })
  @JoinColumn({ name: 'user_id' })
  // @JoinColumn is optional on ManyToOne (it's implied)
  // Use it to customize the foreign key column name
  user: User;

  @Column()
  userId: number;  // The actual FK column — useful for filtering without join

  // One Order has Many OrderItems
  @OneToMany(() => OrderItem, (item) => item.order, {
    cascade: true,
    // cascade: true — when you save an Order, also save its OrderItems
    // This means you can do:
    //   const order = new Order();
    //   order.items = [new OrderItem(), new OrderItem()];
    //   await orderRepository.save(order);  // Saves order AND all items
    // Options: true | false | ['insert'] | ['update'] | ['remove']
  })
  items: OrderItem[];
}

export enum OrderStatus {
  PENDING = 'pending',
  CONFIRMED = 'confirmed',
  SHIPPED = 'shipped',
  DELIVERED = 'delivered',
  CANCELLED = 'cancelled',
}

// @OneToMany(() => Order, (order) => order.user)
// orders: Order[];
```

## 6. QueryBuilder — Advanced SQL Power

### When to use QueryBuilder vs Repository methods

```
Use Repository.find() when:
  ✓ Simple WHERE conditions
  ✓ Basic sorting and pagination
  ✓ Standard relation loading

Use QueryBuilder when:
  ✓ Complex JOINs with conditions
  ✓ Aggregations (SUM, COUNT, AVG, GROUP BY)
  ✓ Subqueries
  ✓ HAVING clauses
  ✓ Window functions
  ✓ UNION queries
  ✓ Performance tuning with specific indexes
  ✓ Any SQL you can't express with find()
```

### QueryBuilder fundamentals

```typescript
// src/users/users.service.ts

import { InjectRepository } from '@nestjs/typeorm';
import { Repository } from 'typeorm';

@Injectable()
export class UsersService {
  constructor(
    @InjectRepository(User)
    private userRepository: Repository<User>,
  ) {}

  // ── Basic QueryBuilder ─────────────────────────────────────────────────
  async findActiveAdmins(): Promise<User[]> {
    return this.userRepository
      .createQueryBuilder('user')
      // 'user' is the ALIAS — how you reference this table in the query
      // Like SQL's: FROM users AS user

      .where('user.isActive = :isActive', { isActive: true })
      // :isActive is a NAMED PARAMETER — prevents SQL injection
      // TypeORM replaces it with a prepared statement parameter
      // NEVER do: .where(`user.id = ${userId}`) — SQL injection risk!

      .andWhere('user.role = :role', { role: UserRole.ADMIN })
      // andWhere: adds AND condition

      .orderBy('user.createdAt', 'DESC')
      // orderBy: ORDER BY clause

      .getMany();
      // getMany(): executes SELECT, returns array of entities
      // getOne(): returns single entity or null
      // getManyAndCount(): returns [entities, totalCount]
      // getRawMany(): returns plain objects (not class instances)
      // getRawOne(): returns single plain object
  }

  // ── Complex JOINs ──────────────────────────────────────────────────────
  async getUsersWithRecentOrders(): Promise<User[]> {
    return this.userRepository
      .createQueryBuilder('user')

      .innerJoinAndSelect('user.orders', 'order')
      // innerJoinAndSelect: JOIN + load the relation
      // 'user.orders': the relation property path (user entity → orders)
      // 'order': alias for the joined table

      .leftJoinAndSelect('order.items', 'item')
      // leftJoinAndSelect: LEFT JOIN + load
      // LEFT JOIN: includes users even if they have no orders
      // INNER JOIN: only users WITH orders

      .where('order.createdAt >= :date', {
        date: new Date(Date.now() - 30 * 24 * 60 * 60 * 1000),
        // Orders from last 30 days
      })

      .andWhere('order.status != :status', { status: OrderStatus.CANCELLED })

      .orderBy('order.createdAt', 'DESC')

      .getMany();
  }

  // ── Aggregations — GROUP BY, HAVING ────────────────────────────────────
  async getOrderStatsByUser() {
    return this.userRepository
      .createQueryBuilder('user')

      .select('user.id', 'userId')
      // .select: specify which columns to include
      // First arg: column expression, Second: alias

      .addSelect('user.email', 'email')
      // addSelect: add another column

      .addSelect('COUNT(order.id)', 'orderCount')
      // Aggregation function — how many orders each user has

      .addSelect('SUM(order.total)', 'totalSpent')
      // Sum of all order totals

      .addSelect('AVG(order.total)', 'avgOrderValue')
      // Average order value

      .leftJoin('user.orders', 'order')
      // leftJoin (without AndSelect): JOIN but don't load as relation
      // Use when you only need aggregate values, not the full order data

      .groupBy('user.id')
      // GROUP BY: required when using aggregations
      .addGroupBy('user.email')
      // addGroupBy: add more columns to GROUP BY

      .having('COUNT(order.id) > :minOrders', { minOrders: 5 })
      // HAVING: filter on aggregated values (like WHERE but after GROUP BY)
      // Can't use WHERE for aggregated conditions — must use HAVING

      .orderBy('"orderCount"', 'DESC')
      // Order by the alias — show most active users first

      .getRawMany();
      // getRawMany(): returns plain objects with your selected aliases
      // [{ userId: 1, email: 'a@b.com', orderCount: '10', totalSpent: '500.00' }]
      // Note: aggregation results are strings — parse them in your service
  }

  // ── Subqueries ─────────────────────────────────────────────────────────
  async getUsersWithHighValueOrders(minTotal: number): Promise<User[]> {
    // Find users who have at least one order over $100
    
    const subQuery = this.userRepository.manager
      .createQueryBuilder()
      .select('order.userId')
      .from('orders', 'order')
      .where('order.total > :minTotal', { minTotal })
      .groupBy('order.userId');
      // Subquery: SELECT DISTINCT user_id FROM orders WHERE total > 100

    return this.userRepository
      .createQueryBuilder('user')
      .where(`user.id IN (${subQuery.getQuery()})`)
      // Embed the subquery inside IN()
      .setParameters(subQuery.getParameters())
      // Pass the subquery's parameters to the main query
      .getMany();
  }

  // ── Pagination with QueryBuilder ────────────────────────────────────────
  async findPaginatedWithSearch(
    page: number,
    limit: number,
    search?: string,
  ) {
    const queryBuilder = this.userRepository
      .createQueryBuilder('user')
      .leftJoinAndSelect('user.profile', 'profile');

    // Conditional WHERE — only add if search term provided
    if (search) {
      queryBuilder.where(
        '(user.firstName ILIKE :search OR user.email ILIKE :search)',
        // ILIKE: case-insensitive LIKE (PostgreSQL)
        { search: `%${search}%` },
        // % wildcard: matches anything before/after the search term
      );
    }

    const [data, total] = await queryBuilder
      .orderBy('user.createdAt', 'DESC')
      .skip((page - 1) * limit)  // OFFSET
      .take(limit)                // LIMIT
      .getManyAndCount();
      // getManyAndCount(): runs TWO queries:
      // 1. SELECT ... LIMIT limit OFFSET skip
      // 2. SELECT COUNT(*) (same WHERE, no LIMIT/OFFSET)

    return {
      data,
      total,
      page,
      limit,
      lastPage: Math.ceil(total / limit),
    };
  }

  // ── Update with QueryBuilder ────────────────────────────────────────────
  async deactivateInactiveUsers(daysInactive: number): Promise<number> {
    const cutoffDate = new Date();
    cutoffDate.setDate(cutoffDate.getDate() - daysInactive);

    const result = await this.userRepository
      .createQueryBuilder()
      .update(User)
      // UPDATE query mode
      .set({
        isActive: false,
        // SET is_active = false
      })
      .where('last_login_at < :cutoffDate', { cutoffDate })
      .andWhere('isActive = true')
      .execute();
      // execute(): runs the query, returns UpdateResult
      // result.affected: number of rows updated

    return result.affected || 0;
  }

  // ── Delete with QueryBuilder ────────────────────────────────────────────
  async deleteOldSessions(expiryDate: Date): Promise<number> {
    const result = await this.userRepository.manager
      .createQueryBuilder()
      .delete()
      .from('sessions')
      .where('expires_at < :expiryDate', { expiryDate })
      .execute();

    return result.affected || 0;
  }
}
```

## 7. Migrations — Production-Safe Schema Changes

### Why migrations are critical

```
synchronize: true (development only):
  + Convenient: TypeORM auto-updates table to match entity
  - DANGEROUS: drops columns, loses data
  - UNPREDICTABLE: can't control exactly what SQL runs
  - NO HISTORY: no record of what changed and when

Migrations (production standard):
  + Explicit: you know exactly what SQL will run
  + Safe: reversible with down() method
  + Version controlled: committed to git, reviewed in PRs
  + Team friendly: everyone applies the same changes
  + Auditable: track schema evolution over time
```

## 8. Transactions — Atomic Operations

### What is a transaction?
A transaction is a set of database operations that **either ALL succeed or ALL fail**. There's no partial state.

```
Without transaction:
  1. Deduct $100 from Alice's account  ✅
  2. Server crashes
  3. Add $100 to Bob's account         ❌ Never happens!
  Result: Alice loses $100, Bob never gets it — DISASTER

With transaction:
  BEGIN TRANSACTION
  1. Deduct $100 from Alice's account
  2. Server crashes → ROLLBACK everything
  Result: Alice keeps her $100 — consistent state preserved
```

### Method 1: queryRunner (most control)

```typescript
// src/payments/payments.service.ts

import { Injectable } from '@nestjs/common';
import { InjectRepository } from '@nestjs/typeorm';
import { Repository, DataSource } from 'typeorm';

@Injectable()
export class PaymentsService {
  constructor(
    @InjectRepository(User)
    private userRepository: Repository<User>,

    @InjectRepository(Transaction)
    private transactionRepository: Repository<Transaction>,

    private dataSource: DataSource,
    // DataSource: TypeORM's connection manager
    // Inject to create queryRunners for manual transaction control
  ) {}

  async transferFunds(
    fromUserId: number,
    toUserId: number,
    amount: number,
  ): Promise<void> {
    // Create a queryRunner — a single database connection
    const queryRunner = this.dataSource.createQueryRunner();

    // Acquire a connection from the pool
    await queryRunner.connect();

    // Start the transaction
    await queryRunner.startTransaction();

    try {
      // All operations use queryRunner.manager — same connection, same transaction

      // Step 1: Load sender
      const sender = await queryRunner.manager.findOne(User, {
        where: { id: fromUserId },
        lock: { mode: 'pessimistic_write' },
        // Pessimistic locking: locks the row for update
        // No other transaction can modify this row until ours completes
        // Prevents race conditions: two transfers reading same balance simultaneously
      });

      if (!sender) throw new NotFoundException('Sender not found');
      if (sender.balance < amount) throw new BadRequestException('Insufficient funds');

      // Step 2: Load receiver
      const receiver = await queryRunner.manager.findOne(User, {
        where: { id: toUserId },
        lock: { mode: 'pessimistic_write' },
      });

      if (!receiver) throw new NotFoundException('Receiver not found');

      // Step 3: Update balances
      sender.balance -= amount;
      receiver.balance += amount;

      await queryRunner.manager.save(User, sender);
      await queryRunner.manager.save(User, receiver);

      // Step 4: Create transaction record
      const transaction = queryRunner.manager.create(Transaction, {
        fromUserId,
        toUserId,
        amount,
        status: 'completed',
      });
      await queryRunner.manager.save(Transaction, transaction);

      // All operations succeeded → commit
      await queryRunner.commitTransaction();

    } catch (error) {
      // Something failed → rollback ALL operations
      await queryRunner.rollbackTransaction();
      // All changes are reverted — database is back to clean state
      throw error;  // Re-throw so controller returns proper error response

    } finally {
      // ALWAYS release the queryRunner — returns connection to pool
      // If you forget this, you'll leak connections and eventually exhaust the pool
      await queryRunner.release();
    }
  }
}
```

### Method 2: dataSource.transaction() (cleaner syntax)

```typescript
async transferFundsClean(
  fromUserId: number,
  toUserId: number,
  amount: number,
): Promise<void> {
  // dataSource.transaction() handles begin/commit/rollback automatically
  await this.dataSource.transaction(async (manager) => {
    // manager: EntityManager — works like repository but not entity-specific
    // All operations inside this callback are in the SAME transaction

    const sender = await manager.findOne(User, { where: { id: fromUserId } });
    const receiver = await manager.findOne(User, { where: { id: toUserId } });

    if (sender.balance < amount) throw new BadRequestException('Insufficient funds');

    sender.balance -= amount;
    receiver.balance += amount;

    await manager.save(User, sender);
    await manager.save(User, receiver);

    // If ANY error is thrown inside this callback → automatic ROLLBACK
    // If no error → automatic COMMIT
    // Much cleaner than manual queryRunner
  });
}
```

## 9. Advanced Patterns — Soft Delete, Listeners, Subscribers

### Soft Delete — keep records, just hide them

```typescript
// On the entity: @DeleteDateColumn() (shown in section 2)
// This enables soft delete for this entity

// In usage:
await userRepository.softDelete(id);
// SQL: UPDATE users SET deleted_at = NOW() WHERE id = 1
// Record NOT removed from database

await userRepository.find();
// SQL: SELECT * FROM users WHERE deleted_at IS NULL
// Soft-deleted records AUTOMATICALLY excluded from all queries

await userRepository.find({ withDeleted: true });
// SQL: SELECT * FROM users (including deleted)
// Use withDeleted: true to see soft-deleted records

await userRepository.restore(id);
// SQL: UPDATE users SET deleted_at = NULL WHERE id = 1
// Un-delete the record

// Why soft delete?
// - Audit trail: know when and that something was deleted
// - Data recovery: restore accidentally deleted records
// - Compliance: GDPR may require keeping records for certain periods
// - Referential integrity: orders reference users — hard delete breaks FK
```

### Entity Listeners — hooks on individual entities

```typescript
// src/users/entities/user.entity.ts

import { BeforeInsert, BeforeUpdate, AfterInsert, AfterUpdate, AfterLoad } from 'typeorm';
import * as bcrypt from 'bcrypt';

@Entity('users')
export class User {
  // ... columns ...

  @AfterLoad()
  // Runs after entity is loaded from database
  // Good for: computed fields, decryption
  computeFullName() {
    this.fullName = `${this.firstName} ${this.lastName}`;
  }

  @BeforeInsert()
  // Runs before INSERT query
  // Good for: hashing passwords, normalizing data
  async hashPasswordBeforeInsert() {
    if (this.password) {
      this.password = await bcrypt.hash(this.password, 12);
    }
    this.email = this.email.toLowerCase().trim();
  }

  @BeforeUpdate()
  // Runs before UPDATE query
  // Good for: hash new password if changed
  async hashPasswordBeforeUpdate() {
    if (this.password && !this.password.startsWith('$2b$')) {
      // Only hash if it looks like a plain password (not already hashed)
      // bcrypt hashes start with $2b$
      this.password = await bcrypt.hash(this.password, 12);
    }
  }

  @AfterInsert()
  // Runs after successful INSERT
  logCreation() {
    console.log(`User created: ${this.email}`);
    // In real app: trigger welcome email, analytics event, etc.
  }

  @AfterUpdate()
  // Runs after successful UPDATE
  onUpdate() {
    // Trigger cache invalidation, send notification, etc.
  }
}
```

### Subscribers — global listeners for all entities

```typescript
// src/common/subscribers/audit.subscriber.ts
// Logs every database operation — audit trail

import {
  EntitySubscriberInterface,
  EventSubscriber,
  InsertEvent,
  UpdateEvent,
  RemoveEvent,
} from 'typeorm';
import { Injectable } from '@nestjs/common';
import { InjectDataSource } from '@nestjs/typeorm';
import { DataSource } from 'typeorm';

@EventSubscriber()
// @EventSubscriber() registers this as a TypeORM event subscriber
@Injectable()
// @Injectable() to use NestJS DI
export class AuditSubscriber implements EntitySubscriberInterface {
  constructor(@InjectDataSource() private dataSource: DataSource) {
    dataSource.subscribers.push(this);
    // Register the subscriber with TypeORM's DataSource
  }

  // listenTo(): which entity to listen to (omit for all entities)
  // listenTo() { return User; }  // Only listen to User events

  afterInsert(event: InsertEvent<any>): void {
    console.log(`INSERTED into ${event.metadata.tableName}:`, event.entity);
    // event.entity: the entity that was inserted
    // event.metadata: TypeORM metadata about the entity
  }

  afterUpdate(event: UpdateEvent<any>): void {
    console.log(`UPDATED ${event.metadata.tableName} #${event.entity?.id}`);
    // event.updatedColumns: which columns changed
    // event.databaseEntity: the entity BEFORE update
    // event.entity: the entity AFTER update
  }

  beforeRemove(event: RemoveEvent<any>): void {
    console.log(`DELETING ${event.metadata.tableName} #${event.entityId}`);
    // event.entityId: the id of entity being deleted
    // Perfect for: audit log, cleanup of related resources
  }
}

// Register in module:
@Module({
  providers: [AuditSubscriber],
})
export class AppModule {}
```

## 10. Raw Queries — When ORM Is Not Enough

```typescript
// Sometimes you need pure SQL — complex analytics, DB-specific features,
// performance-critical queries where ORM overhead matters

@Injectable()
export class ReportsService {
  constructor(
    private dataSource: DataSource,
  ) {}

  // ── Raw query with parameters ──────────────────────────────────────────
  async getUserGrowthReport(year: number) {
    return this.dataSource.query(
      `
      SELECT
        DATE_TRUNC('month', created_at) AS month,
        COUNT(*) AS new_users,
        SUM(COUNT(*)) OVER (ORDER BY DATE_TRUNC('month', created_at)) AS cumulative_users
        -- Window function: running total — not possible with ORM
      FROM users
      WHERE EXTRACT(YEAR FROM created_at) = $1
        AND deleted_at IS NULL
      GROUP BY DATE_TRUNC('month', created_at)
      ORDER BY month
      `,
      [year],  // Parameterized — safe from SQL injection
    );
  }

  // ── Using QueryBuilder for raw with mapping ────────────────────────────
  async getTopSpenders(limit: number) {
    return this.dataSource
      .createQueryBuilder()
      .select([
        'u.id AS "userId"',
        'u.email AS "email"',
        'COALESCE(SUM(o.total), 0) AS "totalSpent"',
        // COALESCE: return 0 if SUM is NULL (user with no orders)
      ])
      .from(User, 'u')
      .leftJoin(Order, 'o', 'o.user_id = u.id AND o.status != :status', {
        status: 'cancelled',
      })
      .groupBy('u.id, u.email')
      .orderBy('"totalSpent"', 'DESC')
      .limit(limit)
      .getRawMany();
  }

  // ── EntityManager query ────────────────────────────────────────────────
  async complexReport() {
    const entityManager = this.dataSource.manager;

    // query() on entity manager — same as dataSource.query()
    const result = await entityManager.query(
      'SELECT pg_database_size(current_database()) AS db_size'
    );

    return result[0];
  }
}
```

## 11. Interview Questions & Answers

**Q1: What is the difference between synchronize:true and migrations in TypeORM?**

> "`synchronize: true` makes TypeORM automatically compare your entity classes to the database schema on every app start and runs ALTER TABLE or CREATE TABLE statements to make them match. It's convenient for local development but catastrophic in production — it can drop columns (losing data), recreate tables, and cause downtime. You have no visibility into what SQL it will run. Migrations are explicit, versioned SQL scripts with `up()` and `down()` methods that you write and commit to git. TypeORM tracks which migrations have run in a `migrations` table. This means schema changes are reviewed in pull requests, are reversible, have a history, and you know exactly what SQL will execute. In production, always use `synchronize: false` and migrations."

---

**Q2: Explain the difference between repository.save() and repository.update().**

> "`save()` is a high-level method that first loads the entity if needed, runs lifecycle hooks like `@BeforeInsert()` and `@BeforeUpdate()`, then runs INSERT or UPDATE depending on whether the entity has an id. It's safer but makes two queries (SELECT then INSERT/UPDATE). `update()` is a low-level method that directly runs an UPDATE query without loading the entity first — it's faster but skips all lifecycle hooks and doesn't trigger `@BeforeUpdate()`. Use `save()` when you need hooks to run (e.g., password hashing) or when working with entity instances. Use `update()` for bulk updates or performance-critical paths where you don't need hooks."

---

**Q3: What is the N+1 query problem and how does TypeORM help you avoid it?**

> "The N+1 problem is when you load N entities, then run 1 additional query per entity to load its relations — resulting in N+1 total queries. For example, loading 100 users and then fetching each user's profile separately makes 101 queries. TypeORM provides several solutions: using `relations` in find options or `leftJoinAndSelect()` in QueryBuilder eager-loads related data in a single JOIN query. For cases where you want relations sometimes but not always, you use `select: false` on the relation and explicitly join when needed. The `DataLoader` pattern is also used with NestJS GraphQL to batch relation loading. The key is to use JOIN queries instead of separate queries per entity."

---

**Q4: When would you use QueryBuilder instead of Repository.find()?**

> "I reach for QueryBuilder when the query requires something find() can't express. Specifically: GROUP BY with aggregations (SUM, COUNT, AVG), HAVING clauses on aggregated values, complex multi-condition JOINs, subqueries, window functions, UNION queries, or when I need to select specific columns and aliases instead of full entities. QueryBuilder gives you SQL's full power with TypeScript safety for parameters (preventing SQL injection) and some type assistance. Another case is performance — sometimes I need to control the exact SQL generated, especially for reports that scan millions of rows."

---

**Q5: How do you implement transactions in NestJS with TypeORM?**

> "I use two approaches depending on complexity. For simple cases, `dataSource.transaction(async manager => { ... })` — the callback receives an `EntityManager`, all operations in the callback run in the same transaction, and TypeORM auto-commits on success or rolls back on any error. For complex cases needing more control (like savepoints or pessimistic locking), I use `queryRunner` manually: `const qr = dataSource.createQueryRunner()`, call `qr.connect()`, `qr.startTransaction()`, perform operations with `qr.manager`, then `qr.commitTransaction()` in try block and `qr.rollbackTransaction()` in catch, always calling `qr.release()` in finally to return the connection to the pool. The inject point is `DataSource` from TypeORM, not a repository."

---

**Q6: What is the difference between eager and lazy loading of relations?**

> "Eager loading loads the related entity automatically every time the parent is queried — you set `eager: true` on the relation decorator. Lazy loading defers loading the relation until you access the property, using Promises. TypeORM also has explicit loading via `relations` option in find() or `leftJoinAndSelect()` in QueryBuilder — this is the recommended approach. In practice, I avoid both `eager: true` and lazy loading. Eager loading causes unnecessary JOINs on every query even when you don't need the relation. Lazy loading requires all methods touching relations to be async, which can be confusing. Explicit loading in the query is best — you JOIN only when you need the data."

---

**Q7: What is pessimistic locking and when do you use it in TypeORM?**

> "Pessimistic locking prevents two transactions from reading and then writing the same row simultaneously — it locks the row when you read it, forcing other transactions to wait until yours completes. You use it with `lock: { mode: 'pessimistic_write' }` in the find options, combined with a transaction. The classic use case is financial transfers: if two transfers try to debit the same account simultaneously, without locking they both read the old balance, both deduct, and you end up with one deduction instead of two. With pessimistic locking, the second transfer waits for the first to complete before reading the balance. The downside is reduced concurrency — locked rows block other transactions. Optimistic locking with `@VersionColumn()` is less blocking — it allows concurrent reads but fails the save if another transaction modified the entity in the meantime."

## Quick Reference — Day 6 Cheat Sheet

```
Setup:
  npm install @nestjs/typeorm typeorm pg
  TypeOrmModule.forRootAsync() in AppModule
  TypeOrmModule.forFeature([Entity]) in feature modules

Entity decorators:
  @Entity('table_name')       → maps class to DB table
  @PrimaryGeneratedColumn()   → auto-increment int PK
  @PrimaryGeneratedColumn('uuid') → UUID PK
  @Column()                   → basic column
  @Column({ nullable: true }) → allows NULL
  @Column({ select: false })  → excluded from default SELECT
  @CreateDateColumn()         → auto-set on INSERT
  @UpdateDateColumn()         → auto-set on INSERT + UPDATE
  @DeleteDateColumn()         → enables soft delete
  @VersionColumn()            → optimistic locking counter
  @Index()                    → creates DB index
  @Unique(['a','b'])          → composite unique constraint

Relations:
  @OneToOne → @JoinColumn (FK on this side)
  @ManyToOne (N side) → @OneToMany (1 side)
  @ManyToMany → @JoinTable (owning side only)
  cascade: true           → save children when saving parent
  onDelete: 'CASCADE'     → delete children when parent deleted

Repository methods:
  find(options)           → SELECT with conditions
  findOne(options)        → SELECT LIMIT 1
  findAndCount(options)   → SELECT + COUNT(*)
  exists(options)         → efficient exists check
  save(entity)            → INSERT or UPDATE with hooks
  update(where, partial)  → direct UPDATE, no hooks
  remove(entity)          → hard DELETE with hooks
  softRemove(entity)      → sets deleted_at
  restore(id)             → clears deleted_at

QueryBuilder when:
  ✓ GROUP BY, HAVING, aggregations
  ✓ Complex JOINs with conditions
  ✓ Subqueries, window functions
  ✓ Selecting specific columns/aliases
  ✓ Bulk UPDATE/DELETE queries

Transactions:
  Simple: dataSource.transaction(async manager => { })
  Full control: queryRunner.connect() → startTransaction()
               → try/commit → catch/rollback → finally/release()

Migrations (ALWAYS in production):
  synchronize: false        → disable auto-sync
  migration:generate        → diff entities vs DB
  migration:run             → apply pending migrations
  migration:revert          → undo last migration
  up() = changes, down() = reversal

Soft Delete: @DeleteDateColumn() + softRemove() + withDeleted: true
Lifecycle hooks: @BeforeInsert @BeforeUpdate @AfterLoad @AfterInsert
```
