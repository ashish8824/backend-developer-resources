# Module 8: Persistence (RDB & AOF)

**Level:** Intermediate

We said Redis is "in-memory," which raises an important question: **what happens to your data if the Redis server restarts or crashes?** This module answers that.

---

## 1. The Core Trade-off

- **Pure memory, no persistence** = blazing fast, but a crash/restart means **total data loss**.
- **Persistence enabled** = Redis periodically (or continuously) writes data to disk, so it can reload everything after a restart.

Redis gives you **two persistence strategies**, which can even be used together: **RDB** and **AOF**.

---

## 2. RDB (Redis Database File) — Point-in-Time Snapshots

RDB works by taking a full **snapshot** of your entire dataset at set intervals and saving it to a single compact binary file (`dump.rdb`).

```
Time:     0s ────────── 60s ────────── 120s ────────── 180s
Snapshot:                📸 saved         📸 saved         📸 saved
```

**Configuration example (in `redis.conf`):**

```
# Save a snapshot if AT LEAST 1 key changed in the last 900 seconds (15 min)
save 900 1

# Save a snapshot if AT LEAST 10 keys changed in the last 300 seconds (5 min)
save 300 10

# Save a snapshot if AT LEAST 10000 keys changed in the last 60 seconds
save 60 10000
```

Or manually, at any time:

```bash
# Blocks Redis until the snapshot finishes (rarely used directly in production)
SAVE

# Non-blocking: Redis forks a child process to save in the background (preferred)
BGSAVE
```

**Pros:**
- Compact single file — easy to back up, copy, and restore.
- Faster restarts than AOF (loading one file is quicker than replaying a long log).
- Minimal performance impact on the main Redis process (since `BGSAVE` uses a forked child process).

**Cons:**
- You can lose data since the *last* snapshot. If Redis crashes 4 minutes after the last snapshot, those 4 minutes of writes are gone.

---

## 3. AOF (Append Only File) — Logging Every Write

AOF works completely differently: instead of periodic snapshots, Redis **logs every write command** to a file, in order, as it happens.

```
SET name "Ashish"     → written to appendonly.aof
INCR views            → written to appendonly.aof
HSET user:15 age 24   → written to appendonly.aof
...
```

If Redis restarts, it simply **replays every command in the log**, from the beginning, to rebuild the exact same dataset.

**Enable AOF (in `redis.conf`):**

```
appendonly yes
```

**Controlling how often AOF is flushed to disk (`appendfsync`):**

```
appendfsync always     # flush to disk after EVERY write — safest, but slowest
appendfsync everysec   # flush once per second — great balance (default, most common)
appendfsync no         # let the OS decide when to flush — fastest, least safe
```

**Pros:**
- Much safer — with `everysec`, you can lose *at most* ~1 second of writes.
- Human-readable-ish log format (can be inspected).

**Cons:**
- Files grow larger over time than a compact RDB snapshot.
- Slightly slower to restart (must replay the whole log), though Redis periodically **rewrites/compacts** the AOF file in the background (`BGREWRITEAOF`) to keep it manageable.

```bash
# Manually trigger AOF compaction/rewrite
BGREWRITEAOF
```

---

## 4. RDB vs AOF — Side-by-Side

| | RDB | AOF |
|---|---|---|
| What it stores | Periodic full snapshots | Every write command, in order |
| Data loss risk | Can lose data since last snapshot (minutes) | Minimal (as low as ~1 second with `everysec`) |
| File size | Compact | Larger (though auto-compacted) |
| Restart speed | Fast (load one file) | Slower (replay the log), but still fast in practice |
| Best for | Backups, disaster recovery, fast restarts | Strong durability guarantees |

---

## 5. Using Both Together (Recommended for Production)

Most production setups enable **both**:

```
appendonly yes
save 900 1
save 300 10
save 60 10000
```

- **AOF** protects against losing recent writes (small window of risk).
- **RDB** provides a compact snapshot great for backups and very fast disaster recovery.

If both are enabled and Redis restarts, it will use **AOF** to rebuild data (since it's more up-to-date), falling back to RDB if AOF is unavailable.

---

## 6. No Persistence At All — When Is That OK?

If Redis is used **purely as a disposable cache** — meaning every single key can be regenerated by re-querying your real database — you might intentionally disable persistence entirely for maximum speed and simplicity. If Redis restarts, your app just experiences a temporary "cold cache" (slightly slower) until it refills naturally. No permanent data is lost, because the database was always the source of truth.

```
# Disable persistence entirely (pure cache scenario)
save ""
appendonly no
```

This is a legitimate, common production choice — just make sure you've genuinely confirmed nothing important lives *only* in Redis.

---

## 7. Real-World Example: Why This Matters

Imagine you're using Redis to store shopping cart data temporarily (before checkout), and your server crashes:

- **No persistence:** Every active shopping cart in the world is gone. Angry users.
- **RDB only, snapshot every 5 minutes:** You lose up to 5 minutes of cart changes.
- **AOF with `everysec`:** You lose at most ~1 second of changes — barely noticeable.

The right choice depends entirely on how costly data loss is for *your specific use case*.

---

## 8. Interview Question: "How would you configure persistence for a session store vs a pure cache?"

**Answer:**
- **Session store** (losing sessions logs everyone out — annoying but not catastrophic): AOF with `appendfsync everysec` is a good balance of safety and performance, optionally combined with RDB snapshots for backups.
- **Pure cache** (all data derivable from the database): persistence can be disabled entirely, or a lightweight RDB snapshot policy can be used just to speed up cold starts after a planned restart — but it's not strictly necessary since the cache will simply repopulate from the database.

---

## 9. Summary

- **RDB** = periodic full snapshots — compact, fast restarts, but can lose recent changes.
- **AOF** = a log of every write — safer (minimal data loss), larger files, still fast in practice thanks to background compaction.
- Production systems often use **both together**.
- Pure, fully-disposable caches can safely disable persistence entirely.
- Choose your persistence strategy based on how expensive losing that specific data would actually be.
